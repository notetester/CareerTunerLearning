# 첨삭 설계 원칙

> 목표는 “더 화려한 글”이 아니라 “사용자가 쓴 사실을 보존하면서 더 명확한 글”이다.

## 1. 원문 보존

첨삭은 A의 이력서나 D의 면접 답변을 자동으로 덮어쓰지 않는다. 원문과 개선문을 함께 저장하고 변경 이유를 보여 준다. 사용자가 검토하고 반영할 권한을 가진다.

## 2. 없는 사실을 만들지 않기

시스템 prompt는 성과, 수치, 회사, 기술, 프로젝트와 경험을 발명하지 못하게 한다. 더 강한 문장에 추가 근거가 필요하면 개선문이 아니라 `suggestions`에 남긴다.

출력 형식을 지키는 것만으로 사실성을 보장할 수는 없다. 그래서 provider 응답은 parser와 payload validation을 통과해야 하고, source snapshot과 변경 이유를 함께 남겨 사후 검토 가능성을 높인다.

## 3. 구조화 출력

프런트는 자유 텍스트 한 덩어리를 추측해서 파싱하지 않는다.

| 필드 | 사용자 가치 |
| --- | --- |
| `improvedText` | 비교 가능한 개선문 |
| `summary` | 전체 개선 방향 |
| `issues` | 원문의 문제 |
| `changeReasons` | 변경의 이유 |
| `suggestions` | 사실 확인 후 보강할 내용 |

자체 모델은 parser와 repair를, 외부 provider는 각 structured contract를 사용하되 최종 `CorrectionPayload` 계약은 같다.

## 4. 사용자 모델 선택

AUTO가 기본이지만 사용자는 CareerTuner, Claude, OpenAI를 선택할 수 있다. 결과가 마음에 들지 않으면 다른 모델로 새 실행을 만들 수 있다. 명시 선택은 선택 tier부터 하위 tier로 폴백한다.

## 5. 거짓 성공보다 정직한 실패

첨삭에는 규칙 Mock이 없다. 모든 provider가 실패하면 `AI_UNAVAILABLE`를 반환한다. 원문을 그대로 복제하거나 임의 템플릿을 성공 결과로 저장하면 사용자가 품질 검증을 잘못할 수 있기 때문이다.

## 6. 실행 전 비용 고지

모델 호출 전에 서버 정책 기반 예상 사용권·크레딧 범위와 환불 정책을 보여 준다. 사용자가 승인한 acknowledgement key를 요청에 포함하고 서버가 preflight에서 다시 확인한다.

## 7. 성공 시에만 차감

유효 payload를 얻지 못하면 차감하지 않는다. 성공 결과·사용량 로그·차감은 같은 트랜잭션으로 확정하며 동일 요청 키는 replay한다.

## 8. 소유권은 서버에서 검증

지원 건과 면접 source ID를 받더라도 서버가 현재 사용자 소유인지 다시 확인한다. 연결된 다른 도메인의 원본을 E가 직접 갱신하지 않는다.

## 9. 소프트 삭제

사용자가 이력 삭제를 요청하면 `deleted_at`을 설정한다. 모든 사용자 데이터의 삭제 정책과 감사 요구를 함께 만족시키는 방식이다.
