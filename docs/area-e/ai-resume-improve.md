# 이력서 개선 [#26]

> `correctionType=RESUME`으로 경험 표현, 역할과 성과 근거를 읽기 좋게 다듬는다. A의 이력서 원본은 참조할 수 있지만 E가 덮어쓰지 않는다.

## 설계 목표

이력서에서 가장 위험한 오류는 없는 정량 성과를 그럴듯하게 추가하는 것이다. 따라서 다음 원칙을 지킨다.

- 원문에 있는 사실만 개선문에 사용한다.
- 수치가 필요하지만 없다면 `suggestions`로 확인을 요청한다.
- 원문과 개선문을 별도 저장한다.
- 사용자가 검토한 뒤 A 원본 반영 여부를 결정한다.

## 공통 도메인

이력서 전용 AI 클라이언트나 테이블을 복제하지 않는다. `CorrectionService`가 RESUME 타입을 검증하고 공통 provider·검증·과금 계약을 사용한다.

| 공통 | RESUME 특화 |
| --- | --- |
| provider 체인, 출력 5필드, 이력, 과금 | 역할 명확화·성과 중심 문장·정량 근거 확인 prompt |

## 현재 연결

- `Correction.tsx` 이력서 탭 실제 실행
- AUTO/CAREERTUNER/CLAUDE/OPENAI 선택
- 지원 건 컨텍스트 연결
- 결과·최근 이력·상세·소프트 삭제
- 서버 정책 비용 고지
- 성공 결과 실제 사용권·크레딧 차감
- 요청 키 멱등 replay

## 면접 답변

> "이력서 첨삭은 공통 correction 도메인의 RESUME 분기입니다. 별도 모델을 복제하는 대신 유형별 prompt만 다르게 하고 provider·출력 검증·이력·과금을 공유했습니다. 원문에 없는 수치를 개선문에 넣지 않고 suggestion으로 돌려 사용자가 사실을 확인하게 하며, 결과는 원문과 별도 행으로 저장합니다. 프런트와 실차감까지 연결돼 있고 같은 요청 키 재시도는 한 번만 처리됩니다."
