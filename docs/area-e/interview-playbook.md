# 영역 E 면접 플레이북

> 첨삭 품질만 설명하면 E의 절반만 말한 것이다. 원문 보존, provider 선택, 출력 검증, 멱등 과금, 결제 승인과 추천까지 하나의 사용자 흐름으로 설명해야 한다.

## 30초 요약

> "E는 면접 답변·자기소개서·이력서·포트폴리오 첨삭과 전사 과금 기반을 담당합니다. 네 첨삭을 단일 correction 도메인으로 통합하고, 자체 LoRA→Claude→OpenAI provider 체인과 사용자 모델 선택을 연결했습니다. 원문과 변경 이유를 보존하고, 성공 결과·사용량 로그·실제 사용권 또는 크레딧 차감을 요청 키 기준으로 멱등하게 확정합니다. 결제는 ready/confirm과 서버 금액 대조로 보호하고, 요금제 추천은 LLM 대신 설명 가능한 규칙으로 구현했습니다."

## 3분 구조

1. **제품 문제**: 사용자가 작성한 사실을 지키면서 더 읽기 좋은 문장으로 개선한다.
2. **도메인 통합**: 네 종류를 `correctionType`으로 분기하고 API·테이블·검증·과금을 공유한다.
3. **모델 계층**: AUTO는 CareerTuner LoRA → Claude → OpenAI, 명시 선택은 선택 tier부터 시작한다.
4. **신뢰 경계**: 모델 출력은 parser·payload validation을 통과해야 하며 Mock 성공은 허용하지 않는다.
5. **과금**: preflight 후 성공 결과·로그·실차감을 한 트랜잭션에 묶고 요청 키로 replay한다.
6. **결제**: 서버 가격, ready/confirm, 상태 조건 전환과 결제 키 멱등을 사용한다.
7. **추천**: 월 사용량·잔액을 규칙으로 평가하고 사용자의 명시적 결제로만 이어진다.

## 핵심 결정과 트레이드오프

| 결정 | 이유 | 대가 |
| --- | --- | --- |

| 첨삭 4종 통합 | 입출력·검증·과금 계약이 동일 | 유형별 프롬프트 분기를 잘 관리해야 함 |
| 자체→Claude→OpenAI | 비용·자체 자산과 가용성 균형 | provider별 출력 편차를 검증해야 함 |
| Mock 없음 | 거짓 개선문을 성공으로 오인하지 않음 | 전 provider 장애 시 기능이 실패함 |

## 첨삭 한 번의 생명주기

```text
Correction.tsx
  -> 비용 고지 + 모델 선택
  -> POST /api/corrections (requestKey)
  -> CorrectionService
       1. 입력·소유권·AI_DATA 동의 검증
       2. 동일 requestKey 결과 확인
       3. 과금 preflight
       4. 컨텍스트와 source snapshot 조립
       5. provider 호출 및 payload 검증
       6. SUCCESS usage log INSERT
       7. correction_request INSERT
       8. AiChargeService 실제 차감
       9. transaction commit
  -> 결과 카드 + 최근 이력
```

AI 호출 실패는 별도 트랜잭션의 실패 로그를 남기고 본 결과·차감 트랜잭션으로 들어가지 않는다.

## 동시성과 멱등

### 첨삭 요청

프런트는 같은 입력의 네트워크 재시도에 같은 요청 키를 재사용한다. 서버는 사용자와 요청 키로 기존 행을 먼저 찾고, 동시에 들어온 두 요청은 DB 유니크 제약 이후 승자 결과를 다시 읽어 반환한다.

### 사용권과 크레딧

- 사용권: `remaining_quantity > 0` 조건 UPDATE
- 크레딧: `credit >= cost` 조건 UPDATE
- 장부: 참조 키 유니크 제약
- 이미 소비한 참조: replay/skipped

애플리케이션 선검사는 친절한 응답을 만들고, DB 조건과 유니크 제약은 최종 경쟁 조건을 막는다.

### 결제

- 주문 ID와 결제 키 유니크
- `READY -> PAID` 조건부 전환
- 공급자 Idempotency-Key
- 프런트 StrictMode 중복 effect 방어

## 모델 선택 질문

### AUTO와 명시 선택의 차이는?

AUTO는 CareerTuner→Claude→OpenAI 전체 순서를 탄다. 명시 선택은 해당 tier부터 하위 tier만 시도한다. Claude를 골랐는데 실패하면 OpenAI로 내려가지만 자체 모델로 되돌아가지는 않는다.

### 재시도할 때 다른 모델을 고를 수 있나?

가능하다. UI 기본값은 직전 또는 최초 선택을 유지할 수 있지만 사용자가 다른 모델을 선택해 새 실행을 만들 수 있다. 동일 요청 키는 동일 실행의 네트워크 replay에만 사용한다.

### 전부 실패하면?

`AI_UNAVAILABLE`로 종료한다. 원문 복제나 임의 문장을 성공 결과로 저장하지 않고 과금하지 않는다.

## 데이터 질문

### 왜 append-only 결과인가?

첨삭은 사용자가 의사결정한 기록이다. 새 실행은 새 행으로 남겨 어떤 원문과 모델이 어떤 결과를 만들었는지 추적한다. 사용자 삭제 요청은 물리 삭제 대신 `deleted_at`을 기록해 일반 조회에서 제외한다.

### source snapshot은 왜 필요한가?

지원 건·프로필·면접 평가가 나중에 바뀌어도 당시 모델에 전달한 컨텍스트와 요청 모델·실제 모델을 재현하기 위해서다.

### `ai_usage_log`와 장부는 왜 분리했나?

사용량 로그는 모델 실행과 token·성공 여부를, 장부는 경제적 잔액 변화를 설명한다. 실패 로그는 남지만 금전 장부에는 차감이 없어야 하므로 책임을 분리한다.

## 결제 질문

### 클라이언트가 금액을 바꾸면?

ready에서 서버가 상품 마스터로 금액을 확정한다. confirm은 로컬 주문 금액과 공급자 응답 금액을 모두 대조한다. 브라우저가 보낸 표시 금액은 승인 권위가 아니다.

### AI 호출은 재시도하면서 결제 confirm은 왜 자동 재시도하지 않나?

AI 생성은 요청 키와 후단 저장 전까지 부작용을 제어할 수 있지만 결제 승인은 중복 호출 위험이 크다. 결제는 공급자·서버 멱등 키와 상태 전환을 중심으로 보수적으로 처리한다.

### 정책이 바뀌면 기존 구매도 바뀌나?

구매 시점 정책을 snapshot으로 동결한다. 새 정책은 이후 구매나 정해진 발효 규칙에 적용하고 과거 주문의 의미를 소급 변경하지 않는다.

## 추천 질문

### #28은 정말 구현됐나?

예. `BillingServiceImpl.recommendPlan`, `GET /api/billing/plan-recommendation`, `Billing.tsx` 카드가 연결돼 있다.

### 판단 기준은?

- 월 15회 이상 + 다음 상위 플랜 있음: `UPGRADE_PLAN`
- 잔액 5 이하 + 월 3회 이상: `BUY_CREDITS`
- 나머지: `KEEP`

KEEP을 기본으로 두고 자동 결제를 하지 않는다.

### 왜 추천 문구도 LLM을 안 쓰나?

근거가 단순한 숫자이고 결제 인접 설명이라 규칙 템플릿이 더 검증 가능하다. 모델 호출 비용·장애·과장 위험을 추가할 이유가 없다.
