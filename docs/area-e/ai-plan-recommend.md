# 사용량 기반 요금제 AI 추천 [#28]

> 한 줄 핵심: 사용자의 이번 달 AI 사용량·크레딧 잔액·소비 패턴을 모아 "지금 어떤 요금제로 바꾸거나 얼마를 충전하면 되는지"를 안내하는 기능. **추천에 필요한 데이터는 이미 다 모이지만, 그 데이터를 묶어 추천을 산출하는 코드는 아직 없다.** 이 페이지는 그 갭을 정직하게 설명한다.

## 1. 한 줄 정의와 이 페이지가 답하는 면접 질문

기능 #28(사용량 기반 요금제 추천)은 "이 사용자에게 지금 가장 맞는 요금제 또는 충전 옵션은 무엇인가?"를 데이터로 판단해 안내하는 영역 E의 AI 기능이다. 입력은 **이번 달 사용량 + 현재 크레딧 잔액 + 보유 사용권 + 플랜별 혜택**이고, 출력은 "당신은 이번 달 공고 분석을 18회 썼고 BASIC 한도(20회)에 거의 도달했으니 PRO를 고려하세요" 같은 **권유성 안내 문구**다.

이 페이지가 답하는 면접 질문:

- "사용량 기반 추천은 어떤 데이터로 판단하나요?"
- "이 기능이 정말 동작합니까, 아니면 화면만 있습니까?" → **정직한 답: 데이터·UI는 있고, 추천을 산출하는 엔진은 미구현이다.**
- "왜 추천에 LLM을 전면적으로 쓰지 않았나요?"

:::warning 가장 먼저 알아야 할 사실
`backend`의 billing/payment/credit 어디에도 **추천을 산출하는 서비스·컨트롤러가 없다.** `recommend`, `추천`, `usagePlan` 으로 전수 검색해도 운영 코드에는 추천 엔진이 잡히지 않는다. `USAGE_PLAN_RECOMMENDATION` 이라는 기능 식별자는 `ai_feature_benefit_policy` 테이블에 **seed row 한 줄로만** 존재한다(`schema.sql:1034`). 즉 #28은 **"설계됨 · 데이터 준비됨 · 산출 로직 미구현"** 상태다. 이 페이지는 그 전제 위에서 "무엇이 준비됐고 무엇이 빠졌는지"를 설명한다.
:::

## 2. 왜 이렇게 설계했나 (의도·트레이드오프)

요금제 추천은 기술 난도보다 **신뢰 리스크**가 큰 기능이다. 사용자는 "AI가 결제를 강요한다"고 느끼는 순간 제품 전체를 의심한다. 그래서 설계 의도가 처음부터 보수적이다.

| 결정 | 이유 |
| --- | --- |
| **1차는 규칙 기반(rule-based)** | 사용량·잔액·한도는 명확한 숫자다. "18/20 도달 → 상위 플랜 안내" 같은 임계값 규칙이면 충분하고, 결과가 결정적(deterministic)이라 검증·디버깅이 쉽다. |
| **LLM은 설명 문구 초안에만 제한** | 추천 대상(어떤 플랜)은 규칙이 정하고, LLM은 그 결정을 부드러운 한국어 문장으로 풀어쓰는 보조 역할만 한다. "결제 강요처럼 보이지 않게" 톤을 다듬는 용도. |
| **"강요처럼 보이지 않게" 안내** | 분담 운영안이 #28 의도로 명시한 원칙. 안내는 사용자가 한도에 다가갈 때 도움이 되는 정보로 제시하고, 결제 버튼을 들이밀지 않는다. |
| **데이터를 먼저, 엔진을 나중에** | 추천 재료(집계·잔액·플랜·혜택)를 만드는 일과 추천을 산출하는 일을 분리. 재료가 모이는 단계까지 먼저 구현하고, 산출 로직은 이후 작업으로 남겼다. 그래서 지금 "재료는 있는데 엔진이 없는" 상태가 된 것. |

트레이드오프의 핵심은 **"LLM 전면 적용 vs 규칙 기반"** 이다. 채용·결제처럼 사용자에게 직접 영향을 주는 판단에서 LLM의 확률적 출력은 과장 추천(없는 한도를 말하거나 비싼 플랜으로 몰기)의 위험을 안는다. #28은 그 위험을 피해 **결정 = 규칙, 표현 = LLM**으로 책임을 쪼갰다.

## 3. 어떤 기술로 구현했나 (실제 클래스·테이블 근거)

추천 엔진은 없지만, **추천에 들어갈 재료를 만드는 코드는 전부 실재**한다. 면접에서 "엔진은 없어도 이 데이터는 다 있다"를 보여주는 게 핵심이다.

### 3-1. 재료 1 — 이번 달 사용량 집계

`BillingServiceImpl.getMonthlyUsage(userId)`(`BillingServiceImpl.java:119-123`)가 이번 달 1일 0시(`LocalDate.now().withDayOfMonth(1)`)부터의 `ai_usage_log`를 feature별로 집계한다. 실제 쿼리(`BillingMapper.xml:197-206`):

```sql
SELECT feature_type            AS featureType,
       COUNT(*)                AS used,
       COALESCE(SUM(credit_used), 0) AS creditUsed
  FROM ai_usage_log
 WHERE user_id = #{userId}
   AND created_at >= #{from}
 GROUP BY feature_type
 ORDER BY used DESC
```

결과는 `UsageRow(featureType, used, creditUsed)` 리스트(`UsageRow.java:13-18`). 즉 "이번 달에 어떤 기능을 몇 번, 크레딧 얼마를 썼는지"가 한 줄씩 나온다. `ai_usage_log`는 **전 영역 공동 사용량 로그**라, 영역 E뿐 아니라 모든 AI 기능 사용이 한 테이블에 쌓인다.

### 3-2. 재료 2 — 플랜과 혜택 정의

`listPlans()`(`BillingService.java:22`)가 요금제 마스터(`subscription_plan`)에 플랜별 혜택 정책(`subscription_benefit_policy`)을 붙여 내려준다. 가격·혜택 수량·초과 정책(`overage_policy`: BLOCK/CREDIT/UPGRADE)이 포함된다. "BASIC은 공고 분석 월 20회, 초과 시 크레딧" 같은 비교 기준이 여기서 나온다.

### 3-3. 재료 3 — 현재 잔액과 사용권 잔여

- `getMyBilling(userId)`(`:90-112`)가 현재 플랜·구독 상태·**보유 크레딧 잔액**(`creditBalance`)을 묶어 반환.
- `myBenefits(userId)`(`:130-144`)가 현재 기간의 **사용권 잔여량**(`user_benefit_balance`의 remaining)을 benefit_code별로 반환.

### 3-4. seed로만 존재하는 기능 식별자

`ai_feature_benefit_policy`에 `USAGE_PLAN_RECOMMENDATION` 행이 seed돼 있다(`schema.sql:1034`):

```text
('USAGE_PLAN_RECOMMENDATION', 'CAREER_STRATEGY', 'PER_REQUEST', 1, 1, 1)
```

즉 추천 기능 자체를 실행하면 benefit_code `CAREER_STRATEGY` 풀에서 1회 차감하도록 **과금 정책만 미리 정의**해 뒀다(첨삭 4종이 `CORRECTION` 풀을 쓰는 것과 다른, 별도 혜택 코드). 하지만 이 정책을 소비할 **실행 코드가 없다.**

:::tip 면접 포인트: "데이터는 다 있다"
세 가지 재료(`getMonthlyUsage` · `listPlans` · `getMyBilling`/`myBenefits`)만 조합하면 "이번 달 사용량 vs 플랜 한도 vs 잔액"이라는 추천의 입력이 완성된다. 빠진 것은 이 셋을 묶어 **"그래서 어떤 플랜"** 을 도출하는 한 개의 서비스뿐이다. 면접에서 이 점을 짚으면 "구현 갭을 정확히 이해하고 있다"는 신호가 된다.
:::

## 4. 동작 원리 (설계된 흐름 · 표 · 작은 코드)

추천 엔진이 들어간다면 **이미 있는 재료를 조립**하는 형태가 된다. 설계 의도(규칙 1차 + LLM 문구)를 코드 모양으로 추상화하면:

```text
1) 재료 수집
   usage   = getMonthlyUsage(userId)   // feature별 used/creditUsed
   plans   = listPlans()               // 플랜·혜택·한도
   balance = getMyBilling(userId)       // 현재 플랜 + 크레딧 잔액
   tickets = myBenefits(userId)         // 사용권 잔여

2) 규칙 평가 (deterministic)
   for each feature:
     한도 대비 사용률 = used / 플랜 한도
     if 사용률 >= 임계값(예 0.8)  → "상위 플랜 후보" 가점
   if 크레딧 잔액 < 임박 소비량   → "충전 필요" 플래그

3) 결정
   추천 = 규칙이 가장 높은 점수를 준 플랜 or "충전 N크레딧"

4) 표현 (LLM 보조, 선택)
   문구 = LLM("이 결정을 강요처럼 안 보이게 1~2문장으로")
```

핵심은 **2~3단계(결정)는 규칙이, 4단계(표현)만 LLM이** 한다는 것. 현재 프론트가 보여주는 사용량 시각화는 이 흐름의 1단계 결과를 그대로 그린 것에 가깝다.

### 4-1. 현재 화면에 살아 있는 "추천의 1단계"

`Billing.tsx`의 **AI 사용량 탭**(`usage`)은 실제 API(`getMonthlyUsage`)를 호출해 이번 달 사용량을 그린다(`Billing.tsx:217-265`). 이게 #28 데이터가 실제로 흐르는 유일한 화면이다.

| 화면 요소 | 근거 | 의미 |
| --- | --- | --- |
| feature별 사용 막대 | `usage.map(...)` + `Progress`(`:228-238`) | 기능별 used를 화면 내 최대값 대비 **상대 비율**로 표시 |
| 한글 라벨 매핑 | `FEATURE_LABEL` 12종(`:29-42`) | `JOB_ANALYSIS` → "공고 분석" 등 |
| 보유 크레딧 카드 + "충전하러 가기" | `:247-263` | 잔액 노출 + 충전 탭으로 유도 (= 가장 단순한 "충전 안내") |

여기서 막대 길이는 **플랜 한도 대비**가 아니라 **이번 달 내 기능 간 상대비율**이다. 진짜 추천이라면 "한도 대비 몇 %"가 돼야 하는데, 현재는 한도와 묶는 로직이 없어 상대비율로만 그린다. 이 차이가 "데이터 시각화"와 "추천"의 경계다.

### 4-2. 마케팅 쪽의 정적 비교표

`Pricing.tsx`(`/pricing`)는 `getFeatureBenefitPolicies`로 기능별 크레딧 소모량 표를 그리고(`Pricing.tsx:74-81, 224-239`), 플랜 비교표를 보여준다. 단 모든 CTA는 결제를 직접 하지 않고 `/billing?tab=...`으로 위임한다. 추천이 아니라 **정적 비교**다 — 사용자 데이터를 보고 "당신에겐 이것"이라고 말하지 않는다.

## 5. 구현 상태 (됨 vs 계획) — 정직한 구분

| 항목 | 상태 | 근거 |
| --- | --- | --- |
| 이번 달 사용량 집계 API | ✅ 구현 | `getMonthlyUsage` + `BillingMapper.monthlyUsage` |
| 플랜·혜택·잔액·사용권 조회 API | ✅ 구현 | `listPlans` / `getMyBilling` / `myBenefits` |
| 사용량 시각화 화면 | ✅ 구현 | `Billing.tsx` usage 탭 (실 API 연동) |
| 기능별 크레딧 소모량 비교표 | ✅ 구현 | `Pricing.tsx` feature policy 표 |
| `USAGE_PLAN_RECOMMENDATION` 과금 정책 | ✅ seed만 | `schema.sql:1034` (benefit_code=`CAREER_STRATEGY`) |
| **추천 산출 엔진(서비스/컨트롤러)** | ❌ **미구현** | 운영 코드에 `recommend`/추천 로직 없음 |
| **규칙 임계값 평가 로직** | ❌ 미구현 | 한도 대비 사용률을 계산하는 코드 없음 |
| **LLM 설명 문구 생성** | ❌ 미구현 | 문구 생성 호출 지점 없음 |
| 추천 결과를 보여줄 전용 UI | ❌ 없음 | usage 탭은 시각화일 뿐 추천 카드 아님 |

:::details 왜 "seed row만 있는데 구현됐다고 착각하기 쉬운가"
`ai_feature_benefit_policy`에 `USAGE_PLAN_RECOMMENDATION` 행이 있으면, 과금 엔진 입장에서는 "이 기능을 실행하면 CAREER_STRATEGY 1장 차감"이라는 **정책이 준비**된 것처럼 보인다. 하지만 정책은 "기능이 실행되면 어떻게 과금할지"를 정의할 뿐, **기능 자체를 실행하지는 않는다.** 추천을 산출하고 화면에 띄우는 코드가 0줄이므로 이 정책은 한 번도 발동되지 않는다. 정책 seed의 존재 ≠ 기능의 구현.
:::

추가로, 영역 E 전체의 알려진 갭과 맞물린다: AI 기능 실행과 실제 과금(크레딧·사용권 차감)이 운영 경로에 **아직 배선되지 않았다.** 추천 기능이 만들어지더라도, 그 추천이 의존하는 "사용량 → 비용" 환산이 실제 차감으로 이어지는지는 별도 배선이 필요하다(차감 엔진 자체는 완성·테스트 통과 상태).

## 6. 면접 답변 3단계

1. **무엇 (1문장):** "#28은 사용자의 이번 달 사용량·크레딧 잔액·사용권 잔여를 모아 맞는 요금제나 충전을 안내하는 기능입니다."
2. **어떻게 (설계):** "추천 대상은 규칙으로 결정하고, LLM은 그 결정을 강요처럼 보이지 않게 풀어쓰는 보조로만 씁니다. 추천 재료인 사용량 집계(`getMonthlyUsage`)·플랜 한도(`listPlans`)·잔액(`getMyBilling`)은 모두 API로 준비돼 있습니다."
3. **정직한 상태:** "다만 이 재료들을 묶어 추천을 산출하는 엔진은 아직 구현 전입니다. 정책 식별자(`USAGE_PLAN_RECOMMENDATION`)는 테이블에 seed돼 있지만 실행 코드가 없어서, 현재는 사용량 시각화까지만 화면에 살아 있고 '당신에겐 PRO' 같은 판단은 나오지 않습니다."

## 7. 꼬리질문 + 모범답안

**Q1. "추천 엔진이 없는데 #28을 어떻게 면접에서 설명하나요?"**
A. 기능을 "데이터 파이프라인"과 "판단 로직" 두 층으로 나눠 설명합니다. 데이터 층(집계·잔액·플랜·혜택)은 전부 구현돼 실제로 화면에 흐릅니다. 판단 층(규칙 평가 + 문구 생성)만 비어 있습니다. 이렇게 나누면 "무엇이 준비됐고 무엇이 남았는지"를 정확히 전달할 수 있고, 과장 없이 정직하게 말하는 게 오히려 신뢰를 줍니다.

**Q2. "왜 추천에 LLM을 전면적으로 안 쓰나요?"**
A. 결제·요금제 판단은 과장 추천 리스크가 큽니다. LLM이 없는 한도를 말하거나 비싼 플랜으로 몰면 사용자 신뢰가 깨집니다. 그래서 결정은 결정적인 규칙(사용률 임계값 등)이 내리고, LLM은 그 결정을 부드럽게 문장화하는 표현 역할로 제한했습니다. 책임을 결정/표현으로 쪼갠 설계입니다.

**Q3. "사용량은 어디서 어떻게 집계하나요?"**
A. 공동 테이블 `ai_usage_log`를 이번 달 1일 기준으로 `feature_type` GROUP BY 해서 `COUNT(*)`와 `SUM(credit_used)`를 뽑습니다(`getMonthlyUsage`). 영역 E뿐 아니라 모든 영역의 AI 사용이 이 한 테이블에 쌓이므로, 한 사용자의 전사 사용 패턴을 한 쿼리로 볼 수 있습니다.

**Q4. "추천이 보여주는 사용량 막대가 진짜 한도 대비 사용률인가요?"**
A. 아닙니다. 현재 usage 탭의 막대는 이번 달 **기능 간 상대비율**(화면 내 최대 used 대비)입니다. 플랜 한도와 묶는 로직이 없어서 그렇습니다. 진짜 추천이라면 "BASIC 한도 20회 중 18회"처럼 한도 대비로 그려야 하는데, 그 결합이 미구현이라 상대비율로만 표시합니다. 이 차이가 "시각화"와 "추천"의 경계입니다.

**Q5. "`USAGE_PLAN_RECOMMENDATION` 정책 seed는 무슨 역할인가요?"**
A. 추천 기능이 실행될 때 어떻게 과금할지를 미리 정의한 과금 정책입니다. benefit_code `CAREER_STRATEGY` 풀에서 PER_REQUEST 1장을 차감하도록 돼 있습니다(첨삭 4종이 쓰는 `CORRECTION` 풀과는 별개). 다만 정책은 "실행되면 어떻게"만 정의하고 기능을 실행하지는 않습니다. 추천을 산출·노출하는 코드가 없으므로 이 정책은 발동되지 않습니다.

**Q6. "추천 엔진을 지금 붙인다면 어디부터 손대겠어요?"**
A. billing 도메인에 `getMonthlyUsage`·`listPlans`·`getMyBilling`을 주입받는 추천 서비스를 하나 추가합니다. 그 안에서 (1) 기능별 사용률 = used / 플랜 한도 계산 → (2) 임계값 초과 feature에 가점 → (3) 최고 점수 플랜 또는 "충전 N크레딧" 결정 → (4) 선택적으로 LLM에 문구 초안 요청, 순서로 조립합니다. 재료 API가 이미 있어 신규 쿼리 없이 조합만으로 1차 규칙 버전을 만들 수 있습니다.

## 8. 직접 말해보기

- "규칙으로 결정, LLM으로 표현"이라는 책임 분리와 그 이유(과장 추천 리스크)를 설명한다.
