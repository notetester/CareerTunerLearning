# 사용량 기반 요금제 추천 [#28]

> #28은 구현된 결정론적 추천 기능이다. 이번 달 AI 사용량, 크레딧 잔액, 현재 요금제와 판매 중인 상품을 조합해 `UPGRADE_PLAN`, `BUY_CREDITS`, `KEEP` 중 하나를 반환한다. LLM은 호출하지 않는다.

## 사용자에게 무엇을 보여주나

`GET /api/billing/plan-recommendation`은 다음을 함께 반환한다.

- 추천 종류와 한 줄 설명
- 현재 요금제와 크레딧 잔액
- 이번 달 전체 AI 호출 수와 사용 크레딧
- 가장 많이 사용한 기능
- 판단 근거 목록
- 추천 요금제 또는 충전 상품

프런트 `Billing.tsx`는 로그인 사용자의 결제 정보를 불러올 때 이 API도 호출하고, 추천 종류에 맞는 카드와 이동 버튼을 표시한다.

## 왜 LLM을 쓰지 않았나

요금제 추천은 숫자로 검증할 수 있는 결제 인접 판단이다. 확률적 모델이 더 비싼 상품을 과장 추천하거나 존재하지 않는 혜택을 설명하면 사용자 신뢰를 해친다. 그래서 추천 대상과 문구 모두 서버 규칙으로 고정했다.

이 선택의 장점은 다음과 같다.

- 같은 입력에는 같은 결과가 나온다.
- 추천 근거를 그대로 사용자와 운영자에게 노출할 수 있다.
- 외부 모델 장애나 비용에 영향을 받지 않는다.
- 추천 분기와 상품 선택을 단위 테스트하기 쉽다.

LLM을 쓰지 않는 기능도 제품 분류상 AI 기능일 수 있다. 중요한 것은 모델 호출 여부가 아니라 사용자 데이터를 바탕으로 다음 행동을 돕는 제품 역할이다.

## 실제 구현

핵심 코드는 `BillingServiceImpl.recommendPlan(userId)`이다.

```text
getMonthlyUsage(userId)
  -> monthlyCalls, monthlyCredits, topFeature 계산

getMyBilling(userId)
  -> currentPlan, creditBalance 조회

listPlans()
  -> 월 가격 오름차순 정렬, 다음 상위 요금제 탐색

규칙 평가
  -> UPGRADE_PLAN | BUY_CREDITS | KEEP
```

### 규칙 1: 상위 요금제

이번 달 AI 사용이 15회 이상이고 현재보다 비싼 활성 요금제가 있으면 `UPGRADE_PLAN`을 반환한다. 추천 상품은 월 가격 기준으로 가장 가까운 상위 플랜이다.

### 규칙 2: 크레딧 충전

상위 요금제 조건에 해당하지 않고, 크레딧이 5개 이하이며 이번 달 AI 사용이 3회 이상이면 `BUY_CREDITS`를 반환한다.

충전 상품은 다음 목표량을 충족하는 가장 작은 상품이다.

```text
target = max(30, 이번 달 소모 크레딧)
```

목표를 충족하는 상품이 없으면 가장 큰 상품을 선택한다.

### 규칙 3: 유지

나머지는 `KEEP`이다. 사용 기록이 없으면 현재 요금제로 시작해도 충분하다고 안내하고, 기록이 있더라도 임계값에 닿지 않으면 굳이 올리지 않아도 된다고 명시한다.

## 판단 순서가 중요한 이유

상위 요금제 조건을 먼저 평가한 뒤 충전을 본다. 과사용자가 낮은 크레딧을 갖고 있을 때 두 추천이 동시에 가능한데, 장기 반복 사용에는 상위 플랜 검토를 먼저 제시하려는 정책이다. 이 우선순위도 모델이 아니라 코드가 소유한다.

## API 계약

`PlanRecommendationResponse`의 핵심 필드는 다음과 같다.

| 필드 | 의미 |
| --- | --- |
| `recommendation` | `UPGRADE_PLAN`, `BUY_CREDITS`, `KEEP` |
| `monthlyUsageCount` | 이번 달 AI 호출 전체 합 |
| `monthlyCreditUsed` | 이번 달 사용 크레딧 합 |
| `topFeatureType` | 가장 많이 쓴 기능, 사용 없으면 `null` |
| `headline` | 규칙으로 만든 사용자 안내 |
| `reasons` | 사용 횟수·잔액 등 결정 근거 |
| `recommendedPlan` | 업그레이드일 때만 존재 |
| `recommendedCreditPack` | 충전일 때만 존재 |

서버 응답은 `ApiResponse<PlanRecommendationResponse>` envelope로 감싼다.

## 프런트 연결

`frontend/src/features/billing/api/billingApi.ts`의 `getPlanRecommendation()`이 API를 호출한다. `Billing.tsx`는 사용자 결제·사용량 데이터를 가져올 때 추천도 함께 불러오고 다음 UI를 분기한다.

| 추천 | UI |
| --- | --- |
| `UPGRADE_PLAN` | 상위 플랜 이름과 요금제 보기 버튼 |
| `BUY_CREDITS` | 충전 상품 이름과 충전 버튼 |
| `KEEP` | 현재 요금제 유지 안내 |

추천 API 실패는 결제 페이지 전체를 깨뜨리지 않고 카드만 숨긴다. 추천은 보조 기능이므로 플랜 조회·결제 이력 같은 핵심 화면과 장애 경계를 분리했다.

## 과금 여부

추천 API의 비용은 0이고 LLM도 호출하지 않는다. 사용량을 설명하기 위한 조회가 다시 잔액을 줄이는 순환을 피한 것이다.

## 면접 답변

> "요금제 추천은 결제에 가까운 판단이라 LLM에 맡기지 않았습니다. `BillingServiceImpl.recommendPlan`이 월 사용량 15회 이상이면 다음 상위 플랜, 크레딧 5개 이하이면서 3회 이상 사용했으면 월 소비량을 덮는 최소 충전 상품, 그 외에는 유지 권고를 반환합니다. API와 Billing 추천 카드까지 연결돼 있고, 과장 추천을 막기 위해 KEEP을 기본값으로 둔 결정론 기능입니다."

## 꼬리질문

### 사용량이 많고 크레딧도 적으면 무엇을 추천합니까?

상위 플랜 규칙을 먼저 평가하므로 가능한 다음 플랜이 있으면 `UPGRADE_PLAN`이다. 상위 플랜이 없거나 과사용 임계값 미만일 때 낮은 잔액 규칙을 평가한다.

### 왜 개인별 예측 모델을 학습하지 않았습니까?

현재 입력은 사용 횟수·잔액·가격처럼 작은 정형 데이터이고, 결제 강권의 위험이 모델 이득보다 크다. 충분한 장기 전환 데이터와 오프라인 평가 기준이 생기기 전에는 해석 가능한 규칙이 더 적절하다.

### 추천이 곧 자동 결제입니까?

아니다. 추천은 카드와 이동 버튼만 제공한다. 결제는 사용자의 명시적 선택과 별도의 ready/confirm 계약을 거친다.

### 임계값을 바꾸려면 무엇을 주의해야 합니까?

현재 임계값은 서비스 상수다. 변경할 때는 기존 분기 테스트, 상품이 하나뿐인 경우, 사용 0회, 상품 목록 없음, 최고 요금제 사용자 같은 경계를 함께 검증해야 한다.
