# AI 사용량과 크레딧 흐름

> 모델 실행 기록과 경제적 차감은 구분하되 성공 요청의 참조 키로 연결한다. 실패는 관측하되 과금하지 않고, 재시도는 두 번 차감하지 않는다.

## 전체 흐름

```text
사용자 AI 실행
  -> 비용 preview와 정책 동의
  -> 서버 preflight
  -> provider 실행
       실패 -> FAILED ai_usage_log, 차감 없음
       성공 -> payload 검증
                 -> SUCCESS ai_usage_log
                 -> 도메인 결과 저장
                 -> 사용권 또는 크레딧 차감
                 -> 거래 장부
  -> Billing 월 사용량 집계
  -> 추천 카드와 잔액 표시
```

## 왜 로그와 장부를 나눴나

`ai_usage_log`는 기술적 실행을 설명한다.

- 기능 타입
- 모델/provider
- token 사용량
- 성공·실패
- 오류와 실행 시각

`credit_transaction`과 `benefit_transaction`은 잔액 변화를 설명한다. 실패 호출은 운영에 중요해 로그는 필요하지만 돈이 움직이지 않았으므로 장부에는 AI 차감이 없어야 한다.

## 비용 고지

프런트는 서버 preview를 조회해 예상 사용권 또는 크레딧 범위를 보여 준다. acknowledgement key와 기능 식별자를 요청에 포함하고 서버가 실행 직전 다시 검증한다.

실제 token 수는 응답 뒤에만 확정되므로 preview와 actual settlement를 분리한다.

## 성공 정산

기능 정책이 사용권을 포함하면 먼저 `user_benefit_balance`를 소비한다. 소진됐고 초과 정책이 CREDIT 계열이면 `users.credit`을 조건부 차감한다. token 단위 정책은 min/max 범위 안에서 비용을 계산한다.

첨삭은 결과 행·SUCCESS 로그·actual charge를 한 트랜잭션으로 묶는다. 공통 helper는 acknowledgement된 HTTP 요청의 첫 성공 usage만 정산해 한 요청에서 여러 내부 로그가 생겨도 중복 차감을 줄인다.

## 재조회와 재생성

저장된 결과를 단순 조회하는 것은 새 모델 실행이 아니므로 새 사용량 로그와 차감이 없다. 사용자가 명시적으로 다른 모델이나 새 분석을 실행하면 새 요청과 비용 고지를 거친다.

동일 네트워크 요청의 retry는 같은 request key를 replay한다. “다시 보기”, “전송 실패 retry”, “새 모델 재생성”을 구분해야 한다.

## Billing 집계

월 1일부터 현재까지 `ai_usage_log`를 feature별로 묶어 호출 수와 `credit_used` 합을 보여 준다. Progress는 월 한도 소진율이 아니라 가장 많이 쓴 기능 대비 상대 비율이다.

## 추천

같은 월 사용량, 현재 플랜과 잔액으로 결정론 추천을 만든다.

- 과사용: 다음 상위 플랜 검토
- 활동 중 저잔액: 충전 상품 검토
- 그 외: KEEP

추천 조회는 무료이고 결제를 자동 승인하지 않는다.

## 관리자 경계

AI 사용량 조회는 세부 읽기 권한, 정책 변경과 환불·크레딧 조정은 각각 쓰기/수정 권한을 요구한다. 메뉴도 읽기 권한이 없으면 노출하지 않고 API는 fail-closed로 다시 검사한다.
