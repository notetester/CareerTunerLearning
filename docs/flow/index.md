# 흐름 개요 — 6개 영역이 어떻게 연결되나

> 영역별로는 다 봤다. 이제 그 6개가 **하나의 지원 건(Application Case)** 위에서 어떻게 맞물려 돌아가는지, 즉 "전체 그림"을 잡는 섹션이다.

영역 A~F 페이지가 "각 부품이 무엇을 하나"를 다룬다면, 이 흐름 섹션은 **"부품들이 어떻게 한 제품으로 연결되나"** 를 다룬다. 면접에서 진짜 변별력이 갈리는 건 개별 기능 설명이 아니라 **"이 데이터가 어디서 와서 어디로 가고, 왜 그렇게 경계를 그었는가"** 다. 그걸 흐름으로 설명할 수 있으면 "프로젝트를 통째로 이해한 사람"으로 보인다.

## 1. 왜 흐름 섹션이 따로 필요한가

영역별 학습은 깊이를 주지만, 면접관이 던지는 단골 질문은 영역 경계를 가로지른다.

- "전체 아키텍처를 1~2분으로 설명해보세요." → 한 영역으로는 답이 안 됨
- "사용자가 처음 들어와서 면접 준비를 끝낼 때까지 데이터가 어떻게 흐르나요?" → A→F 전 구간
- "AI 기능이 30개가 넘는다는데, 서로 어떻게 의존하나요?" → 의존 그래프
- "여러 사람이 나눠 짰는데 경계는 어떻게 지켰나요?" → 소유권 맵

이 질문들은 **개별 기능을 아무리 깊게 알아도** 흐름 관점이 없으면 답이 흩어진다. 그래서 이 섹션은 영역 상세를 **반복하지 않고**, 연결·여정·경계만 압축해서 다룬다. 세부가 필요하면 그때그때 해당 영역 페이지로 링크한다.

## 2. 6영역 한 장 요약

CareerTuner는 6명이 **수직 분담**한 6개 영역이고, 각자가 `사용자 화면 + REST API + 관리자 + AI + DB 테이블`을 통째로 소유한다. AI 기능 번호(#1~34)도 영역별로 끊어 가진다.

| 영역 | 한 줄 정체성 | 핵심 소유 테이블 | AI # | 다른 영역에 주는 것 | 상세 |
| --- | --- | --- | --- | --- | --- |
| **A** | 기반 신뢰 데이터(인증·프로필) 원천 | `users`, `user_profile`(+버전 스냅샷) | 1~5 | 프로필 스냅샷 → B·C·D·E | [/area-a/](/area-a/) |
| **B** | 지원 건·공고 구조화의 시작점 | `application_case`, `job_posting`, `job_analysis`, `company_analysis` | 6~11 | 공고 분석·면접포인트 → C·D·E | [/area-b/](/area-b/) |
| **C** | 적합도·전략 (여정의 중심 허브) | `fit_analysis`, `career_analysis_run` | 12~18 | 적합도 점수·근거 → D | [/area-c/](/area-c/) |
| **D** | 면접 세션·질문·평가 | `interview_session/question/answer`, `interview_agent_step` | 19~23 | 답변·평가 → E·C | [/area-d/](/area-d/) |
| **E** | 첨삭 + 결제·크레딧 원장 | `correction_request`, `payment`, `credit_transaction` | 24~28 | (원장 분리, 사용량 집계 소비) | [/area-e/](/area-e/) |
| **F** | 커뮤니티·고객센터·인테이크 챗봇 | `community_*`, `support_ticket`, `notification` | 29~34 | 실제 면접질문 → D, 슬롯 수집 → 오케스트레이터 | [/area-f/](/area-f/) |

데이터 규모는 기준 커밋의 canonical `schema.sql`에서 **172개 테이블**이다. 허브 테이블은 사용자 축의 `users`(A)와 지원 건 축의 `application_case`(B)이며, 숫자는 [문서 기준선](/project/source-baseline)의 SHA와 함께 갱신한다.

:::tip 가장 중요한 한 문장
**핵심 단위는 "공고"가 아니라 "지원 건(Application Case)"이다.** 프로필·공고분석·적합도·면접·첨삭·결제·후기가 전부 하나의 지원 건에 매달린다. 면접에서 이 한 문장으로 시작하면 나머지 설명이 전부 이 축에 정렬된다.
:::

## 3. 지원 건 중심의 큰 그림

### 3-1. 사용자 여정 (A→F를 가로지름)

```text
   A          B           C            D            E
프로필 ──▶ 공고분석 ──▶ 적합도 ──▶ 예상질문/면접 ──▶ 첨삭/결제
스펙 원천   공고 구조화   (중심허브)   답변·평가      개선·크레딧
   │                       ▲              │            │
   │  프로필 스냅샷         │              │            │
   └──(전 영역 공통입력)───┘              ▼            ▼
                            ◀───── 장기경향(C#16) ◀── 사용량/재분석
                                  (D 답변평가 + E 결과가 다시 C로 순환)

   F  커뮤니티 후기 ── 실제 면접질문 추출(#31) ──▶ D 참고 데이터 (여정 바깥에서 교차·순환)
```

화살표의 의미: **A→B** 스펙으로 공고를 분석, **B→C** 공고 요건과 스펙을 비교, **C→D** 적합도를 면접 질문에 반영, **D→E** 면접 답변을 첨삭, **E⇢C** 사용량·재분석이 다시 적합도/경향으로 순환, **F 교차** 후기가 면접 참고 데이터로 들어옴. C가 중심 허브이고, F는 여정 바깥에서 순환 데이터를 공급한다.

### 3-2. 대표 화면 8단계

| 단계 | 화면(라우트) | 영역 | 받는 입력 | 내놓는 산출 |
| --- | --- | --- | --- | --- |
| 01 | 프로필 `/profile` | A | 이력서·경력·자소서 | 프로필 스냅샷(전 영역 공통 입력) |
| 02 | 공고분석 `/applications/1/job-analysis` | B | 공고 원문·OCR | 공고 구조화(필수/우대/담당업무/기업) |
| 03 | 적합도 `/applications/1/fit` | C | A 스냅샷 + B 필수/우대 | 점수 + 근거 + 다음 행동 |
| 04 | 예상질문 `/interview?tab=questions` | D | B 공고·면접포인트 + C 적합도 | 직무/프로젝트/인성/기업 질문 |
| 05 | 리포트 `/interview?tab=report` | D | 세션·답변 평가 | 총평·강점·약점·과제 |
| 06 | 첨삭 `/correction?tab=answer` | E | D 답변 + A 원문 | 원문 비수정 개선안(크레딧 소모) |
| 07 | 사용량·결제 `/billing?tab=usage` | E | AI 사용량·크레딧 | 요금제·충전 추천 |
| 08 | 커뮤니티 `/community` | F | 후기·태그 | 후기 요약(→ B 참고로 순환) |

핵심은 **03 적합도가 "점수만"이 아니라 "점수 + 근거 + 다음 행동"을 함께 준다는 것**, 그리고 **06 첨삭이 원문을 절대 덮어쓰지 않고 별도 저장한다는 것**이다. 이 두 가지가 흐름 전체의 설계 철학을 압축한다.

### 3-3. 두 갈래 진입 — 수동 여정 vs AutoPrep

위 8단계는 사용자가 화면을 직접 밟는 **수동 경로**다. CareerTuner에는 이걸 한 줄로 압축하는 **AI 오케스트레이터(AutoPrep)** 진입이 따로 있다.

```text
"네이버 백엔드 신입 통째로 준비해줘"
   │
   ▼  F 인테이크 챗봇이 슬롯(지원건·모드) 수집
   ▼  두뇌(플래너)가 한 줄을 해석 → 실행계획
   ▼  오케스트레이터가 6파트를 의존 그래프대로 병렬 실행
   ▼  SSE로 6파트 타임라인이 실시간으로 채워짐
```

AutoPrep는 **새 AI를 만들지 않는다.** A~F의 기존 도메인 서비스를 그대로 호출(래핑)해 묶을 뿐이다. 전체 메커니즘은 [/flow/ai-orchestrator](/flow/ai-orchestrator)에서 깊게 다룬다.

## 4. 이 섹션을 어떻게 읽을 것인가 (권장 학습순서)

흐름 페이지는 "넓게 → 좁게" 순으로 쌓는 게 가장 효율적이다. 아래 순서대로 읽으면 면접에서 "전체 → 세부 → 다시 전체"로 자유롭게 줌인/줌아웃할 수 있게 된다.

1. **[흐름 개요](/flow/)** (지금 이 페이지) — 6영역 지도와 큰 그림
2. **[전체 아키텍처](/flow/architecture)** — 4계층·`ApiResponse` 규약·JWT 인증 경계·요청 생명주기
3. **[사용자 end-to-end 여정](/flow/user-journey)** — 8단계 화면을 따라가는 입력/산출 체인
4. **[지원 건 중심 흐름](/flow/application-case)** — 왜 "지원 건"이 핵심 단위인가, 의존 체인
5. **[데이터 소유권 경계 맵](/flow/data-ownership)** — 누가 무엇을 소유하고, 어디까지 읽기전용인가
6. **[AI 오케스트레이터 전체](/flow/ai-orchestrator)** — 인테이크→플래너→의존그래프→SSE
7. **[AI 기능 #1-34 맵](/flow/ai-function-map)** — 34개 AI 기능의 입출력 의존 관계
8. **[AI 공급자·폴백 전략](/flow/ai-providers-fallback)** — 자체 OSS→Haiku→OpenAI→Mock과 영역 편차
9. **[크레딧·사용량 흐름](/flow/credit-usage)** — 성공 기준 차감·멱등·요금제 게이팅
10. **[팀 협업·시스템 경계](/flow/team-collaboration)** — 수직 분담·공통영역 Owner·승인 규칙
11. **[프로젝트 전체 면접 플레이북](/flow/interview-whole-project)** — 위 전부를 한 답변으로 엮는 연습

면접 직전이라면 **[전체 아키텍처](/flow/architecture) → [지원 건 중심 흐름](/flow/application-case) → [프로젝트 전체 면접 플레이북](/flow/interview-whole-project)** 만 읽어도 "전체를 설명해보세요"에 대응할 골격이 선다.

## 5. 흐름을 관통하는 4가지 설계 원칙

세부 페이지로 흩어지기 전에, 6영역을 하나로 묶는 **불변 원칙 4개**를 먼저 머리에 박아두면 어느 질문이 와도 같은 축에서 답할 수 있다.

| 원칙 | 한 줄 | 근거 위치 |
| --- | --- | --- |
| **지원 건이 중심 축** | 모든 데이터가 하나의 Application Case에 매달린다 | [/flow/application-case](/flow/application-case) |
| **소유는 1곳, 참조는 읽기전용** | 각 영역이 자기 결과 테이블을 소유하고, 타 영역 원본은 비수정 참조 | [/flow/data-ownership](/flow/data-ownership) |
| **점수·판단은 서버 규칙(뉴로-심볼릭)** | LLM은 설명·생성만, 점수·신뢰도·분류는 결정적 코드가 확정 | [/area-c/fit-analysis](/area-c/fit-analysis) |
| **크레딧은 성공 시에만, 멱등하게** | 실패한 AI는 미차감, 같은 호출은 두 번 차감 안 됨 | [/flow/credit-usage](/flow/credit-usage) |

이 4개를 "왜 그렇게 했나"까지 말할 수 있으면, 꼬리질문 대부분은 이 원칙들의 변주다. 예컨대 "AI가 다 죽으면 화면이 깨지나요?"는 세 번째(서버 규칙 폴백)와 네 번째(실패 미차감)로 답한다.

## 6. 면접 답변 — "프로젝트 전체를 설명해보세요"

> CareerTuner는 채용공고에 맞춰 스펙과 면접 답변을 조정해주는 AI 취업 전략 플랫폼입니다. 핵심 설계는 모든 데이터의 단위를 **"공고"가 아니라 "지원 건(Application Case)"** 으로 잡은 것입니다. 한 사람이 같은 회사에 다시 지원할 수도 있고, 여러 회사를 동시에 준비하기도 하니까요.
>
> 데이터는 **A의 프로필**에서 시작합니다. A가 인증과 프로필이라는 기반 데이터를 소유하고, 이 프로필 스냅샷을 B~E 전 영역이 읽기전용으로 가져갑니다. **B**가 채용공고를 구조화(필수/우대/담당업무/기업)하면, **C**가 그 요건과 프로필을 비교해 적합도 점수와 보완 전략을 냅니다. C가 여정의 중심 허브예요. **D**는 그 적합도와 공고를 바탕으로 면접 질문을 만들고 답변을 평가하고, **E**가 그 답변과 원문을 첨삭합니다. E는 결제·크레딧 원장도 소유하고요. **F**는 커뮤니티 후기에서 실제 면접질문을 뽑아 D로 다시 흘려보냅니다.
>
> 기술적으로는 Spring Boot 4 · Java 21 · MyBatis · MySQL 8 백엔드와 React SPA를 `ApiResponse` 단일 규약과 STATELESS JWT로 묶었습니다. 6명이 영역을 **수직 분담**해서, 각자 화면·API·관리자·AI·DB 테이블을 통째로 소유하고 **타 영역 원본은 읽기전용으로만 참조**하는 경계 규칙을 지켰습니다.
>
> 마지막으로 AI 오케스트레이터가 있습니다. "○○ 통째로 준비해줘" 한 줄을 받으면, 6영역의 기존 AI 기능을 **의존 그래프대로 병렬 실행**해서 결과를 실시간으로 보여줍니다. 새 AI를 만든 게 아니라 기존 도메인 서비스를 지휘하는 구조입니다.

이 답변의 장점은 **"지원 건 → A에서 시작 → C가 허브 → 순환 → 오케스트레이터로 마무리"** 라는 한 축으로 6영역을 전부 꿰었다는 것이다. 면접관이 어느 영역을 더 파고들어도 이 축으로 되돌아올 수 있다.

## 7. 꼬리질문 + 모범답안

:::details "6명이 나눠 짰으면 통합이 깨지기 쉬운데, 어떻게 막았나요?"
두 가지 계약으로 막았습니다. 첫째, **데이터 소유권 규칙** — 각 영역이 자기 결과 테이블만 쓰고, 타 영역 원본은 읽기전용으로만 참조합니다. 예를 들어 E의 첨삭은 원문을 절대 덮어쓰지 않고 `correction_request`에 별도 저장합니다. 둘째, **공통영역 Owner 규칙** — `schema.sql`, 라우팅, 인증/권한, AI 프롬프트 공통 엔진 같은 공유 자원은 팀장 승인 없이 못 고칩니다. 실제로 오케스트레이터가 공통영역을 건드린 건 SSE 비동기 재디스패치 401을 막는 SecurityConfig 한 줄뿐입니다. 자세한 경계는 [/flow/team-collaboration](/flow/team-collaboration)에 있습니다.
:::

:::details "AI 기능이 34개라는데, 서로 어떻게 의존하나요?"
번호로 끊어 가지지만 의존선이 분명합니다. A의 프로필 요약(#1)이 C의 적합도(#12)로 들어가고, B의 필수조건 추출(#7)이 C 적합도의 기준이 됩니다. C 적합도(#12)는 다시 D의 예상 질문(#19)에 쓰이고, D의 답변 평가(#22)는 E의 첨삭(#24)과 C의 장기경향(#16)으로 순환합니다. F의 실제 면접질문 추출(#31)은 D의 참고 데이터로 들어가고요. 전체 의존 맵은 [/flow/ai-function-map](/flow/ai-function-map)에 표로 정리돼 있습니다.
:::

:::details "그럼 적합도 점수는 LLM이 매기나요?"
아니요, 그게 핵심 설계 포인트입니다. **LLM은 근거 텍스트와 설명만 생성하고, 점수·신뢰도·조건 매트릭스 severity는 서버의 결정적 규칙엔진이 계산**합니다. 입력 상태가 같으면 mock이든 실제 AI든 같은 점수가 나오게 했어요. 재현성·설명가능성·비용·가용성을 동시에 잡는 뉴로-심볼릭 구조입니다. 덕분에 LLM이 전부 실패해도 규칙엔진 Mock이 받쳐서 화면이 안 깨집니다. 상세는 [/area-c/fit-analysis](/area-c/fit-analysis)에 있습니다.
:::
