# 데이터 생명주기: 생성부터 소프트 삭제까지

> 기능 사이의 회색지대는 “만드는 기능”보다 “수정·삭제·재활성·계정 탈퇴 뒤 무엇이 남는가”에서 자주 깨진다. 이 페이지는 A~F가 공유하는 데이터 생명주기를 설명한다.

## 데이터 종류별 정책

모든 행을 같은 방식으로 보존하지 않는다.

| 종류 | 기본 정책 | 이유 |
| --- | --- | --- |
| 사용자 생성 콘텐츠·업무 결과 | `deleted_at` 소프트 삭제 | 감사·복구·연결 데이터 정합성 |
| 좋아요·스크랩·관계 | 소프트 삭제 후 재활성 | UNIQUE 키를 지키며 재행동 복원 |
| 프로필 버전·분석 provenance | 불변 스냅샷 + 탈퇴 시 개인정보 비식별/삭제 표시 | 분석 재현성과 개인정보 삭제 요구 균형 |
| 결제·크레딧 원장 | 취소·환불 보정 행 | 잔액 직접 덮어쓰기보다 감사 가능 |
| 임시 토큰·세션·파일 | 만료·폐기 또는 물리 정리 가능 | 보존 가치보다 노출 위험이 큼 |

“CareerTuner의 모든 데이터는 무조건 물리 삭제하지 않는다”도, “모든 데이터를 영원히 보존한다”도 정확하지 않다. 도메인 기록은 소프트 삭제 중심이고, 보안 토큰·임시 파일은 목적이 끝나면 폐기한다.

## 조회 계약: 활성 행만 기본

소프트 삭제를 넣고 조회 조건을 고치지 않으면 삭제된 글·사용자·분석이 다시 노출된다. 그래서 mapper의 기본 읽기에는 다음 조건이 포함된다.

```sql
WHERE deleted_at IS NULL
```

사용자 상태를 함께 보는 집계는 `users.status`와 계정 `deleted_at`도 확인한다. 관리자 감사처럼 삭제 행을 볼 필요가 있는 경로만 명시적으로 포함한다.

## 관계 데이터의 재활성

좋아요·스크랩·팔로우처럼 `(user_id, target_id)` UNIQUE 관계를 물리 삭제했다가 다시 insert하면 경쟁 조건과 카운트 오류가 생기기 쉽다. 현재 패턴은 다음과 같다.

1. 최초 행동: 관계 행 insert
2. 취소: `deleted_at` 기록
3. 재행동: 같은 행의 `deleted_at = NULL` 복원
4. 파생 카운트: 실제 활성 관계와 맞추거나 트랜잭션 안에서 보정

이렇게 하면 중복 행 없이 행동 이력과 현재 상태를 함께 다룰 수 있다.

## 계정 삭제 경계

계정을 삭제할 때 `users.status`만 바꾸면 프로필 원문, 업로드 파일, AI 결과, 커뮤니티 관계가 고아가 된다. 계정 삭제 서비스와 DB 패치는 다음 축을 함께 처리한다.

- 현재 프로필과 불변 프로필 버전의 개인정보 제거·삭제 표시
- 프로필 AI 산출물 삭제 표시
- 연결 파일의 저장소 정리 또는 비활성화
- 게시글·댓글·반응·스크랩의 활성 필터 정합
- 친구·대화·알림·구독 관계 정리
- refresh token 폐기로 재로그인 차단
- 관리자 집계에서 삭제 계정 제외

FK `ON DELETE CASCADE`만으로 해결하지 않는 이유는 계정 행 자체를 감사·상태 이력 때문에 남기기 때문이다.

## 영역 간 provenance

데이터를 소프트 삭제해도 과거 AI 결과가 어떤 입력에서 나왔는지 설명해야 한다.

- A는 `user_profile_version`으로 프로필 저장 시점을 보존한다.
- `profile_ai_analysis.profile_version_id`가 A 분석 입력 버전을 가리킨다.
- C는 `fit_analysis.source_snapshot`과 분석 메타데이터를 저장한다.
- D/E는 지원 건·질문·원문 reference와 source snapshot을 보존한다.
- 공급자·모델·상태는 AI 사용 로그와 결과 provenance에 남긴다.

원본을 덮어쓰지 않고 결과가 “그때 읽은 입력”을 들고 있어야 A의 수정이 C/D/E의 과거 설명을 바꾸지 않는다.

## 외부 파일 저장소와 DB의 원자성

DB transaction은 외부 object storage까지 자동 rollback하지 못한다. 따라서 업로드·삭제 경로는 다음 실패를 따로 고려한다.

- 파일 업로드 성공, DB 연결 실패 → 미연결 파일 정리
- DB soft delete 성공, storage 삭제 실패 → 재시도 가능한 상태 기록
- 계정 삭제 중 일부 파일 실패 → 나머지 개인정보 정리와 별도 보상 작업

파일 메타데이터만 지우고 실제 파일을 남기지 않는지, 반대로 DB가 가리키는 파일을 먼저 없애지 않는지가 통합 테스트 포인트다.

## 회귀 체크리스트

1. 삭제한 행이 사용자 목록·검색·집계에 나오지 않는가
2. 관리자 감사 경로만 삭제 행을 의도대로 볼 수 있는가
3. 좋아요 취소→재좋아요 시 행과 카운트가 중복되지 않는가
4. 부모 삭제 뒤 자식 데이터가 orphan이 되지 않는가
5. 계정 삭제 뒤 프로필 원문·업로드·AI 결과가 남지 않는가
6. 탈퇴 계정이 대시보드·관리자 활성 사용자 집계에서 제외되는가
7. 과거 분석은 당시 version/snapshot으로 설명 가능한가
8. 같은 DB 패치를 두 번 적용해도 lifecycle 상태가 바뀌지 않는가

## 면접에서의 짧은 답변

> “사용자 콘텐츠와 관계는 deleted_at 기반 소프트 삭제를 기본으로 하고, 좋아요 같은 UNIQUE 관계는 취소 후 같은 행을 재활성합니다. 계정 탈퇴 때는 상태만 바꾸지 않고 프로필 버전·AI 결과·파일·관계까지 함께 정리합니다. 반면 refresh token과 임시 파일은 보안상 목적이 끝나면 폐기합니다. 조회 mapper의 활성 필터와 관리자 예외 경로를 계약 테스트로 고정했습니다.”
