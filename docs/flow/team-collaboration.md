# 팀 협업 · 시스템 경계

> 6명이 한 제품을 깨뜨리지 않고 동시에 만드는 방법은 "착하게 굴자"가 아니라 **경계로 강제한 정합성**이다. 영역별 폴더 소유, 공통영역 팀장 게이트, 개인 브랜치→dev PR, 읽기전용 참조 규약 — 이 네 가지가 64만 줄짜리 충돌을 코드 리뷰 한 번으로 줄인다.

---

## 1. 이 흐름이 답하는 면접 질문

이 페이지는 코드의 동작이 아니라 **조직과 코드 구조를 어떻게 일치시켰나**를 설명한다. 면접에서 이런 질문에 답하는 근거다.

- "6명이 같은 레포에서 작업하는데 충돌은 어떻게 관리했나요?"
- "공통 코드(인증, DB 스키마, 라우팅)는 누가, 어떤 절차로 바꾸나요?"
- "남의 영역 데이터가 필요할 때 어떻게 가져오나요? 직접 수정해도 되나요?"
- "브랜치 전략과 릴리스 기준은 뭔가요?"
- "AI 도구로 작업한 흔적은 왜 커밋에 안 남기나요?"

핵심 한 줄: **수직 분담(각자 사용자화면+API+관리자+AI+DB를 통째로 소유) + 공통영역 단일 게이트 + 읽기전용 참조 규약**. 경계가 곧 정합성 보증 장치다.

## 2. 전체 그림 — 수직 분담과 두 종류의 경계

### 2-1. 6인 수직 분담 (영역 = 폴더 = 소유권)

각 담당은 가로(계층)로 자르지 않고 **세로(기능)로** 잘린 한 영역을 끝까지 책임진다. 사용자 화면, 사용자 API, 관리자 화면, 관리자 API, AI 기능, DB 테이블을 한 사람이 수직으로 소유한다.

| 영역 | 기능 도메인 | AI 기능 번호 | 대표 소유 테이블 |
| --- | --- | --- | --- |
| A | 회원/프로필/설정 | #1~5 | `users`, `user_profile`, `user_profile_version` |
| B | 지원 건/공고·기업 분석 | #6~11 | `application_case`, `job_posting`, `job_analysis`, `company_analysis` |
| C | 적합도/취업전략/대시보드 | #12~18 | `fit_analysis`, `career_analysis_run` |
| D | 가상 면접/리포트 | #19~23 | `interview_session/question/answer`, `interview_agent_step` |
| E | 첨삭/결제/크레딧 | #24~28 | `correction_request`, `payment`, `credit_transaction`, `plan` |
| F | 커뮤니티/고객센터/공지/알림 | #29~34 | `community_*`, `support_ticket`, `notice`, `notification` |

영역별 상세 개요는 각 영역 페이지로: [영역 A](/area-a/) · [영역 B](/area-b/) · [영역 C](/area-c/) · [영역 D](/area-d/) · [영역 E](/area-e/) · [영역 F](/area-f/).

### 2-2. 두 종류의 경계

```text
                        ┌──────────────────────────────────────────┐
                        │          공통영역 (Owner = 팀장)            │
                        │  common/  ai/common  ai/prompt 공통엔진     │
                        │  routes.ts  schema.sql  data.sql            │
                        │  application.yaml  build.gradle             │
                        │  → 변경 = 팀장 승인 / 팀 합의 필수            │
                        └──────────────────────────────────────────┘
                              ▲ 합의 게이트 (수정 전 협의)
                              │
   ┌─────┐  읽기전용 참조  ┌─────┐  읽기전용 참조  ┌─────┐
   │  A  │ ─────────────▶ │  C  │ ◀───────────── │  B  │   ← 영역 간 경계
   │소유 │   user_profile │소유 │   job_analysis │소유 │     (남의 원본은
   └─────┘                └─────┘                └─────┘      read-only)
   자기 테이블 = 쓰기 소유 / 남의 원본 = 읽기전용 동결 참조
```

- **수직 경계(영역 간)**: 각 영역은 자기 결과 테이블만 쓰고, 타 영역 원본은 읽기전용으로 참조한다.
- **수평 경계(공통영역)**: 모든 영역이 함께 쓰는 코드는 팀장이 단일 소유한다. 아무나 못 바꾼다.

## 3. 단계별 상세 — 경계가 작동하는 네 지점

### 3-1. 영역 폴더 소유 — "내 도메인 안에서 먼저 해결"

백엔드는 도메인 패키지 `controller → service → mapper → domain`(+필요 시 `dto`) 4계층, 영속성은 MyBatis만(`@Mapper` + `resources/mapper/**/*.xml`, JPA 금지). 프런트는 `frontend/src/features/<기능>/{pages,components,api,hooks,types}`, 관리자는 `frontend/src/admin/features/<기능>/`.

규칙은 단순하다. **공통화하고 싶은 충동이 들어도, 일단 자기 기능 폴더 안에서 해결한다.** 진짜로 여러 영역이 공유해야 할 때만 팀장과 합의해 공통영역으로 옮긴다. 이 한 줄이 "성급한 추상화 → 공통 코드 비대화 → 모두의 충돌"을 막는다.

### 3-2. 읽기전용 참조 경계 — 데이터 소유권 (테이블 단위)

원칙: **각 영역이 자기 결과 테이블을 소유하고, 타 영역 원본은 읽기전용으로 참조·비수정한다.**

| 영역 | 소유(쓰기) | 타 영역에 제공(읽기전용) | 핵심 규약 |
| --- | --- | --- | --- |
| A | `user_profile`(+버전 스냅샷) | → B/C/D/E | 원본 수정책임은 A에만 |
| B | `job_analysis`, `company_analysis` | → C·D·E | 구조화 결과만 노출 |
| C | `fit_analysis` | → D | `source_snapshot`/`input_snapshot` **동결**, `input_fingerprint` 캐싱 |
| D | `interview_answer` 등 | → C(장기경향)·E(첨삭 입력) | 세션 트레이스 보존 |
| E | `correction_request` | (원장 분리) | 첨삭은 원문 비수정 — `original_text`/`improved_text` 분리저장 |
| F | `community_*` | F#31 추출질문 → D 참고 | 후기는 D 면접 준비로 순환 |

읽기전용 참조의 진짜 의미는 단순 권한이 아니라 **정합성 동결**이다. 예를 들어 C는 A의 프로필과 B의 공고분석을 비교해 적합도를 내는데, 분석 시점의 입력을 `source_snapshot`으로 동결한다. A가 나중에 프로필을 바꿔도 과거 적합도 결과는 그대로 재현된다(상세: [/area-c/fit-analysis](/area-c/fit-analysis), [/area-c/caching-fingerprint](/area-c/caching-fingerprint)). 영역 간 데이터 의존의 전체 그림은 [데이터 소유권](/flow/data-ownership) 참조.

### 3-3. 공통영역 — 팀장 단일 소유 게이트

아래는 한 사람이 임의로 바꾸면 전 영역이 깨지는 코드다. 그래서 Owner를 팀장으로 못 박고, 수정 사유와 영향 범위를 공유한 뒤 **승인 또는 합의**를 거친다.

```text
backend:  common/   ai/common(현재 package-info 스켈레톤)   ai/prompt 공통엔진
          admin/prompt 루트셸   schema.sql   data.sql   application.yaml   build.gradle
          공통 로그 스키마·수집기
frontend: app/routes.ts   admin/routes.ts   app/components/{layout,ui,media,upload}
          app/lib   admin/features/prompts 루트셸
```

게이트가 필요한 변경 종류: **라우팅 · 공통 컴포넌트 · 공통 API · DB 구조 · 인증/권한 · AI 프롬프트 공통 엔진 · 공통 로그 구조.** 단, 단순 오타·주석·명백한 문서 오류는 예외로 바로 고친다. 기능별 프롬프트와 기능별 로그 하위 폴더는 각 담당이 소유하므로(`admin/features/<feature>/logs` 식) 게이트 대상이 아니다.

:::tip 게이트가 실제로 작동한 사례
AutoPrep 오케스트레이터가 SSE 스트리밍을 붙이면서 공통영역(`SecurityConfig`)을 건드린 변경은 **딱 하나**다 — 비동기 재디스패치 401을 막는 `dispatcherTypeMatchers(ASYNC, ERROR).permitAll()`(`SecurityConfig.java:42-43`). 오케스트레이터는 새 AI를 만들지 않고 A~F 기존 도메인 서비스를 래핑하기 때문에, 6영역을 가로지르는 거대 기능인데도 공통영역 침범이 한 줄로 끝났다. 이게 경계 설계의 효과다(상세: [/area-d/orchestrator-interview](/area-d/orchestrator-interview)).
:::

### 3-4. 인증/권한 경계 — 공통영역이 보증하는 단일 진입

인증은 전 영역 공통이라 당연히 팀장 소유다. STATELESS + JWT 구조로, `JwtAuthenticationFilter`를 `UsernamePasswordAuthenticationFilter` 앞에 배치하고(`SecurityConfig.java:38,77`) `Authorization: Bearer` 토큰을 `AuthUser`로 파싱한다. 관리자 경로 `/api/admin/**`는 `hasAnyRole("ADMIN","SUPER_ADMIN")`, 그 외는 `anyRequest().authenticated()`. 각 영역은 컨트롤러에서 `@AuthenticationPrincipal AuthUser`만 받으면 되고, 토큰 검증·역할 부여·CORS는 공통영역이 책임진다. 영역마다 인증을 재구현하지 않는 것 자체가 경계의 이득이다.

## 4. 설계 포인트 — 왜 이렇게 연결했나

### 4-1. 수직 분담을 택한 이유

수평 분담(누구는 컨트롤러, 누구는 서비스)을 택하면 기능 하나에 6명이 동시에 손대야 한다. 수직 분담은 **한 기능 = 한 사람 = 한 폴더**라 책임 경계가 코드 경로와 일치한다. 디렉터리만 보면 누가 소유자인지 안다. 충돌은 자연히 폴더 경계에서 멈춘다.

### 4-2. 읽기전용 참조가 정합성을 지키는 방식

각 영역이 남의 원본을 직접 고칠 수 있으면, A가 프로필을 바꿨을 때 C/D/E의 분석 결과가 소리 없이 깨진다. 그래서 **쓰기는 소유자만, 나머지는 동결 스냅샷 읽기**로 묶었다. 이 규약 덕에 영역 간 의존이 단방향이 되고, 시점 정합성(과거 분석의 재현성)이 보장된다. C의 `input_fingerprint` 캐싱이 이 동결을 활용한 대표 사례다.

### 4-3. 공통영역 단일 게이트의 트레이드오프

공통영역은 충돌이 가장 자주 나는 곳이라 속도를 위해 꼭 필요하지만, 동시에 가장 위험하다. 그래서 "기능 폴더에서 먼저 해결 → 정말 공통일 때만 팀장과 합의해 이동"이라는 비대칭 규칙을 둔다. 추상화를 **늦게, 합의로** 끌어올리는 쪽에 비용을 둬서 공통 코드의 무분별한 비대화를 막는다.

### 4-4. 브랜치 전략 = 보호 게이트

```text
개인 브랜치(LEE-..., SHIN-... 등)  ──PR──▶  dev (보호 브랜치)
        │                                     │
   기능 작업·커밋                        리뷰 통과 후 머지
        │                                     │
   dev 직접 push 금지 ◀──────────────────  보호 게이트
```

각자 개인 브랜치에서 작업하고 `dev`로 PR을 올린다. **dev/main 등 보호 브랜치 직접 push는 금지.** push 절차는 (1) 현재 브랜치가 보호 브랜치면 중단·보고 → (2) `git fetch origin` → (3) `origin/dev`에 새 커밋이 있고 내 수정 파일과 겹치면 사용자에게 보고하고 "그대로 push / merge 후 push" 선택 → (4) 커밋 메시지 검사 → (5) push. PR/브랜치 운영 흐름은 [팀 협업 개요](/flow/team-collaboration)와 영역별 playbook을 함께 본다.

### 4-5. 커밋 위생 규칙

- 커밋 메시지 형식: `feat:` / `fix:` / `docs:` / `chore:` / `refactor:` prefix, **본문은 한국어**.
- **커밋·PR에 AI 도구 흔적 금지** — `Co-Authored-By`, "Generated with ...", 서명 이모지 등 일절. 이 규칙은 도구 기본 동작보다 우선한다. 면접에서 묻는다면: "커밋 히스토리는 사람 책임의 기록이라, 작성 도구가 아니라 변경 이유만 남긴다"가 답이다.

### 4-6. 서브모듈 — 산출물과 코드의 경계

스토리보드/기획 산출물(`docs/storyboard/`)은 별도 repo를 가리키는 **git 서브모듈**이다. 메인을 그냥 클론하면 포인터만 받아 본체 용량·빌드에 영향이 없고, 필요할 때만 `git submodule update --init docs/storyboard`로 받는다. 산출물(PPTX/PDF/DB설계서)과 런타임 코드를 물리적으로 분리해, 무거운 문서가 코드 레포를 오염시키지 않게 한 경계다.

## 5. 구현 상태

| 항목 | 상태 | 비고 |
| --- | --- | --- |
| 수직 분담 6영역 폴더 구조 | 구현됨 | 백엔드 도메인 패키지·프런트 features 분리 |
| 4계층 + `ApiResponse<T>` 공통 규약 | 구현됨 | `common/web/ApiResponse` |
| STATELESS JWT 인증/권한 경계 | 구현됨 | `SecurityConfig`, `JwtAuthenticationFilter` |
| 읽기전용 참조·스냅샷 동결 | 구현됨(영역별 편차) | C `source_snapshot`/`input_fingerprint` 확인 |
| 공통영역 팀장 게이트(`schema.sql` 등) | 운영 규칙 | 코드가 아닌 협업 프로세스로 강제 |
| `ai/common` 공통 엔진 | 스켈레톤 | 현재 `package-info.java` 위주, 실제 AI는 영역별 서비스에 분산 |
| 개인 브랜치 → dev PR | 운영 중 | dev 보호 브랜치 |
| 스토리보드 서브모듈 | 구현됨 | 선택 다운로드 |

:::warning 정직한 구분
"전 영역 공통 AI 클라이언트"는 **목표**이고, 현재 `ai/common`은 스켈레톤이다. 실제 AI 호출은 각 영역 도메인 서비스에 흩어져 있고(예: C `FallbackFitAnalysisAiService`, D `FallbackInterviewLlmGateway`), 공통화된 건 사용량 로그 **스키마**(`ai_usage_log`)와 `ApiResponse` 규약 정도다. 면접에서 "공통 AI 엔진"을 완성된 것처럼 말하지 말 것.
:::

또 하나 — 자체 LLM 트랙(6명 동일 파이프라인: 합성데이터→LoRA→Ollama→외부API 폴백)은 **팀 합의 목표**다. 런타임에서 실제 동작하는 폴백 범위는 영역마다 다르므로([AI 공급자·폴백](/flow/ai-providers-fallback) 참조), 분담 규칙과 실제 적용을 섞어 말하지 않는다.

## 6. 면접 답변 — 전체를 하나의 흐름으로

> "6명이 하나의 모놀리식 레포에서 작업했는데, 충돌 관리의 핵심은 **조직 구조와 코드 구조를 일치시킨 것**입니다. 계층(컨트롤러/서비스)으로 나누지 않고 기능(영역 A~F)으로 수직 분담해서, 한 사람이 자기 영역의 사용자 화면·API·관리자·AI·DB 테이블을 통째로 소유했습니다. 그래서 디렉터리만 보면 소유자가 명확하고 충돌이 폴더 경계에서 멈춥니다.
>
> 영역을 가로지르는 두 종류의 경계가 정합성을 지킵니다. 첫째, 데이터는 **소유자만 쓰고 나머지는 읽기전용으로 참조**합니다. 예컨대 제 영역(C 적합도)은 A의 프로필과 B의 공고분석을 비교하지만 원본은 안 고치고, 분석 시점 입력을 `source_snapshot`으로 동결한 뒤 `input_fingerprint`로 캐싱합니다. 그래서 A가 나중에 프로필을 바꿔도 과거 결과가 재현됩니다. 둘째, 인증·DB 스키마·라우팅 같은 **공통영역은 팀장이 단일 소유**하고, 수정은 사유·영향범위 공유 후 합의를 거칩니다. 거대 기능인 AutoPrep 오케스트레이터조차 공통영역(SecurityConfig)을 SSE용 한 줄만 건드렸습니다.
>
> 협업은 개인 브랜치에서 작업해 dev로 PR하고, dev 직접 push는 금지했습니다. 커밋은 prefix 규칙과 한국어 본문, AI 도구 흔적 미표기 위생을 지켰습니다. 요약하면, **'착하게 굴자'가 아니라 경계로 강제한 정합성**이 핵심입니다."

## 7. 꼬리질문 + 모범답안

:::details Q. 영역 간 데이터가 필요하면 직접 join 쿼리로 가져오면 되지 않나요? 왜 읽기전용·스냅샷인가요?
직접 참조 자체는 막지 않지만 **쓰기는 금지**입니다. 단순 권한 문제가 아니라 정합성 때문입니다. C 적합도는 분석 시점의 A 프로필·B 공고를 비교한 결과인데, 원본이 나중에 바뀌면 과거 결과의 근거가 사라집니다. 그래서 입력을 `source_snapshot`으로 동결하고 `input_fingerprint`로 캐시 키를 만듭니다. 같은 입력이면 같은 결과를 재현하고, 입력이 바뀌면 재분석이 트리거됩니다. 읽기전용 + 스냅샷이 시점 정합성과 재현성을 동시에 보장합니다.
:::

:::details Q. schema.sql을 영역마다 바꾸면 충돌이 날 텐데 어떻게 했나요?
`schema.sql`/`data.sql`/`application.yaml`/`build.gradle`은 공통영역이라 Owner가 팀장입니다. 영역 담당이 새 테이블이 필요하면 사유와 영향 범위를 공유하고 합의 후 반영합니다. 덕분에 마이그레이션이 한 곳에서 직렬화되고, 두 사람이 같은 파일을 동시에 깨는 일이 줄어듭니다. 반대로 기능별 프롬프트·기능별 로그 폴더는 각자 소유라 게이트 없이 자유롭게 만집니다 — 충돌이 잦은 것만 게이트에 묶고 나머지는 풀어준 비대칭 설계입니다.
:::

:::details Q. 수직 분담의 단점은 없나요?
있습니다. 영역마다 비슷한 코드(폴백 처리, 사용량 로깅)가 중복되기 쉽고, 실제로 C·D·E의 폴백 체인이 영역별로 따로 구현돼 있습니다. 그래서 공통 규약(`ApiResponse`, `ai_usage_log` 스키마)으로 인터페이스만 맞추고, 진짜 공통화는 충분히 패턴이 드러난 뒤 합의해 끌어올리는 전략을 택했습니다. 성급한 공통화로 모두가 공통 코드 충돌에 묶이는 것보다, 약간의 중복을 감수하는 편이 6인 병렬 개발에는 유리하다고 판단했습니다.
:::

:::details Q. 오케스트레이터처럼 6영역을 가로지르는 기능은 누가 소유하나요? 경계가 깨지지 않나요?
AutoPrep 오케스트레이터는 **새 AI를 만들지 않고 A~F의 기존 도메인 서비스를 래핑**합니다. PROFILE→A, JOB→B, FIT→C, INTERVIEW→D, WRITE→E, COMMUNITY→F 핸들러가 각 영역 서비스를 호출할 뿐이라, 비즈니스 로직 소유권은 그대로 각 영역에 남습니다. 오케스트레이터는 의존그래프(FIT·INTERVIEW는 JOB 뒤)와 병렬 실행·SSE만 책임집니다. 공통영역 변경도 SSE 401 방지 한 줄로 끝났고요. 가로지르는 기능을 "조율자"로만 두고 로직 소유는 영역에 남긴 게 경계를 지킨 비결입니다.
:::
