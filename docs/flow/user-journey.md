# 사용자 end-to-end 여정 — 한 사용자의 하루

> 회원가입·프로필(A) → 지원 건 생성·공고분석(B) → 적합도·전략(C) → 가상면접(D) → 답변·자소서 첨삭(E) → 커뮤니티 후기(F). CareerTuner는 흩어진 6개 기능이 아니라, **하나의 지원 건(Application Case)** 위에서 앞 단계 산출물이 뒤 단계 입력이 되는 단방향 파이프라인이다.

이 페이지는 영역별 상세가 아니라 **"각 화면이 무엇을 받아 다음 화면에 무엇을 넘기는가"** 에 집중한다. 영역 내부 구현(스코어 엔진, 폴백 게이트웨이, RAG 등)은 각 영역 페이지로 링크한다.

## 1. 이 여정이 답하는 면접 질문

- "프로젝트를 처음부터 끝까지, 사용자 입장에서 한 번 설명해 보세요."
- "기능들이 따로 노는 게 아니라 서로 연결돼 있다고 했는데, 구체적으로 무엇이 무엇을 넘깁니까?"
- "사용자가 프로필을 한 번 입력하면 그게 어디까지 재사용됩니까?"
- "면접 결과가 다시 적합도 분석으로 돌아간다고 했는데, 그 순환(loop)은 어떻게 닫히나요?"
- "이 흐름에서 여러분(C 담당)이 만든 화면은 왜 '중심 허브'입니까?"

핵심은 **단위가 '공고'가 아니라 '지원 건'** 이라는 점이다. 프로필·공고분석·적합도·면접·첨삭·결제·후기가 전부 하나의 지원 건에 매달려 있고, 그래서 "이 회사에 지원하는 나"를 일관되게 추적할 수 있다.

## 2. 전체 그림 — 한 사용자의 하루

데모 사용자 `김데모`가 한 회사에 지원한다고 하자. 로그인 직후부터 후기 작성까지 8개 대표 화면을 거친다.

```text
[A 프로필]   이력서·자소서·기술스택 입력
   │  프로필 스냅샷 (전 영역 공통 입력)
   ▼
[B 공고분석]  지원 건 생성 → 공고 원문/OCR → 필수·우대·담당업무 구조화
   │  job_analysis / company_analysis (읽기전용으로 하류 제공)
   ▼
[C 적합도] ◀── 중심 허브 ── A 스냅샷 × B 요구조건
   │  점수 + 근거 + 다음 행동(부족역량·로드맵)
   ▼
[D 가상면접]  예상질문 → 답변 → 평가 → 리포트
   │  interview_answer / 평가 결과
   ▼
[E 첨삭·결제]  답변·자소서 첨삭(원문 비수정) → 크레딧 차감 → 사용량·요금제
   ╎  (E ⇢ C) 사용량·재분석으로 순환
   ▼
[F 커뮤니티]  면접 후기 공유 → AI 요약·태그·추천
   ╎  (F ⇢ B) 실제 면접질문이 다음 지원 준비의 참고로 순환
```

화살표 의미를 한 줄씩:

| 구간 | 무엇이 넘어가나 | 방향 |
|---|---|---|
| A → B | 정리된 스펙(요약·키워드)이 공고 비교 기준이 됨 | 단방향 |
| B → C | 필수/우대 조건이 적합도 점수의 비교 대상이 됨 | 단방향 |
| C → D | 부족 역량·검증 포인트가 면접 질문 소재가 됨 | 단방향 |
| D → E | 면접 답변·평가가 첨삭 입력이 됨 | 단방향 |
| E ⇢ C | AI 사용량/재분석 트리거가 적합도 허브로 돌아옴 | 순환 |
| D ⇢ F | 면접 경험이 후기로 공유됨 | 분기 |
| F ⇢ B | 후기에서 추출한 실제 질문이 다음 지원 참고가 됨 | 순환 |

C가 한가운데 있는 이유는, **A의 "나"와 B의 "회사 요구"를 처음으로 한 화면에서 충돌시키는 지점**이기 때문이다. 여기서 점수가 나오고, 그 점수가 D 면접과 E 첨삭의 우선순위를 정한다.

## 3. 단계별 상세 — 무엇을 받아 무엇을 넘기나

### 01. 회원가입·프로필 — 모든 분석의 출발점 (A)

- 화면: `/profile` · 영역 A
- **받는 것**: 이력서·경력·프로젝트·자소서·기술스택(사용자 직접 입력 + 파일 업로드)
- **하는 것**: 프로필 요약(#1), 기술스택 추출(#2), 자소서·경력 키워드 추출(#3·#4), 완성도 진단(#5)
- **넘기는 것**: **프로필 스냅샷** — 이후 B·C·D·E 전 영역이 공유하는 단일 입력 원천

여기서 핵심은 A의 5개 AI 기능이 화면에서는 **3개 버튼(요약 / 스킬 / 완성도)** 으로 노출된다는 점이다. 자소서·경력 키워드는 요약 응답 필드 안에 함께 담겨 나온다. 면접에서 "프로필 AI가 5개 엔드포인트냐"고 물으면 "AI 기능은 5개지만 엔드포인트는 `/profile/ai/{summary,skills,completeness}` 3개"라고 답해야 한다.

원본 데이터의 **수정 책임은 A에만 있다.** 하류 영역은 `user_profile`을 읽기만 한다.

:::tip 왜 스냅샷인가
프로필은 계속 바뀐다. C 적합도는 "분석 당시의 나"를 동결해야 재현 가능하므로, 분석 시점의 프로필을 `source_snapshot`으로 떠서 비교에 쓴다. 회원가입·프로필 상세는 [영역 A 개요](/area-a/)와 [프로필 버전 관리](/area-a/profile-versioning)를 참고.
:::

→ 상세: [/area-a/](/area-a/) · [인증/JWT](/area-a/auth-jwt) · [프로필 데이터 모델](/area-a/profile-data-model)

### 02. 지원 건 생성·공고분석 — 무엇을 요구하는가 (B)

- 화면: `/applications/1/job-analysis` · 영역 B
- **받는 것**: 공고 원문 텍스트, 또는 이미지/PDF(OCR·텍스트 추출 경유)
- **하는 것**: 공고문 분석(#6) → 필수조건(#7)·우대조건(#8)·담당업무(#9)·기업현황(#10)·면접포인트(#11)로 파생
- **넘기는 것**: `job_analysis`·`company_analysis` 구조화 결과를 C·D·E에 **읽기전용**으로 제공

여기서 비로소 **지원 건(`application_case`)이 생성**된다. A의 프로필이 "나"라면, B는 "이 회사가 원하는 것"을 구조화한다. 공고 텍스트 추출은 ML 워커(문장 분류 기반)와 백엔드 텍스트 추출이 협업한다.

→ 상세: [/area-b/](/area-b/) · [필수·우대 추출](/area-b/required-preferred) · [텍스트 추출·OCR](/area-b/text-extraction-ocr) · [지원 건 라이프사이클](/area-b/application-lifecycle)

### 03. 적합도·전략 — 지원해도 되는가 (C · 중심 허브)

- 화면: `/applications/1/fit` · 영역 C
- **받는 것**: A 프로필 스냅샷 + B 필수/우대 조건
- **하는 것**: 적합도 점수(#12), 부족 역량(#13), 학습 로드맵(#14)·자격증(#15), 장기 경향(#16), 다음 지원 방향(#17), 대시보드 요약(#18)
- **넘기는 것**: "점수 + 근거 + 다음 행동" 묶음 — D 면접과 E 첨삭의 **우선순위**가 된다

C가 통합 여정의 중심인 이유는 두 가지다. 첫째, A와 B가 **처음 만나는 지점**이다. 둘째, 결과가 단순 점수가 아니라 **의사결정 카드**("보완 후 지원 권장" + 학습 로드맵)라서 다음 단계의 방향을 정한다.

설계상 가장 중요한 포인트: **점수·판단은 LLM이 아니라 서버 규칙이 결정한다.** AI는 후보 텍스트(설명)를 생성하지만, 적합도 신뢰도와 조건 매트릭스 severity는 입력 상태 기반의 결정적 계산이다. 그래서 mock이든 실제 AI든 같은 점수가 나오고, 면접에서 "AI가 점수를 직접 매기면 재현이 안 되지 않나요?"라는 함정 질문을 막을 수 있다.

:::warning C는 원본을 절대 수정하지 않는다
적합도는 A 프로필과 B 공고를 비교하되, 두 원본 어디도 고치지 않는다. 입력을 `input_snapshot`으로 동결하고 `input_fingerprint`로 캐싱한다. 같은 입력이면 다시 LLM을 부르지 않고 캐시를 돌려준다.
:::

→ 상세: [/area-c/](/area-c/) · [적합도 분석](/area-c/fit-analysis) · [뉴로-심볼릭 점수](/area-c/neuro-symbolic) · [캐싱·핑거프린트](/area-c/caching-fingerprint)

### 04~05. 가상면접 — 무엇을 묻고, 무엇이 부족했나 (D)

- 화면: `/interview?tab=questions` (질문) → `/interview?tab=report` (리포트) · 영역 D
- **받는 것**: B 공고/면접포인트(#11) + C 적합도(#12) + A 스냅샷
- **하는 것**: 예상질문(#19) → 꼬리질문(#20) → 면접관 진행(#21) → 답변 평가(#22) → 리포트(#23)
- **넘기는 것**: `interview_answer`와 평가 결과를 E 첨삭과 C 장기경향(#16)에 제공

D는 "C가 짚은 부족 역량"을 질문으로 바꿔 사용자를 압박한다. 답변을 평가하고(#22), 빈틈으로 꼬리질문을 만들고, 세션 전체를 리포트로 종합한다(#23). 리포트의 약점이 두 갈래로 흐른다 — **E 첨삭의 대상**이 되고, **C 장기 경향 분석의 입력**이 되어 순환을 만든다.

D는 멀티에이전트 트레이스(`interview_agent_step`)와 RAG 원본(`interview_knowledge`)을 별도로 관리한다. 면접 RAG 벡터는 Qdrant를 best-effort로 쓰고, 비어 있으면 건너뛴다.

→ 상세: [/area-d/](/area-d/) · [질문 생성](/area-d/question-generation) · [답변 평가](/area-d/answer-evaluation) · [면접 리포트](/area-d/interview-report) · [RAG 그라운딩](/area-d/rag-grounding)

### 06~07. 답변·자소서 첨삭, 사용량·결제 (E)

- 화면: `/correction?tab=answer` (첨삭) → `/billing?tab=usage` (사용량) · 영역 E
- **받는 것**: D 면접 답변·평가 + A 자소서/이력서 원문
- **하는 것**: 답변 첨삭(#24), 자소서·이력서·포폴 첨삭(#25~#27), 요금제 추천(#28)
- **넘기는 것**: 개선안을 **원문과 분리 저장**(`original_text`/`improved_text`), 사용량을 `ai_usage_log`로 집계

E의 철칙은 **원문 비수정**이다. 첨삭은 `correction_request`에 별도로 쌓이고, A의 원본을 절대 덮어쓰지 않는다. 각 첨삭은 크레딧을 소모한다.

여기서 전 영역이 만난다. 공고분석·적합도·면접·첨삭 등 **모든 영역의 AI 호출이 결국 E의 사용량·결제 화면 하나로 묶인다.** 잔액·패턴을 근거로 요금제·충전을 추천한다(강요 아님).

:::tip 크레딧은 성공할 때만, 한 번만 차감
크레딧 차감은 success-gated + 멱등이다. AI 호출이 실패(`FAILED`)하면 차감하지 않고, 같은 호출을 중복 차감하지도 않는다. 그래서 "AI가 실패했는데 돈만 나갔다"가 구조적으로 불가능하다. 결제·크레딧 상세는 [크레딧 시스템](/area-e/credit-system)과 [크레딧·사용량 흐름](/flow/credit-usage)을 참고.
:::

→ 상세: [/area-e/](/area-e/) · [답변 첨삭](/area-e/ai-answer-correction) · [첨삭 원칙](/area-e/correction-principles) · [결제 흐름](/area-e/payment-flow)

### 08. 커뮤니티 후기 — 경험이 모이고 다시 흐른다 (F)

- 화면: `/community` · 영역 F
- **받는 것**: 사용자 후기·태그·댓글
- **하는 것**: 후기 요약(#29), 태그 추천(#30), 실제 면접질문 추출(#31), 관심 기반 추천(#32), 신고 분류(#33), 문의 답변초안(#34)
- **넘기는 것**: #31이 추출한 **실제 면접질문**이 D의 참고 데이터로, 후기 자체가 다음 지원 준비(B)의 참고로 순환

F는 여정의 끝이자, **여정 바깥에서 순환을 닫는 고리**다. 한 사용자의 면접 경험이 후기로 공유되면, 그 후기의 실제 질문이 다른 사용자의 D 면접 준비나 B 지원 준비로 흘러든다. F는 인테이크 챗봇으로 D의 AutoPrep 슬롯 수집까지도 담당한다(실행 스트림은 D).

→ 상세: [/area-f/](/area-f/) · [인테이크 챗봇](/area-f/intake-chatbot) · [후기 요약](/area-f/ai-review-summary) · [실제 질문 추출](/area-f/ai-question-extract)

## 4. 설계 포인트 — 왜 이렇게 연결했나

**1) 단일 단위(지원 건)로 묶어 맥락을 잃지 않는다.** 프로필·공고·적합도·면접·첨삭이 같은 `application_case` 아래 있으니, "이 회사에 맞춘 나"를 한 번에 끌어올 수 있다. 화면이 바뀌어도 입력을 다시 모을 필요가 없다.

**2) 데이터 소유권을 영역별로 못 박는다.** 각 영역은 자기 결과 테이블만 쓰고, 타 영역 원본은 읽기전용으로 참조한다.

| 영역 | 자기가 소유(쓰기) | 하류에 제공(읽기전용) |
|---|---|---|
| A | `users`, `user_profile`(+버전) | → B/C/D/E |
| B | `application_case`, `job_analysis`, `company_analysis` | → C/D/E |
| C | `fit_analysis`, `career_analysis_run` | → D 참조(원본 비수정·핑거프린트 캐싱) |
| D | `interview_session/question/answer`, `interview_knowledge` | → E 첨삭·C 장기경향 |
| E | `correction_request`, `payment`, `credit_transaction` | (원장 분리·원문/결과 분리 저장) |
| F | `community_*`, `support_ticket`, `notification` | #31 추출 질문 → D 참고 |

이 경계 덕분에 6명이 수직 분담으로 병렬 개발하면서도 서로의 데이터를 깨뜨리지 않는다. 데이터 소유권 맵 상세는 [데이터 소유권 경계 맵](/flow/data-ownership)을 참고.

**3) 순환은 단방향 흐름을 깨지 않는다.** E→C, F→B의 순환은 "이전 결과를 새 입력으로 다시 던지는" 것이지, 양방향 의존이 아니다. C는 여전히 A·B만 읽고, D는 여전히 B·C만 읽는다. 순환은 "사용량 트리거"나 "참고 데이터"로 느슨하게 닫힌다.

**4) 점수·판단은 코드, 설명만 AI.** 적합도 점수, 크레딧 차감, 조건 severity 등 **돈과 의사결정에 직결되는 값은 서버 규칙이 결정**한다. LLM은 그 위에 자연어 설명을 입힐 뿐이다. 이게 재현성·비용 통제·설명가능성을 동시에 잡는다.

## 5. 구현 상태

::: details 구현된 것 (현재 동작)
- A~F 6영역 화면과 AI #1~34의 도메인 서비스가 각 영역에 구현돼 있다.
- 지원 건 단위로 A→B→C→D→E 단방향 파이프라인이 실제 데이터로 흐른다.
- AutoPrep 오케스트레이터가 6개 영역 서비스를 래핑해 의존그래프 병렬 실행 + SSE 보고를 한다([AI 오케스트레이터 전체](/flow/ai-orchestrator)).
- 크레딧 success-gated 멱등 차감, C 결정적 신뢰도 계산은 코드로 검증됨.
:::

::: warning 설계 목표 / 부분 구현 (정직하게 구분)
- **자체 OSS LLM → Claude Haiku → OpenAI → Mock** 4단 폴백은 "사상"이고, 실제로 Claude Haiku 폴백이 코드로 작동하는 건 **D(면접) + 두뇌(플래너)** 뿐이다. C는 OSS→OpenAI→Mock(Haiku 건너뜀), A·B 일부·E 자소서는 OpenAI→규칙엔진 폴백이다.
- 자체 모델 생성 화이트리스트는 현재 **빈 집합**이라, 면접 질문 생성은 실제로는 외부 API가 담당한다([AI 공급자·폴백 전략](/flow/ai-providers-fallback)).
- F 인테이크 챗봇 → D AutoPrep은 프런트 칩·파일 handoff까지 연결되고, 확정 슬롯은 `chatbot_intake_slot`에 영속돼 PENDING 상태를 복원한다.
- 기준 커밋의 canonical `schema.sql`은 서로 다른 테이블 선언 172개다.
:::

## 6. 면접 답변 — 전체를 흐름으로 설명하기

> "CareerTuner를 한 사용자의 여정으로 설명해 보세요."

"CareerTuner의 핵심 단위는 공고가 아니라 **지원 건**입니다. 사용자가 회사 하나에 지원할 때, 그 지원 건 위에 프로필·공고분석·적합도·면접·첨삭·결제가 전부 매달립니다.

흐름은 단방향 파이프라인입니다. 먼저 **A 프로필**에서 이력서·자소서를 입력하면 AI가 요약·키워드로 정리하고, 이 스냅샷이 이후 전 영역의 공통 입력이 됩니다. **B 공고분석**은 지원 건을 만들고 공고에서 필수·우대 조건을 구조화합니다. **C 적합도**는 A의 '나'와 B의 '회사 요구'를 처음으로 충돌시켜 점수와 부족 역량을 냅니다 — 여기가 중심 허브입니다.

C가 짚은 약점을 **D 가상면접**이 질문으로 바꿔 묻고, 답변을 평가해 리포트를 냅니다. 그 답변과 자소서를 **E 첨삭**이 원문을 건드리지 않고 개선하고, 모든 영역의 AI 사용량이 E의 결제 화면 하나로 묶입니다. 마지막으로 **F 커뮤니티**에 후기가 공유되고, 후기에서 추출한 실제 면접질문이 다시 다음 지원 준비로 순환합니다.

설계상 중요한 두 가지는, 첫째 **점수와 크레딧 같은 돈·판단 값은 서버 규칙이 결정**하고 LLM은 설명만 한다는 것, 둘째 각 영역이 **자기 결과 테이블만 쓰고 타 영역 원본은 읽기전용**으로 참조해 6명이 병렬 개발해도 데이터가 안 깨진다는 것입니다."

## 7. 꼬리질문 + 모범답안

::: details Q1. C가 '중심 허브'라는 게 그냥 화면 위치 얘기 아닌가요?
아닙니다. 데이터 흐름상 C는 **A와 B 두 상류가 처음 합류하는 지점**이고, 그 산출물(점수+부족역량)이 **D 면접 질문과 E 첨삭 우선순위를 결정**합니다. 즉 상류 둘을 입력으로 받고 하류 둘의 방향을 정하는 분기점이라서 허브입니다. 게다가 D·E 결과가 다시 C의 장기 경향(#16)으로 순환해 들어옵니다.
:::

::: details Q2. E→C, F→B 순환이 있으면 단방향 파이프라인이라는 말과 모순 아닌가요?
순환은 "이전 결과를 새로운 입력으로 다시 던지는" 것이지 양방향 의존이 아닙니다. C는 코드 레벨에서 여전히 A·B만 읽습니다. E의 순환은 '사용량·재분석 트리거'로, F의 순환은 '참고 데이터'로 느슨하게 닫힙니다. 강결합(circular dependency)이 아니라 데이터 재투입(feedback)입니다.
:::

::: details Q3. 프로필을 분석 도중에 사용자가 바꾸면 적합도 결과가 흔들리지 않나요?
안 흔들립니다. C는 분석 시점의 프로필을 `source_snapshot`/`input_snapshot`으로 동결하고 `input_fingerprint`로 캐싱합니다. 입력이 같으면 LLM을 다시 부르지 않고 캐시를 돌려주고, 입력이 바뀌면 새 행을 INSERT합니다. 그래서 "그때 그 분석"이 항상 재현됩니다.
:::

::: details Q4. AI 호출이 실패했는데 사용자에게 과금되는 사고는 어떻게 막나요?
크레딧 차감이 success-gated이기 때문입니다. 사용량 로그의 status가 `SUCCESS`가 아니면 차감 로직이 그대로 skip하고, 같은 호출의 중복 차감도 멱등 키로 막습니다. 실패한 AI는 구조적으로 무료입니다. 폴백이 일어나도 사용량 로그에는 최종 성공 모델만 기록됩니다.
:::

## 8. 직접 말해보기

다음을 소리 내어 답해 보세요. 막히면 위 섹션으로 돌아가세요.

1. A에서 입력한 프로필이 정확히 어느 영역들까지 재사용되는지, '읽기전용'이 무슨 뜻인지 설명하라.
2. C가 받는 입력 2개와, C가 넘기는 출력이 어느 두 영역의 우선순위를 정하는지 말하라.
3. D 면접 리포트의 약점이 흐르는 두 갈래(E와 C)를 각각 무엇이라 부르는지 설명하라.
4. "점수는 AI가 안 매긴다"는 주장을 면접관에게 설득력 있게 30초로 변호하라.
5. 이 여정에서 순환(loop)이 두 군데(E→C, F→B) 있는데, 왜 이게 단방향 파이프라인을 깨지 않는지 변호하라.
