# 프로젝트 전체 면접 플레이북

> "이 프로젝트에 대해 설명해 주세요." — 이 한 질문에 1분·3분·5분으로 막힘없이 답하고, 어떤 꼬리질문이 들어와도 **하나의 일관된 그림**에서 답을 꺼내기 위한 종합 인덱스. 영역별 깊이는 [/area-a](/area-a/)~[/area-f](/area-f/)에 있고, 이 페이지는 그것을 **말로 엮는 법**을 다룬다.

## 1. 이 흐름이 답하는 면접 질문

면접관이 "프로젝트 소개"를 시키는 진짜 의도는 세 가지다. (1) 전체를 한 그림으로 압축할 수 있는가(추상화), (2) 그 안에서 내가 한 일과 안 한 일을 정직히 구분하는가(주인의식), (3) 기술 선택마다 "왜"를 대는가(설계 사고). 이 페이지는 그 셋을 동시에 만족하는 스크립트 트리를 만든다.

대비하는 질문군:

- "프로젝트를 1분 / 3분 / 5분으로 소개해 주세요."
- "전체 아키텍처를 그려가며 설명해 주세요."
- "이 프로젝트에서 가장 잘 설계했다고 보는 한 가지는?"
- "본인은 어디를 담당했고, 다른 사람 영역과 어떻게 연결됩니까?"
- "왜 그 기술을 골랐나요? 안 그랬으면 어떻게 됐나요?"
- "구현하면서 가장 어려웠던 점과, 지금 더 개선한다면?"

::: tip 모든 답이 갈라져 나오는 한 문장
**"CareerTuner는 '공고'가 아니라 '지원 건(Application Case)'을 핵심 단위로, 프로필·공고분석·적합도·면접·첨삭·결제를 하나의 케이스에 꿰는 AI 취업 전략 플랫폼이고, AI는 34개 기능이 영역별로 수직 분담되되 하나의 오케스트레이터가 의존 그래프대로 병렬 실행합니다."**
1분·3분·5분 스크립트가 전부 이 문장의 확대/축소다.
:::

## 2. 전체 그림 (표 · 흐름도)

### 2-1. 30초 압축

| 축 | 내용 |
| --- | --- |
| 한 줄 정의 | 채용공고에 맞춰 스펙·면접 답변을 조정하는 AI 취업 전략 플랫폼 |
| 핵심 단위 | **지원 건(Application Case)** — 공고 아님. 모든 데이터가 케이스에 연결 |
| 스택(백) | Spring Boot 4.0.6 · Java 21 · MyBatis(JPA 금지) · MySQL 8 · REST `/api/**` |
| 스택(프) | React 18 · Vite 6 · TS · Tailwind v4 SPA · `/api/*` Vite 프록시 → 8080 |
| 응답 규약 | 항상 `ApiResponse<T>` 엔벨로프(`common/web/ApiResponse`) |
| 인증 | STATELESS + JWT(Access 30분 / Refresh 14일) |
| 데이터 | 약 68개 테이블, 허브 = `users`(A) · `application_case`(B) |
| 팀 | 6인 수직 분담 A~F + AI 기능 #1~34, 각자 사용자화면+API+관리자+AI+DB 소유 |
| AI 통합 | `ai/autoprep` 오케스트레이터가 6영역 도메인 서비스를 의존 그래프로 병렬 실행 |
| 내 영역 | **C(적합도·대시보드, AI #12~18)** — 여정의 중심 허브 |

### 2-2. 사용자 여정 — A에서 F를 가로지른다

```text
 [A 프로필]  스펙 스냅샷(전 영역 공통 입력)
     │
     ▼ 스펙 제공
 [B 공고분석] 공고 구조화: 필수/우대/담당업무/기업현황/면접포인트
     │
     ▼ 필수·우대 조건
 [C 적합도]  ◀── 중심 허브 ──  "점수 + 근거 + 다음 행동"
     │  (A 스냅샷 + B 조건을 비교 → 적합도·부족역량·로드맵·대시보드)
     ▼ 적합도 전략
 [D 가상면접] 예상질문→대화진행→답변평가→리포트
     │
     ▼ 답변·원문
 [E 첨삭/결제] 원문 비수정 개선안(크레딧 소모) · 사용량·요금제
     │
     └──⇢ C로 순환: 사용량·재분석·장기 취업경향(#16)
 [F 커뮤니티]  여정 바깥에서 교차: 실제 면접질문(#31) → D 참고, 후기 → B 참고
```

핵심 명제: **프로필 스냅샷이 전 영역 공통 입력**이고, **C 적합도는 점수만 주지 않고 "근거 + 다음 행동"을 함께** 준다. **E 첨삭은 원문을 덮어쓰지 않고** 별도 저장하며, **F 후기가 다시 B/D 참고로 순환**한다. 즉 여정은 직선이 아니라 C를 중심으로 한 순환 구조다. 단계별 화면은 [사용자 end-to-end 여정](/flow/user-journey), 케이스 중심 데이터 흐름은 [지원 건 중심 흐름](/flow/application-case)에 상세하다.

### 2-3. 요청 한 줄의 생명주기

```text
브라우저 SPA (/api/*)
  → Vite 프록시(5173→8080)
  → JwtAuthenticationFilter (Bearer 파싱 → AuthUser → ROLE_*)
  → @RestController (인가·검증)
  → Service (트랜잭션·비즈니스, MyBatis Mapper로 DB)
  → ApiResponse<T> 직렬화
  ※ SSE(오케스트레이터)만 엔벨로프를 안 타고 text/event-stream 직송
```

[전체 아키텍처](/flow/architecture)에서 4계층·보안 경계를 더 깊이 다룬다.

## 3. 단계별 상세 — 무엇을 받아 무엇을 넘기나

각 단계가 "어느 영역·어떤 입력·어떤 산출·다음으로 무엇을 넘기는가"를 한 줄로 잡아두면, 5분 스크립트가 자동으로 만들어진다.

| 단계 | 영역 | 받는 것(의존 출처) | 만드는 것 | 다음에 넘기는 것 | 대표 클래스/테이블 |
| --- | --- | --- | --- | --- |
| 프로필 | A | 이력서·경력·자소서 | 프로필 요약·스킬·완성도 | 전 영역 공통 입력 | `user_profile`(+`user_profile_version`) |
| 공고분석 | B | 공고원문·OCR | 필수/우대/담당업무/기업/면접포인트 | C 기준·D 입력 | `job_analysis` · `company_analysis` |
| 적합도 | **C** | **A 스냅샷 + B 필수/우대** | 점수·근거·부족역량·로드맵 | D 질문·E 첨삭·장기경향 | `fit_analysis` · `career_analysis_run` |
| 가상면접 | D | **B 공고/포인트 + C 적합도** | 질문·평가·리포트 | C 장기경향·E 첨삭 | `interview_session/question/answer` |
| 첨삭 | E | **D 답변 + A 원문** | 원문 비수정 개선안 | (별도 저장) | `correction_request` |
| 결제/사용량 | E | AI 사용량·크레딧 | 요금제/충전 추천 | — | `payment` · `credit_transaction` |
| 커뮤니티 | F | 후기·태그 | 후기 요약·실제질문 | B/D 참고로 순환 | `community_*` |

볼드는 **타 영역 결과를 입력으로 받는 핵심 의존**이다. 이 표 한 장이 곧 "이 프로젝트는 케이스 하나에 6영역이 직렬·순환으로 엮인다"는 메시지의 증거다. 내 파트 C의 입력→산출 파이프라인은 [적합도 분석](/area-c/fit-analysis)에 가장 깊다.

## 4. 설계 포인트 — 왜 이렇게 연결했나 (경계·정합성)

면접에서 가장 점수가 큰 구간이다. "연결했다"가 아니라 "이 경계를 이렇게 그어서 이런 정합성을 샀다"로 말해야 한다.

### 4-1. 지원 건(Application Case) 단위 — 도메인의 중심을 케이스로 잡은 이유

공고를 단위로 잡으면 "같은 공고에 두 번 지원" "공고는 같아도 내 스펙이 바뀜"을 표현 못 한다. 핵심 행동(분석·면접·첨삭·결제)은 전부 **"이 회사에 이 시점의 나로 지원한다"**는 케이스에 매달리므로, 허브를 `application_case`로 잡았다. 그 결과 프로필·공고·적합도·면접·첨삭·사용량이 자연스럽게 한 케이스로 join된다. 상세: [지원 건 중심 흐름](/flow/application-case).

### 4-2. `ApiResponse<T>` 단일 엔벨로프 — 프런트 계약을 하나로

모든 응답을 `ApiResponse<T>`(success/data/error)로 통일해, 프런트의 `api()` 래퍼가 한 곳에서 에러·토큰 갱신을 처리한다. **예외는 SSE 하나** — 스트림은 엔벨로프를 안 타므로 프런트가 `fetch` + 수동 토큰 + `\n\n` 파싱으로 직접 처리한다. 이 "원칙 + 의도된 예외 1개" 구조를 말하면 일관성과 현실 감각을 동시에 보여준다.

### 4-3. 데이터 소유권 경계 — "각 영역은 자기 결과를 소유, 타 영역 원본은 읽기전용"

| 영역 | 소유(대표) | 타 영역엔 |
| --- | --- | --- |
| A | `users`, `user_profile`(+버전) | 읽기전용 제공, 수정 책임은 A |
| B | `application_case`, `job_analysis`, `company_analysis` | C·D·E에 읽기전용 |
| C | `fit_analysis`, `career_analysis_run` | 원본 비수정, `input_snapshot` 동결 · `input_fingerprint` 캐싱 |
| D | `interview_*`, `interview_agent_step`, `interview_knowledge`(RAG) | C 장기경향·E 첨삭 입력 |
| E | `correction_request`, `payment`, `credit_transaction` | 첨삭 원문/결과 분리 저장 |
| F | `community_*`, `notification`(전영역 공통) | F#31 추출질문 → D 참고 |

핵심 규율 두 개: **(1) 타 영역 원본은 절대 덮어쓰지 않는다**(E 첨삭은 `original_text`/`improved_text` 분리 저장, C는 `source_snapshot` 동결). **(2) `ai_usage_log`는 스키마만 공통이고 기록은 영역별 기록기가 한다.** 이 경계 덕에 6명이 서로의 테이블을 깨지 않고 병렬 개발할 수 있었다. 상세: [데이터 소유권 경계 맵](/flow/data-ownership).

### 4-4. 오케스트레이터 — 새 AI를 안 만들고 기존 도메인 서비스를 래핑

`ai/autoprep`는 **새로운 AI 기능을 만들지 않는다.** 6영역이 이미 가진 도메인 서비스(`fitAnalysisService.generate()` 등)를 6개 핸들러로 감싸 **의존 그래프대로 병렬 실행**한다. 핵심은 의존 그래프가 **단 두 줄**이라는 점이다:

```java
// AutoPrepOrchestrator.java
private static final Map<String, List<String>> DEPS = Map.of(
        "FIT", List.of("JOB"),
        "INTERVIEW", List.of("JOB"));
```

PROFILE(A)·JOB(B)·WRITE(E)·COMMUNITY(F)는 독립이라 **동시 출발**하고, FIT(C)·INTERVIEW(D)만 JOB이 DB에 커밋된 뒤 시작한다. 완주 보장도 명확하다 — 핸들러 없거나 비활성이면 `SKIPPED`, 예외는 `FAILED`로 변환해 **한 파트 실패가 다른 파트로 번지지 않는다.** 상세: [AI 오케스트레이터 전체](/flow/ai-orchestrator), [/area-d/orchestrator-interview](/area-d/orchestrator-interview).

### 4-5. 뉴로-심볼릭 — 점수·판단은 LLM이 아니라 서버 규칙

채용 의사결정 도메인의 통찰: **"왜"를 못 대면 제품 가치가 0이다.** 그래서 C는 LLM에게서 점수를 빼앗았다. AI는 후보 설명을 생성하되, **신뢰도·점수·severity는 입력 상태 기반의 결정적 계산**으로 서버가 확정한다(`FitAnalysisServiceImpl`이 "AI 판단이 아니라 입력 상태 기반 결정적 계산이라 mock/실 AI 모두 동일"이라 주석으로 명시). 이로써 재현·책임·비용·가용성을 동시에 산다. 상세: [/area-c/neuro-symbolic](/area-c/neuro-symbolic).

### 4-6. 폴백 — "공급자 하나 죽어도 시스템은 안 죽는다"

이상적 사상은 `자체 OSS → Claude Haiku → OpenAI → Mock(규칙엔진)`이지만, **정직하게 말하면 영역 편차가 크다.** Haiku 폴백이 실제 코드로 작동하는 곳은 **D(면접)와 두뇌(플래너)뿐**이고, C는 `OSS→OpenAI→Mock`(Haiku 건너뜀), A·B·일부는 OpenAI 실패 시 규칙엔진 폴백만 있다. 자체 모델 생성 화이트리스트(`OSS_GENERATION_TASKS`)는 현재 **빈 집합**이라 생성은 외부 모델이 담당한다. 상세: [AI 공급자·폴백 전략](/flow/ai-providers-fallback).

### 4-7. 크레딧 — 성공 시에만 차감 · 멱등

AI 호출 비용은 `ai_usage_log`에 기록하되 **차감은 status가 SUCCESS일 때만**(`CreditServiceImpl`: `if (!"SUCCESS".equals(status)) return skipped()`), 그리고 중복 차감을 막는 멱등 보장(`ALREADY_DEDUCTED`)이 있다. 즉 **실패한 AI에는 과금하지 않는다.** 요금제 게이팅은 `ai_feature_benefit_policy`가 사용권→크레딧→거절 우선순위를 결정한다. 상세: [크레딧·사용량 흐름](/flow/credit-usage), [/area-e/credit-system](/area-e/credit-system).

## 6. 면접 답변 — 전체를 흐름으로 (1분 / 3분 / 5분)

### 6-1. 1분 (엘리베이터)

> "CareerTuner는 채용공고에 맞춰 스펙과 면접 답변을 조정하는 AI 취업 전략 플랫폼입니다. 핵심 설계는 단위를 '공고'가 아니라 **'지원 건'**으로 잡은 것입니다. 프로필·공고분석·적합도·면접·첨삭·결제가 전부 하나의 케이스에 연결되고, AI 34개 기능이 6개 영역에 수직 분담돼 있습니다. 스택은 Spring Boot·MyBatis·MySQL 백엔드와 React SPA, 응답은 `ApiResponse` 엔벨로프로 통일했습니다. 저는 그중 **C 영역, 적합도·대시보드**를 맡아 여정의 중심 허브를 만들었습니다."

### 6-2. 3분 (구조 + 내 기여)

위 1분에 더해 **세 축**으로 확장한다.

1. **여정**: "A 프로필이 전 영역 공통 입력이 되고, B가 공고를 구조화하면, **C가 둘을 비교해 점수·근거·다음 행동을 냅니다.** 그 적합도가 D 면접 질문의 입력이 되고, D 답변이 E 첨삭으로 가며, 사용량·장기 경향이 다시 C로 순환합니다."
2. **AI 통합**: "이걸 한 번에 돌리는 게 `autoprep` 오케스트레이터입니다. 새 AI를 만들지 않고 6영역 서비스를 래핑해, **의존 그래프 두 줄**(FIT·INTERVIEW는 JOB 뒤)대로 병렬 실행하고 SSE로 진행을 스트리밍합니다."
3. **내 기여(C)**: "제 영역의 핵심 철학은 **뉴로-심볼릭**입니다. '73점입니다'에 근거가 없으면 사용자가 안 믿으니, **점수·판단은 서버 규칙엔진이 확정하고 LLM은 설명만** 하게 했습니다. 여기에 입력 fingerprint 캐시로 같은 입력 재호출을 막았습니다."

### 6-3. 5분 (설계 의도까지)

3분 골격에 **설계 포인트(4장)를 근거로** 살을 붙인다. 추천 순서: 케이스 단위(왜 도메인 중심을 케이스로) → 데이터 소유권 경계(6명이 안 깨지고 병렬 개발) → 오케스트레이터(완주 보장: SKIPPED/FAILED) → 뉴로-심볼릭(재현·책임) → 폴백·크레딧(가용성·비용을 아는 엔지니어) → 구현 상태 정직 구분. 마지막은 반드시 **트레이드오프를 먼저 인정**하며 닫는다: "규칙엔진 방식은 '규칙 품질 = 점수 정교함'이라는 부담을 떠안지만, 채용 도메인에선 신뢰가 우선이라 그 트레이드오프를 택했습니다."

## 7. 꼬리질문 + 모범답안

::: details Q. 왜 '공고'가 아니라 '지원 건'을 핵심 단위로 잡았나요?
사용자의 실제 행동(분석·면접·첨삭·결제)은 "이 회사에, 이 시점의 나로 지원한다"에 매달립니다. 공고를 단위로 잡으면 같은 공고 재지원이나 시점에 따라 달라지는 내 스펙을 표현하지 못합니다. 그래서 허브를 `application_case`로 두고 프로필·공고·적합도·면접·첨삭·사용량을 한 케이스로 묶었습니다.
:::

::: details Q. 6명이 한 코드베이스에서 어떻게 안 부딪혔나요?
**수직 분담 + 읽기전용 경계**입니다. 각자 사용자화면+API+관리자+AI+DB를 통째로 소유하고, 타 영역 테이블은 읽기전용으로만 참조하며 원본을 절대 덮어쓰지 않습니다. 예를 들어 E 첨삭은 원문을 고치지 않고 `correction_request`에 따로 저장하고, C는 `source_snapshot`을 동결합니다. 공통 영역(`common/`, `schema.sql`, 라우팅, 인증)은 팀장 승인이 필요해, 경계 충돌을 구조적으로 막았습니다.
:::

::: details Q. 오케스트레이터가 한 파트에서 터지면 전체가 죽나요?
아닙니다. 핸들러가 없거나 비활성이면 `SKIPPED`, 예외는 try-catch로 `FAILED`로 변환해 기록하고 나머지는 계속 진행합니다. 마지막 요약에 "완료 N·건너뜀 N·실패 N"이 남습니다. 병렬 파트가 같은 SSE emitter에 동시에 쓰므로 `send()`를 `synchronized(emitter)`로 보호하고, 전용 데몬 스레드풀과 5분 타임아웃을 둬 자원 누수도 막았습니다.
:::

::: details Q. 점수까지 LLM에 맡기지 그랬어요? 더 똑똑할 텐데요.
채용 의사결정에서는 "왜 73점인지"를 재현·설명·책임질 수 있어야 제품이 신뢰를 얻습니다. LLM은 같은 입력에도 출력이 흔들리고 근거 추적이 어렵습니다. 그래서 **점수·판단·severity는 결정적 규칙엔진**이 확정하고 LLM은 자연어 설명만 하게 했습니다. 신뢰·재현·책임·비용·가용성을 한 번에 사는 대신 "규칙 품질이 곧 점수 정교함"이라는 부담을 의도적으로 떠안았습니다.
:::

::: details Q. 폴백이 전 영역에 다 적용되나요?
사상은 OSS→Haiku→OpenAI→Mock이지만 **현재 코드로는 영역 편차가 큽니다.** Haiku 폴백이 실제 작동하는 곳은 D(면접)와 두뇌(플래너)뿐이고, C는 OSS→OpenAI→Mock, 일부 영역은 OpenAI 실패 시 규칙엔진 폴백만 있습니다. 자체모델 생성 화이트리스트도 지금은 빈 집합이라 생성은 외부 모델이 합니다. 이렇게 "사상과 현재 구현"을 구분해 말하는 게 정확합니다.
:::

::: details Q. AI 비용 관리는 어떻게 했나요?
두 가지입니다. **(1) 성공 기준 멱등 차감** — `status`가 SUCCESS일 때만 크레딧을 깎고 중복 차감을 막아, 실패한 AI에는 과금하지 않습니다. **(2) 캐시** — C는 입력을 SHA-256 fingerprint로 키 삼아 같은 입력 재호출을 막습니다. 게이팅은 `ai_feature_benefit_policy`가 사용권→크레딧 폴백→거절 순으로 판단합니다.
:::

::: details Q. SSE를 왜 썼나요? WebSocket이 아니라?
오케스트레이터의 `plan`→`part-start`→`substep`→`part-done`→`done`은 서버에서 클라이언트로 흐르는 **단방향 진행 신호**입니다. 양방향이 필요 없으므로 연결 관리 부담이 큰 WebSocket은 과하고, HTTP 인프라와 자연스럽게 맞는 SSE가 더 단순합니다. 다만 SSE는 비동기 재디스패치 때문에 Security에서 `ASYNC/ERROR.permitAll()`을 한 줄 열어줘야 했는데, 그게 오케스트레이터가 건드린 유일한 공통 영역 변경입니다.
:::
