# AI 기능 #1-34 전체 맵

> CareerTuner의 34개 AI 기능은 흩어진 기능 목록이 아니라, **지원 건(Application Case)** 하나를 따라 흐르는 단일 의존 그래프다. A의 프로필 추출이 C의 적합도로, 적합도가 D의 질문으로, 질문이 E의 첨삭으로, F의 후기 질문이 다시 D로 — "어느 기능의 출력이 어느 기능의 입력이 되는가"를 한 장으로 본다.

이 페이지는 흐름(flow) 섹션이라 **연결과 의존**에 집중한다. 각 기능의 프롬프트·신뢰도·스키마 같은 내부 구현은 해당 영역 페이지로 링크한다: [/area-a/](/area-a/) · [/area-b/](/area-b/) · [/area-c/](/area-c/) · [/area-d/](/area-d/) · [/area-e/](/area-e/) · [/area-f/](/area-f/).

## 1. 이 맵이 답하는 면접 질문

- "AI 기능이 34개라는데, 그게 따로 노는 거 아니에요? 어떻게 엮여 있죠?"
- "프로필을 한 번 입력하면 그게 면접 질문·첨삭까지 어떻게 흘러가나요?"
- "적합도 분석은 무슨 데이터를 받아서 무엇을 만들어 어디로 넘기나요?"
- "커뮤니티 후기 같은 외부 데이터가 면접 준비에 어떻게 다시 들어가나요?"
- "AI가 34개면 각각 따로 호출하나요, 아니면 한 번에 묶어 돌리나요?"

핵심 답: **34개 기능은 6영역(A~F)에 수직 분담되어 있고, 영역 간 화살표는 곧 데이터 의존이다.** 그리고 그 흐름을 한 요청으로 묶어 병렬 실행하는 게 AI 오케스트레이터(AutoPrep)다. 전체 오케스트레이션은 [/flow/ai-orchestrator](/flow/ai-orchestrator)에서 따로 다룬다.

## 2. 전체 그림 — 영역별 34기능

번호·명칭·담당은 `docs/TEAM_WORK_DISTRIBUTION.md`의 AI 표(`:49-84`) 기준이다. 각 영역이 자기 기능 + 화면 + API + 관리자 + DB를 수직 소유한다.

| 영역 | 담당 범위 | AI 번호 | 대표 산출물 | 영역 개요 |
|---|---|---|---|---|
| A | 회원·프로필·스펙 추출 | #1~5 | 프로필 요약·기술스택·완성도 | [/area-a/](/area-a/) |
| B | 지원 건·공고·기업 분석 | #6~11 | 공고 구조화·필수/우대·면접포인트 | [/area-b/](/area-b/) |
| C | 적합도·커리어 전략·대시보드 | #12~18 | 점수+근거+다음행동 | [/area-c/](/area-c/) |
| D | 가상 면접·리포트 | #19~23 | 예상질문·답변평가·리포트 | [/area-d/](/area-d/) |
| E | 첨삭·결제·크레딧 | #24~28 | 원문 비수정 개선안·요금추천 | [/area-e/](/area-e/) |
| F | 커뮤니티·고객센터 | #29~34 | 후기요약·실제질문·문의초안 | [/area-f/](/area-f/) |

영역 간 큰 흐름을 화살표로 보면 다음과 같다. A·B가 입력을 만들고, C가 중심 허브에서 비교하고, D가 질문으로 펼치고, E가 다듬고, F가 바깥에서 순환한다.

```text
        [A 프로필]            [B 공고/기업]
        #1 요약               #6 공고분석
        #2 기술스택           #7 필수조건 ─┐
        #3 자소서키워드       #8 우대조건 ─┤
        #4 경력키워드         #11 면접포인트│
            │                    │        │
            └──────┬─────────────┘        │
                   ▼                       │
            [C 적합도 #12]  ◀── A스냅샷 + B필수/우대
                   │ 점수+근거               │
        ┌──────────┼───────────┐            │
        ▼          ▼           ▼            │
     #13 부족   #14 로드맵   #18 대시보드    │
     #15 자격증  #16 장기경향 #17 다음방향   │
                   │                        │
                   ▼                        ▼
            [D 예상질문 #19] ◀── B#6/#11 + C#12 + A스냅샷
                   │
        #20 꼬리질문  #21 면접관진행  #22 답변평가 ──┐
                   │                                │
                   ▼                                ▼
            [D 리포트 #23]                    [E 첨삭 #24]
                                              #25 자소서 #26 이력서 #27 포폴
                                                   │
                                              [E 요금추천 #28]
                                                   │
   [F #29 후기요약 #30 태그 #31 실제질문 #32 추천 #33 검열 #34 문의초안]
                   │
            #31 실제 면접질문 ─────────────▶ D 참고데이터(순환)
            #22 평가 + E결과 ──────────────▶ C#16 장기경향(순환)
```

핵심 두 가지: **(1) C가 비교 허브**라 A와 B의 출력이 여기서 만난다. **(2) 두 개의 순환 고리**가 있다 — F#31(실제 면접질문)이 D로, 그리고 D#22+E결과가 C#16(장기경향)으로 되돌아간다. 사용자 여정 전체는 [/flow/user-journey](/flow/user-journey)에서 화면 단위로 본다.

## 3. 의존 체인 단계별 — 무엇을 받아 무엇을 넘기나

### 3-1. A 입력 생성 (#1~5) — 모든 흐름의 원천

A는 `user_profile`(원천 테이블)을 읽어 구조화한다. 출력은 **프로필 스냅샷**이 되어 전 영역 공통 입력으로 동결된다.

| # | 기능 | 입력 | 출력 → 누구에게 |
|---|---|---|---|
| 1 | 이력서/프로필 요약 | resume·career·projects | 경력·강점 요약 → **C#12 입력** |
| 2 | 기술스택 추출 | resume·projects·skills | 기술명/숙련도 → C·D |
| 3 | 자소서 핵심키워드 | self_intro | 강점·가치관 → **E#25 첨삭 참조** |
| 4 | 경력/프로젝트 키워드 | career·projects | 역할·기술·성과 → D·**E#26 입력** |
| 5 | 프로필 완성도 진단 | user_profile 전체 | 누락·보강 우선순위 |

:::warning A의 5기능 = 3엔드포인트
AI 번호는 5개지만 사용자에게는 `/profile/ai/{summary, skills, completeness}` **3버튼으로 통합 노출**된다. 자소서·경력 키워드(#3·#4)는 요약(#1) 응답의 필드 안에 들어간다. "5엔드포인트"가 아니다. 상세: [/area-a/ai-resume-summary](/area-a/ai-resume-summary).
:::

### 3-2. B 공고 구조화 (#6~11) — 내부에서도 사슬이 있다

B는 공고원문(또는 OCR 텍스트)을 받아 1차로 #6을 만들고, 거기서 #7~11을 파생한다. B 내부 의존이 중요하다 — **#6이 #7/#8/#9/#11의 입력**이다.

```text
공고원문/OCR ─▶ #6 공고분석 ─┬─▶ #7 필수조건 ─┐
                             ├─▶ #8 우대조건 ─┤─▶ C#12 적합도 기준
                             ├─▶ #9 담당업무  │
기업명/외부자료 ─▶ #10 기업현황 ─┤            │
                             └─▶ #11 면접포인트 ─▶ D#19 질문 입력
```

OCR·텍스트 추출이 #6 앞단에 있다: [/area-b/text-extraction-ocr](/area-b/text-extraction-ocr). 필수/우대 추출은 [/area-b/required-preferred](/area-b/required-preferred), 면접포인트는 [/area-b/interview-points](/area-b/interview-points).

### 3-3. C 적합도 허브 (#12~18) — A와 B가 만나는 지점

#12(공고-스펙 적합도)가 흐름의 중심이다. **입력 = A 프로필 스냅샷 + B의 #7/#8(필수/우대)**, 출력 = 점수·근거. 여기서 #13~18이 다시 갈라진다.

| # | 기능 | 입력(의존 출처) | 출력 → |
|---|---|---|---|
| 12 | 공고-스펙 적합도 | **A스냅샷 + B#7/#8** | 점수·근거 → **D#19 질문** |
| 13 | 부족역량 추천 | #12 | 단기/장기 보완 |
| 14 | 학습 로드맵 | #13 | 학습순서·실습 |
| 15 | 자격증 추천 | #13 + 희망직무 | 자격증 우선순위 |
| 16 | 장기 취업경향 | **여러 지원건 + D#22 + E결과** | 반복부족·패턴 (순환 수신) |
| 17 | 다음지원 방향 | #16 + 최근결과 | 재지원 전략 |
| 18 | 대시보드 요약 | #12·#13·#14 | 핵심요약·다음액션 |

:::tip 점수는 AI가 정하지 않는다 (뉴로-심볼릭)
#12에서 LLM은 근거 텍스트를 **생성**할 뿐, 점수와 신뢰도는 서버 규칙으로 결정된다. `FitAnalysisServiceImpl`(`:84-85`)이 "AI 판단이 아니라 입력 상태 기반의 결정적 계산"이라 명시하고, condition_matrix severity·scoreBreakdown도 서버가 계산한다. 그래서 mock AI든 실 AI든 같은 점수가 나온다. 깊이: [/flow/ai-providers-fallback](/flow/ai-providers-fallback) 및 [/area-c/fit-analysis](/area-c/fit-analysis).
:::

### 3-4. D 면접 (#19~23) — 적합도를 질문으로 펼친다

#19(예상 질문)는 **B#6/#11 + C#12 + A스냅샷**을 한꺼번에 받는다. 즉 A·B·C의 결과가 D에서 합류한다. 세션이 돌면 #20(꼬리질문)·#21(면접관 진행)·#22(답변평가)가 작동하고, 세션이 끝나면 #23(리포트)이 전체를 요약한다.

```text
B#6/#11 ─┐
C#12    ─┼─▶ #19 예상질문 ─▶ 세션 ─▶ #20 꼬리질문
A스냅샷  ─┘                      └─▶ #21 면접관 진행
                                 └─▶ #22 답변평가 ─┬─▶ #23 리포트
                                                  ├─▶ E#24 첨삭(입력)
                                                  └─▶ C#16 장기경향(순환)
```

#22(답변평가)는 **세 곳으로 출력이 갈라지는 분기점**이다: 리포트(#23), E 첨삭(#24), C 장기경향(#16). 면접 오케스트레이션·RAG는 [/area-d/orchestrator-interview](/area-d/orchestrator-interview).

### 3-5. E 첨삭·결제 (#24~28) — 원문은 건드리지 않는다

E는 D의 답변과 A의 원문을 받아 개선안을 만든다. **원칙: 원문 덮어쓰기 금지** — `correction_request`에 `original_text`/`improved_text`를 분리 저장한다.

| # | 기능 | 입력(의존 출처) | 비고 |
|---|---|---|---|
| 24 | 면접답변 첨삭 | **D#19/#22** | STAR 개선 |
| 25 | 자소서 첨삭 | **A자소서 + A#3** | 흐름·강점 |
| 26 | 이력서표현 개선 | **A이력서 + A#4** | 성과중심 문장 |
| 27 | 포트폴리오 개선 | 포폴설명 + **A#4** | 문제해결·결과 |
| 28 | 요금제 추천 | **AI사용량 + 크레딧잔액** | 요금/충전 추천 |

#28은 다른 AI들과 결이 다르다 — 콘텐츠가 아니라 **사용량 데이터**(ai_usage_log·크레딧)를 입력으로 받는 메타 기능이다. 크레딧·요금 흐름은 [/flow/credit-usage](/flow/credit-usage)와 [/area-e/credit-system](/area-e/credit-system).

### 3-6. F 커뮤니티 (#29~34) — 여정 바깥에서 순환

F는 사용자 여정 메인 라인 바깥에 있지만, **#31(실제 면접질문 추출)이 D의 참고 데이터로 되돌아간다.** 후기 → 실제질문 추출 → D 예상질문 보강의 순환이다.

| # | 기능 | 입력 | 출력 → | 비고 |
|---|---|---|---|---|
| 29 | 면접후기 요약 | 후기·태그·댓글 | 기업/난이도 요약 | |
| 30 | 게시글 태그추천 | 제목·본문 | 검색/필터 태그 | conf≥0.7 자동 |
| 31 | 실제 면접질문 추출 | 후기·댓글 | **→ D 참고데이터** | 순환 |
| 32 | 관심기반 추천 | 관심직무·읽은글 | 추천글 | bge-m3 코사인 |
| 33 | 부적절/신고 분류 | 신고사유·본문 | 운영분류 | AI 라벨, 운영자 확정 |
| 34 | 고객문의 답변초안 | 문의·FAQ | 초안 | RAG, 미저장·운영자 확정 |

#31 상세는 [/area-f/ai-question-extract](/area-f/ai-question-extract). F의 슬롯수집 챗봇이 면접 준비(D)로 이어지는 계약은 [/area-f/intake-chatbot](/area-f/intake-chatbot).

## 4. 설계 포인트 — 왜 이렇게 엮었나

### 4-1. 화살표 = 데이터 소유권 경계

영역 간 화살표는 "복사"가 아니라 **읽기전용 참조**다. 각 영역은 자기 결과 테이블만 소유하고, 타 영역 원본은 비수정으로 읽는다.

- A `user_profile`이 원천이고 **수정 책임은 A**. B/C/D/E는 읽기만.
- C는 분석 시점의 입력을 `source_snapshot`/`input_snapshot`으로 **동결**하고 `input_fingerprint`로 캐싱한다. 원본이 바뀌어도 그때 점수의 근거가 재현된다.
- E는 첨삭 결과를 원문에 덮어쓰지 않고 별도 행에 저장한다.

데이터 소유권 전체 맵은 [/flow/data-ownership](/flow/data-ownership)에서 테이블 단위로 본다.

### 4-2. 의존을 코드로 강제하는 곳은 단 한 군데

34개 기능 전체를 한 그래프로 그릴 수 있지만, 오케스트레이터가 **실제 코드로 강제하는 의존은 단 두 줄**이다.

```java
// AutoPrepOrchestrator.java:45-47  /  AutoPrepPlanner.java:36-38 (동일)
private static final Map<String, List<String>> DEPS = Map.of(
        "FIT",       List.of("JOB"),
        "INTERVIEW", List.of("JOB"));
```

FIT(C)·INTERVIEW(D)는 JOB(B)이 DB에 커밋된 뒤 시작하고, PROFILE(A)·JOB(B)·WRITE(E)·COMMUNITY(F)는 독립이라 동시 출발한다. 나머지 미세 의존(#13←#12, #14←#13 등)은 각 영역 서비스 **안에서** 순차로 처리되지 표면에 그래프로 노출되지 않는다. 즉 34기능 의존 맵은 "논리적 전체 그림"이고, 오케스트레이터의 실행 의존은 그중 6파트로 추상화한 것이다.

### 4-3. 오케스트레이터는 새 AI를 만들지 않는다

AutoPrep의 6파트 핸들러는 각 영역의 기존 도메인 서비스를 **래핑**할 뿐이다. 새 모델·새 프롬프트가 아니다.

| 파트 key | 호출 서비스 | 영역 | 묶는 AI |
|---|---|---|---|
| PROFILE | `profileAiService.evaluate(...)` | A | #1·#2·#5 |
| JOB | `jobAnalysisService.createJobAnalysis()` | B | #6~11 |
| FIT | `fitAnalysisService.generate()` | C | #12~ |
| WRITE | `correctionService.create("SELF_INTRO",...)` | E | #25 |
| INTERVIEW | `interviewService.createSession()` + `generateQuestions()` | D | #19~ |
| COMMUNITY | `communityPostService.getHotPosts()` | F | (읽기전용) |

이렇게 묶으면 한 영역이 실패해도 다른 파트로 번지지 않는다(핸들러 없음/비활성→SKIPPED, 예외→FAILED로 변환하고 완주). 오케스트레이터의 전체 동작은 [/flow/ai-orchestrator](/flow/ai-orchestrator).

## 5. 구현 상태 — 구현됨 vs 계획

- **34기능의 의존 맵**은 제품 분류와 소유권 계약(`TEAM_WORK_DISTRIBUTION.md`)이다. 일부 번호는 여러 기능을 하나의 엔드포인트·응답 필드로 통합하므로 “34개 독립 API”를 뜻하지 않는다. 현재 구현·모델 활성 조건은 각 영역 페이지와 기준 SHA의 런타임 소스를 따른다.
- **C #12 적합도 신뢰도/점수 = 서버 규칙(결정적)** — 코드로 확인됨(`FitAnalysisServiceImpl.java:84-85`).
- **크레딧 차감 = 성공 시에만, 멱등** — 코드로 확인됨(`CreditServiceImpl.java:47-48,51`). 실패한 AI는 미차감.
- **폴백(자체 OSS → Claude Haiku → OpenAI → Mock)은 사상이며 영역 편차가 크다.** Claude Haiku 폴백이 실제 코드로 작동하는 건 현재 **D(면접) + 두뇌(플래너)뿐**이고, 자체 OSS 생성 화이트리스트는 현재 빈 집합이다. C는 OSS→OpenAI→Mock(Haiku 건너뜀), A/B/일부 C/E는 OpenAI+규칙 fallback, F·인테이크 챗봇은 Ollama 직접(폴백 없음). 상세: [/flow/ai-providers-fallback](/flow/ai-providers-fallback).
- 통합 클래스 설계서의 담당별 자체모델(C `careertuner-c-career-strategy-3b` 등)은 **팀 합의 목표**이며, 런타임 적용 여부는 각 파트 본문 표기를 따른다.

## 6. 면접 답변 — 34기능을 흐름으로 한 번에

> "AI 기능이 34개 있는데, 핵심은 **하나의 지원 건을 따라 흐르는 의존 그래프**라는 점입니다. A가 프로필을 구조화해 요약·기술스택·키워드를 뽑으면, 그게 스냅샷으로 동결됩니다. B는 공고를 받아 #6 공고분석에서 필수·우대·면접포인트를 파생하고요. 이 둘이 만나는 허브가 C의 #12 적합도입니다 — A 스냅샷과 B의 필수/우대 조건을 비교해 점수와 근거를 냅니다. 여기서 중요한 건 **점수는 LLM이 아니라 서버 규칙으로 결정**한다는 거예요. AI는 근거 설명만 생성합니다. 적합도가 나오면 D가 그걸 받아 예상 질문을 만들고, 답변 평가(#22)는 다시 세 갈래로 갈라집니다 — 리포트, E의 첨삭 입력, 그리고 C의 장기 경향 분석으로 되돌아갑니다. E는 원문을 절대 덮어쓰지 않고 개선안을 별도 저장합니다. F 커뮤니티는 여정 바깥에 있지만 후기에서 실제 면접질문(#31)을 추출해 D로 순환시킵니다. 이 흐름 전체를 한 요청으로 묶어 병렬 실행하는 게 AutoPrep 오케스트레이터인데, 의존은 'FIT·INTERVIEW는 JOB 뒤'라는 단 두 줄로 강제하고 나머지는 동시에 출발시킵니다. 오케스트레이터는 새 AI를 만들지 않고 각 영역 서비스를 래핑할 뿐이라, 한 파트가 실패해도 나머지는 완주합니다."

## 7. 꼬리질문 + 모범답안

:::details "34개를 다 따로 호출하면 느리지 않나요?"
사용자가 기능을 하나씩 누르면 그 기능만 호출됩니다. 하지만 "통째로 준비해줘"라고 하면 AutoPrep 오케스트레이터가 의존 그래프대로 병렬 실행합니다. PROFILE·JOB·WRITE·COMMUNITY는 독립이라 동시에 출발하고, FIT·INTERVIEW만 JOB이 끝난 뒤 시작합니다(`CompletableFuture.allOf(...).thenRunAsync`). 그래서 직렬 합보다 빠르고, 진행 상황은 SSE로 실시간 전송합니다.
:::

:::details "C의 적합도 점수를 LLM에 안 맡기는 이유는?"
재현성과 책임 때문입니다. 같은 입력에 같은 점수가 나와야 사용자가 신뢰하고, 나중에 "왜 이 점수였나"를 감사할 수 있습니다. LLM은 비결정적이라 점수를 맡기면 호출마다 흔들립니다. 그래서 점수·신뢰도·severity는 condition_matrix 기반 서버 규칙으로 확정하고(`FitAnalysisServiceImpl:84-85`), LLM은 사람이 읽을 근거 설명만 생성합니다. 이게 뉴로-심볼릭 분담입니다.
:::

:::details "F가 여정 바깥인데 왜 AI 기능으로 묶었나요?"
F의 가치는 순환에 있습니다. #31이 실제 후기에서 진짜 나온 면접질문을 뽑아 D의 예상질문을 보강하고, 후기 요약(#29)은 다른 지원자의 공고 참고 자료가 됩니다. 단방향 여정만 보면 안 보이지만, 데이터가 커뮤니티를 거쳐 다시 준비 흐름으로 돌아오는 두 번째 고리입니다.
:::

:::details "AI가 실패하면 크레딧은 어떻게 되나요?"
차감되지 않습니다. 크레딧 차감은 success-gated이고 멱등합니다 — `CreditServiceImpl`이 status가 SUCCESS가 아니면 skip하고(`:47-48`), 중복 차감도 ALREADY_DEDUCTED로 막습니다(`:51`). 사용량 로그(`ai_usage_log`)도 최종 성공 모델만 기록하고, 폴백 횟수는 앱 로그 warn에만 남깁니다. 사용자는 실패한 호출에 돈을 내지 않습니다.
:::
