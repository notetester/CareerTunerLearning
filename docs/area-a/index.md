# 영역 A 개요 — 회원 · 프로필 · 인증

> A는 CareerTuner의 "기반 신뢰 데이터 소유자"다. 사용자가 **누구인지(인증)**, **자신을 어떻게 설명하는지(프로필)**, **무엇을 보완해야 하는지(AI 진단)** 를 책임지고, 이 데이터를 B·C·D·E 전 영역이 입력 원천으로 읽어 간다.

## 1. 이 영역은 무엇이고, 어떤 면접 질문에 답하는가

CareerTuner의 핵심 단위는 공고가 아니라 **지원 건(Application Case)** 이다. 그런데 그 지원 건을 분석하려면 먼저 "분석 대상이 되는 사람"이 있어야 한다. 그 사람의 **계정·로그인 상태**와 **스펙 원천 데이터**를 보유·관리하는 곳이 영역 A다.

A를 한 문장으로 정의하면 이렇다.

> **A는 인증(누구인지)과 프로필(어떻게 설명하는지)이라는 두 종류의 "기반 신뢰 데이터"를 소유하고, 거기에 프로필을 진단하는 AI 기능을 얹은 영역이다.**

이 개요 페이지는 다음 면접 질문에 답하기 위한 지도다.

- "이 프로젝트에서 사용자 데이터는 어디서 시작되고, 누가 소유하나요?"
- "AI 분석들이 공통으로 참조하는 입력은 무엇인가요?"
- "회원·인증 영역에서 본인이 맡은 게 정확히 어디까지인가요?"
- "프로필을 AI로 분석한다는데, 구체적으로 어떤 기능이고 몇 개인가요?"

세부 주제(데이터 모델, JWT, OAuth, AI 진단, 동의 게이팅 등)는 각각 별도 페이지로 깊게 다룬다. 이 페이지는 **그 페이지들로 들어가기 전에 전체 그림을 잡는 장**이다.

## 2. 정체성 — "기반 신뢰 데이터 소유자"와 읽기전용 경계

A의 정체성을 이해하는 핵심은 **소유와 참조의 분리**다.

- **A가 소유(쓰기)하는 것:** 계정 상태, 인증/세션 이력, 동의 이력, 그리고 사용자가 직접 입력한 프로필 1행.
- **다른 영역이 하는 것:** A의 프로필을 **읽어서** 자기 분석의 입력으로 쓸 뿐, **원본 프로필을 수정하지 않는다.**

```text
            ┌─────────────────────── 영역 A (소유·쓰기) ───────────────────────┐
            │  users · user_social · refresh_token · user_login_history          │
            │  user_status_history · email_verification · user_consent           │
            │  user_profile (1:1)  ← 스펙 원천 데이터                              │
            └────────────────────────────┬──────────────────────────────────────┘
                                         │ 읽기 전용 참조
        ┌────────────────┬───────────────┼────────────────┬────────────────┐
        ▼                ▼               ▼                ▼                ▼
   B 공고 분석      C 적합도 분석     C 경향 분석      D 면접 질문      E 첨삭
   (프로필 읽음)   (프로필 vs 공고)  (여러 건 패턴)   (프로필 기반)    (프로필 참조)
```

:::tip 경계를 한 문장으로
"A는 프로필의 **단일 쓰기 책임자**이고, B·C·D·E는 A 프로필의 **읽기 소비자**입니다. 그래서 스펙 데이터 정합성의 책임이 한 곳으로 모입니다."
:::

이 "읽기전용 원칙"이 중요한 이유는 **데이터 정합성의 책임 소재**를 한 곳으로 모으기 때문이다. 만약 적합도 분석(C)이 분석 도중 프로필을 임의로 고칠 수 있다면, "지금 이 사용자의 진짜 스펙이 뭐냐"는 질문에 영역마다 답이 달라진다. A가 유일한 쓰기 주체라는 규칙이 이 혼란을 막는다.

## 3. A가 빌려 쓰는 공통 인프라 (소유 아님)

A는 자급자족하지 않는다. 몇몇 공통 인프라는 **소유하지 않고 어댑터/참조로만** 쓴다. 면접에서 "그건 네가 만든 거냐"는 질문이 들어왔을 때 정직하게 구분할 수 있어야 한다.

| 공통 자원 | 실제 소유 | A의 사용 방식 |
| --- | --- | --- |
| 공유 AI 프로바이더 (`CareerAnalysisOpenAiClient`) | C(팀장 공통 엔진) | A의 `OpenAiProfileAiService`가 **어댑터로 감싸서** 호출만 |
| `ai_usage_log` 테이블 | **스키마 소유는 A**, append는 전 영역 공통 | A 프로필 AI는 B 도메인 매퍼(`insertAiUsageLog`)를 빌려 기록 |
| 알림 설정 (`NotificationSettings`) | F 영역 | A의 설정 화면에 알림 탭으로 **임베드만** |

여기서 미묘한 지점 하나. `ai_usage_log`는 **테이블 스키마 소유 자체는 A**지만, 거기에 기록을 남기는 행위는 모든 영역이 공유한다. 그래서 A의 프로필 AI도 자기 매퍼가 아니라 B 도메인의 매퍼를 빌려서 insert한다. "AI 사용량 로그의 스키마 주인은 A, 사용자는 전부"라고 말하면 정확하다.

## 4. 담당 AI 기능 — 계획 5개 vs 실제 구현 3개 엔드포인트

여기가 A를 설명할 때 **가장 사실 오류가 나기 쉬운 지점**이다. 분담 계획서는 프로필 AI를 #1~#5 다섯 기능으로 나열하지만, 실제 코드(`ProfileController.java`)에는 AI 엔드포인트가 **정확히 3개뿐**이다.

```java
// ProfileController — AI 엔드포인트는 이게 전부 (요청 바디 없이 인증 주체만 받음)
POST /api/profile/ai/summary       → service.summarize(authUser)
POST /api/profile/ai/skills        → service.extractSkills(authUser)
POST /api/profile/ai/completeness  → service.diagnoseCompleteness(authUser)
```

계획상 5기능이 실제 구현에 어떻게 매핑되는지가 핵심이다.

| # | 계획상 AI 기능 | 실제 구현 위치 | featureType / 엔드포인트 |
| --- | --- | --- | --- |
| 1 | 이력서/프로필 요약 | `ProfileAiResult.summary` | `PROFILE_SUMMARY` / `POST /profile/ai/summary` |
| 2 | 기술스택 추출 | `extractedSkills` | `PROFILE_SKILL_EXTRACT` / `POST /profile/ai/skills` |
| 3 | 자기소개서 키워드 | 요약 응답의 `strengths`에 **흡수** | 전용 엔드포인트 없음 |
| 4 | 경력/프로젝트 키워드 | 요약 응답의 `gaps`+`evidence`에 **흡수** | 전용 엔드포인트 없음 |
| 5 | 프로필 완성도 진단 | 6기준 가중 합산 | `PROFILE_COMPLETENESS` / `POST /profile/ai/completeness` |

:::warning #3·#4는 "구현됨"이 아니다
자소서 키워드(#3)와 경력·프로젝트 키워드(#4)는 **독립된 엔드포인트·서비스·featureType이 존재하지 않는다.** 요약 응답의 `strengths`/`gaps`/`evidence` 필드에 부분적으로 흡수되어 있을 뿐이다. 프론트도 버튼은 3개(요약 / 역량추출 / 완성도진단)만 노출한다. 면접에서 "5개 다 만들었다"고 말하면 사실과 어긋난다 — **"3개 엔드포인트로 통합했고, 키워드 두 기능은 요약 응답 필드에 녹였다"** 가 정확하다.
:::

세 기능 모두 단일 진입점 `ProfileAiService.evaluate(UserProfile, featureType)` 를 호출하고, **featureType 문자열로만 분기**하며 출력 매핑만 다르다. 깊은 동작 원리(직무군 분류 → 가중치 → 서버 점수 계산, 그리고 키 미발급 시 규칙엔진 폴백)는 각 세부 페이지에서 다룬다.

## 5. 구현 상태 — 됨 vs 계획만 (정직 구분)

면접 신뢰도는 "정직하게 구분할 줄 안다"에서 나온다.

**구현 완료된 것**

- 인증 전체: 회원가입 / 로그인 / 토큰 갱신 / 로그아웃 / 전기기 로그아웃 / 내 정보 조회
- 이메일 인증(VERIFY), 비밀번호 재설정(RESET_PW), 휴면 해제(DORMANT_RELEASE)
- 소셜 OAuth(google / kakao / naver), 5회 실패 시 10분 자동 잠금, refresh 토큰 회전
- 프로필 CRUD(단일 행 upsert)
- AI #1 요약 · #2 스킬추출 · #5 완성도 진단 (규칙엔진은 항상 동작, LLM은 키 주입 시 활성)
- 동의(append-only 이력 + AI 실행 게이트)
- 관리자 회원/프로필/동의 화면 (회원 상태변경만 쓰기, 프로필·동의는 읽기 전용)

**미구현 / 계획만 / 타영역 (구현됨으로 적으면 오류)**

- `user_profile_version`(분석 재현성용 프로필 스냅샷 테이블) — 스키마에 없음. 프로필은 **단일 행 upsert + `updated_at`** 만 있고 버전 이력 미구현. 관리자의 "프로필 스냅샷"은 버전 테이블이 아니라 현재 1행을 평면화한 읽기 모델이다.
- AI #3 자소서 키워드 · #4 경력 키워드 — 전용 엔드포인트/서비스 없음(요약에 흡수)
- 휴면 전환(ACTIVE→DORMANT) 배치 — 해제 경로만 있고 전환 스케줄러는 없음
- 회원가입 `userType` 저장 — 프론트 UI는 받지만 register 요청 본문에 미전달

**설계만 / 진행중**

- 자체 파인튜닝 모델 `careertuner-a-profile-3b`(Qwen2.5-3B LoRA) — 1차 경로로 설계됐으나 키 미발급이라 **운영 기본값은 규칙엔진**(`profile-rule-v2`, status=SUCCESS). 폴백 구조로 보호된다.

:::details 보안 트레이드오프(정직하게 짚어둘 관찰점)
완벽하지 않은 부분도 알고 있어야 한다. (1) 비밀번호 재설정 요청이 미존재 이메일에 404를 주어 계정 존재 여부가 노출될 수 있고, (2) OAuth 콜백이 토큰을 URL `#fragment`로 전달하며, (3) 토큰을 localStorage에 보관(XSS 노출면), (4) 클라이언트 라우트 가드가 없다(서버 401에 의존). 면접에서 "개선 여지를 어디서 보나"는 질문에 이걸 답하면 깊이가 드러난다.
:::

## 6. 권장 학습 순서

A는 "데이터 모델 → 인증 → 동의 → AI 진단 → 화면" 순으로 보면 의존관계가 자연스럽게 풀린다.

1. [프로필 데이터 모델](/area-a/profile-data-model) — 모든 것의 출발점. users 중심 트리, 1:1 프로필, JSON 컬럼 8종, 동의 append-only.
2. [JWT 인증 흐름](/area-a/auth-jwt) — access(무상태 JWT) + refresh(DB opaque UUID) 이원화, 상태 검증, 자동 잠금.
3. [OAuth2 소셜 로그인](/area-a/oauth-social) — Spring oauth2Login 미사용, 수동 REST + 서명 state JWT.
4. [비밀번호·이메일 인증](/area-a/password-email) — BCrypt, 메일 인증/재설정/휴면해제 토큰의 purpose 분기.
5. [동의 관리·게이팅](/area-a/consent-gating) — append-only 이력, "현재 동의"=최신 1행, AI 실행 전제 조건.
6. [이력서/프로필 요약](/area-a/ai-resume-summary) · [기술스택 추출](/area-a/ai-skill-extraction) · [완성도 진단](/area-a/ai-profile-completeness) — AI 3기능. 뉴로-심볼릭 분리(서버가 점수 계산)와 폴백이 핵심.
7. [자소서/경력 키워드](/area-a/ai-keyword-extraction) — 왜 독립 기능이 아니라 요약에 흡수됐는지.
8. [프론트엔드 UI/UX](/area-a/frontend-ui) · [관리자 화면](/area-a/admin) — 화면 구조와 관리자 위임 재사용.
9. [면접 플레이북](/area-a/interview-playbook) — A 전체를 면접 답변으로 압축.

공통 기반이 궁금하면 곁다리로 [공통 구조화 출력](/ai/openai-structured-output), [폴백 전략](/ai/fallback), [JWT 보안](/backend/jwt-security), [OAuth2](/backend/oauth2) 페이지를 참고하면 좋다. 다른 영역과의 연결은 [영역 C 개요](/area-c/)에서 적합도 분석이 A 프로필을 어떻게 소비하는지 확인할 수 있다.

## 7. 단골 면접 질문 5개

면접에서 A를 물어볼 때 반복되는 다섯 질문과 핵심 답의 방향이다. (상세 모범답안은 각 세부 페이지에 있다.)

**Q1. 이 프로젝트에서 사용자 데이터의 "주인"은 누구인가요?**
A는 인증 데이터와 프로필(스펙 원천)을 소유하는 영역이고, 프로필의 **단일 쓰기 책임자**입니다. B·C·D·E는 이 프로필을 **읽기만** 합니다 — 그래서 스펙 정합성 책임이 한 곳으로 모입니다.

**Q2. access 토큰과 refresh 토큰을 왜 다르게 설계했나요?**
access는 짧은 수명의 **무상태 JWT**(서버가 상태를 안 들고 검증), refresh는 **DB에 저장된 opaque UUID**입니다. refresh를 DB에 두는 이유는 회전·폐기·세션 감사(IP/UA 기록) 때문입니다. 무상태 JWT의 약점(즉시 무효화 불가)을 짧은 수명 + refresh 폐기로 보완합니다.

**Q3. AI로 프로필을 분석한다는데, LLM이 점수를 멋대로 만들지 않나요?**
아닙니다. **서버가 점수를 계산합니다.** LLM에는 각 기준의 원점수(0~100)와 근거·개선 문장만 요구하고, 직무군별 가중치로 총점을 합산하는 건 서버(`ProfileScoreCalculator`)입니다. 이 뉴로-심볼릭 분리 덕에 재현성과 직무군 공정성을 서버가 강제합니다.

**Q4. 키가 없는데 AI 기능이 어떻게 동작하나요?**
2단 폴백 구조입니다. 키가 없으면 즉시 결정론적 **규칙엔진**(`RuleBasedProfileAiService`)으로 동작하고 status는 SUCCESS, 모델은 `profile-rule-v2`로 남습니다. LLM 호출이 떠도 실패하면 규칙엔진 결과에 status=FALLBACK을 붙입니다. 운영자가 응답·로그의 status로 폴백 발생을 추적할 수 있습니다.

**Q5. 동의(consent)는 왜 그냥 컬럼 하나로 안 하고 이력 테이블로 만들었나요?**
개인정보보호 요건상 "언제 동의·철회했는지"를 완전히 재구성할 수 있어야 하기 때문입니다. `user_consent`는 현재값을 덮어쓰지 않는 **append-only 이벤트 이력**이고, "현재 동의"는 같은 type의 최신 1행이 `agreed=true AND revoked_at IS NULL`인지로 판정합니다. AI 실행은 `AI_DATA` 동의가 켜져 있어야만 통과하는 게이트로 묶여 있습니다.
