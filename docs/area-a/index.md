# 영역 A 개요 — 회원·프로필·인증 (AI #1-5)

> 영역 A는 CareerTuner의 **기반 신뢰 데이터 소유자**다. 사용자가 **누구인지(인증)**, **자신을 어떻게 설명하는지(프로필)**, **무엇을 보완해야 하는지(AI 진단)** 를 책임지며, 이 데이터를 B·C·D·E·F가 입력 원천으로 읽어 간다. CareerTuner의 핵심 단위는 공고가 아니라 **지원 건(Application Case)** 이고, 그 지원 건을 분석하려면 먼저 "분석 대상이 되는 사람"이 있어야 한다 — 그 사람을 소유하는 곳이 A다.

상위 허브로 돌아가기: [영역별 심화 전체 개요](/areas/) · [전체 흐름](/flow/) · [AI #1-34 맵](/flow/ai-function-map)
인접 영역: [영역 B 개요](/area-b/) (지원 건·공고) · [영역 F 개요](/area-f/) (커뮤니티·챗봇)

## 1. 이 영역이 책임지는 것 — 정체성

영역 A를 한 문장으로 정의하면 이렇다.

> **A는 인증(누구인지)과 프로필(어떻게 설명하는지)이라는 두 종류의 "기반 신뢰 데이터"를 소유하고, 거기에 프로필을 진단하는 AI 기능을 얹은 영역이다.**

A의 정체성을 이해하는 핵심은 **소유와 참조의 분리**다.

- **A가 소유(쓰기)하는 것:** 계정 상태, 인증/세션 이력, 동의 이력, 그리고 사용자가 직접 입력한 프로필 1행(`user_profile`, 1:1).
- **다른 영역이 하는 것:** A의 프로필을 **읽어서** 자기 분석의 입력으로 쓸 뿐, **원본 프로필을 수정하지 않는다.**

이 "읽기전용 원칙"이 중요한 이유는 **데이터 정합성의 책임 소재**를 한 곳으로 모으기 때문이다. 만약 적합도 분석(C)이 분석 도중 프로필을 임의로 고칠 수 있다면, "지금 이 사용자의 진짜 스펙이 뭐냐"는 질문에 영역마다 답이 달라진다. A가 유일한 쓰기 주체라는 규칙이 이 혼란을 막는다. 전체 소유권 그림은 [데이터 소유권 맵](/flow/data-ownership)에서 6영역을 가로질러 본다.

A가 소유하는 대표 테이블(`db/schema.sql` 기준):

| 분류 | 테이블 | 역할 |
| --- | --- | --- |
| 계정 허브 | `users` | 회원의 모든 것이 매달리는 FK 허브 |
| 소셜 연결 | `user_social` | OAuth 제공자별 식별자 매핑 |
| 메일 토큰 | `email_verification` | 가입 인증·비번 재설정·휴면 해제 토큰(purpose 분기) |
| 세션 | `refresh_token` | DB 저장 opaque refresh 토큰(회전·폐기) |
| 감사 | `user_login_history`, `user_status_history` | 로그인 시도·상태 변경 이력 |
| 동의 | `user_consent` | append-only 동의 이벤트 이력 |
| 스펙 원천 | `user_profile` | `user_id` UNIQUE, 1:1. skills·career·projects·self_intro 등 |

:::tip 경계를 한 문장으로
"A는 프로필의 **단일 쓰기 책임자**이고, B·C·D·E·F는 A 프로필의 **읽기 소비자**입니다. 그래서 스펙 데이터 정합성의 책임이 한 곳으로 모입니다."
:::

## 2. 6개 영역 속에서 이 영역의 위치

CareerTuner는 6명이 한 지원 건을 **수직 분담**으로 함께 채운다. 6개 영역을 한 줄씩 보면 A가 어디에 서 있는지 또렷해진다.

| 영역 | 담당 범위 | AI 번호 | 대표 산출물 | 개요 |
| --- | --- | --- | --- | --- |
| **A (이 페이지)** | **회원·프로필·인증** | **#1~5** | **프로필 요약·기술스택·완성도** | **/area-a/** |
| B | 지원 건·공고·기업 분석 | #6~11 | 공고 구조화·필수/우대·면접포인트 | [/area-b/](/area-b/) |
| C | 적합도·커리어 전략·대시보드 | #12~18 | 점수+근거+다음 행동 | [/area-c/](/area-c/) |
| D | 가상 면접·리포트 | #19~23 | 예상질문·답변평가·리포트 | [/area-d/](/area-d/) |
| E | 첨삭·결제·크레딧 | #24~28 | 원문 비수정 개선안·요금 추천 | [/area-e/](/area-e/) |
| F | 커뮤니티·고객센터·챗봇 | #29~34 | 후기 요약·실제 질문·문의 초안 | [/area-f/](/area-f/) |

A는 이 표에서 **맨 위 = 데이터의 시작점**이다. 데이터가 어디서 와서 어디로 가는지 정리하면 이렇다.

```text
            ┌─────────────────────── 영역 A (소유·쓰기) ───────────────────────┐
            │  users · user_social · refresh_token · user_login_history          │
            │  user_status_history · email_verification · user_consent           │
            │  user_profile (1:1)  ← 스펙 원천 데이터                              │
            └────────────────────────────┬──────────────────────────────────────┘
                                         │ 읽기 전용 참조 (원본 비수정)
     ┌────────────┬──────────────┬───────┼──────────────┬──────────────┐
     ▼            ▼              ▼       ▼              ▼              ▼
  B 공고분석   C 적합도분석   C 경향분석  D 면접질문    E 첨삭        F 챗봇/커뮤니티
 (프로필읽음) (프로필vs공고) (여러 건패턴)(프로필기반)  (프로필참조)  (프로필 참조)
```

- **A가 받는 입력:** 외부 없음. 사용자가 직접 입력한 프로필과 OAuth 제공자(Kakao/Naver/Google) 응답이 원천이다.
- **A가 내보내는 출력:** `user_profile` 현재 행, `user_profile_version` 불변 버전, 저장된 프로필 AI 분석이 전 영역의 공통 입력이 된다. 단, **원본 쓰기는 오직 A만** 한다.
- **정합성 포인트:** 현재 행은 저장 때 `version_no`가 증가하고 같은 transaction에서 불변 버전이 쌓인다. 프로필 AI 결과는 실제 입력 version ID를 가리키며, C는 프로필+공고를 조합한 실행 snapshot을 추가로 보존한다.

:::tip 현재 값과 버전은 둘 다 있다
`user_profile`은 사용자별 현재 1행이고, `user_profile_version`은 저장·문서 가져오기·AI 평가 시점의 불변 이력이다. 관리자 화면도 현재 상세와 버전 목록을 별도 API로 조회한다.
:::

전 영역이 어떻게 한 요청으로 묶여 병렬 실행되는지는 [AI 오케스트레이터](/flow/ai-orchestrator)에서, A→C→D로 이어지는 큰 흐름은 [사용자 여정](/flow/user-journey)·[아키텍처](/flow/architecture)에서 본다.

## 3. 담당 AI 기능 — 계획 #1~5 vs 실제 엔드포인트 3개

여기가 A를 설명할 때 **가장 사실 오류가 나기 쉬운 지점**이다. 분담 계획서는 프로필 AI를 #1~#5 다섯 기능으로 나열하지만, 실제 코드(`ProfileController.java`)에는 AI 엔드포인트가 **정확히 3개**다.

```text
ProfileController — AI 엔드포인트는 이게 전부 (요청 바디 없이 인증 주체만 받음)
POST /api/profile/ai/summary       → 이력서/프로필 요약
POST /api/profile/ai/skills        → 기술스택 추출
POST /api/profile/ai/completeness  → 프로필 완성도 진단
```

계획상 5기능이 실제 구현에 어떻게 매핑되는지가 핵심이다.

| # | 계획상 AI 기능 | 실제 구현 | featureType / 엔드포인트 |
| --- | --- | --- | --- |
| 1 | 이력서/프로필 요약 | 요약 응답 `summary` | `PROFILE_SUMMARY` / `POST /profile/ai/summary` |
| 2 | 기술스택 추출 | `extractedSkills` | `PROFILE_SKILL_EXTRACT` / `POST /profile/ai/skills` |
| 3 | 자기소개서 키워드 | 요약 응답의 `strengths`에 **흡수** | 전용 엔드포인트 없음 |
| 4 | 경력/프로젝트 키워드 | 요약 응답의 `gaps`+`evidence`에 **흡수** | 전용 엔드포인트 없음 |
| 5 | 프로필 완성도 진단 | 6기준 가중 합산 | `PROFILE_COMPLETENESS` / `POST /profile/ai/completeness` |

:::warning #3·#4는 "구현됨"이 아니다
자소서 키워드(#3)와 경력·프로젝트 키워드(#4)는 **독립된 엔드포인트·서비스·featureType이 존재하지 않는다.** 요약 응답의 `strengths`/`gaps`/`evidence` 필드에 부분적으로 흡수되어 있을 뿐이다. 프론트도 버튼은 3개(요약 / 역량추출 / 완성도진단)만 노출한다. 정확한 표현은 **"3개 엔드포인트로 통합했고, 키워드 두 기능은 요약 응답 필드에 녹였다"** 이다.
:::

세 기능 모두 단일 진입점 `ProfileAiService.evaluate(UserProfile, featureType, requestedModel)`을 호출한다. 자체·Claude·OpenAI provider와 규칙 안전망이 같은 출력 계약을 사용하고, 점수는 서버가 재계산한다. 성공 결과는 `profile_ai_analysis` 최신본과 입력 version ID로 저장된다.

::: details 구현 상태 — 됨 vs 계획만 (정직 구분)
**구현 완료**
- 인증 전체: 회원가입 / 로그인 / 토큰 갱신 / 로그아웃 / 전 기기 로그아웃 / 내 정보 조회
- 메일 인증·비밀번호 재설정·휴면 해제(`email_verification` purpose 분기)
- 소셜 OAuth(google/kakao/naver), 5회 실패 시 자동 잠금, refresh 토큰 회전
- 프로필 CRUD + 불변 버전 이력 + 포트폴리오 파일·문서 import
- AI #1 요약·#2 스킬추출·#5 완성도, 모델 선택·다단 폴백·최신 결과 영속
- 동의(append-only 이력 + AI 실행 게이트), 관리자 회원/프로필/동의 화면

**미구현 / 계획만 / 타영역**
- AI #3 자소서 키워드·#4 경력 키워드 — 전용 엔드포인트/서비스 없음(요약에 흡수)
- 휴면 전환(ACTIVE→DORMANT) 배치 — 해제 경로만 있고 전환 스케줄러 없음

**상태를 구분해 말할 것**
- Qwen3 4B Profile LoRA v4의 학습·비교 기록은 있으나 runtime 기본 활성과 새 clone 재현 가능성은 별도다. [모델 근거 매트릭스](/ai/model-evidence-matrix) 참고.
:::

## 4. 권장 학습 순서

A는 "데이터 모델 → 인증 → 동의 → AI 진단 → 화면" 순으로 보면 의존관계가 자연스럽게 풀린다. 아래 하위 페이지를 주제별로 묶었다.

**1단계 — 데이터의 출발점**

- [프로필 데이터 모델](/area-a/profile-data-model) — `users` 중심 트리, 1:1 프로필, JSON 컬럼, 동의 append-only. 모든 것의 기준점.

**2단계 — 인증/세션 (누구인지)**

- [JWT 인증 흐름](/area-a/auth-jwt) — access(무상태 JWT) + refresh(DB opaque UUID) 이원화, 상태 검증, 자동 잠금.
- [OAuth2 소셜 로그인](/area-a/oauth-social) — 수동 REST + 서명 state JWT(Kakao/Naver/Google).
- [비밀번호·이메일 인증](/area-a/password-email) — BCrypt, 메일 인증/재설정/휴면 해제 토큰의 purpose 분기.

**3단계 — 동의 게이트 (AI 실행 전제)**

- [프로필 버전·스냅샷](/area-a/profile-versioning) — 현재 행과 불변 버전, 분석 입력 provenance와 동시 저장 정합성을 어떻게 지키는가.
- [동의 관리·게이팅](/area-a/consent-gating) — append-only 이력, "현재 동의"=최신 1행, AI 실행 전제 조건.

**4단계 — AI 진단 #1~5 (무엇을 보완해야 하는지)**

- [이력서/프로필 요약](/area-a/ai-resume-summary) (#1)
- [기술스택 추출](/area-a/ai-skill-extraction) (#2)
- [자소서/경력 키워드](/area-a/ai-keyword-extraction) (#3·#4 — 왜 요약에 흡수됐는가)
- [프로필 완성도 진단](/area-a/ai-profile-completeness) (#5 — 서버가 점수 계산하는 뉴로-심볼릭 분리)

**5단계 — 화면·관리자·면접 정리**

- [프론트엔드 UI/UX](/area-a/frontend-ui) — 사용자 화면 구조.
- [관리자 화면](/area-a/admin) — 회원/프로필/동의 관리(상태변경만 쓰기, 나머지 읽기 전용).
- [면접 플레이북](/area-a/interview-playbook) — A 전체를 면접 답변으로 압축.

다른 영역과의 연결은 [영역 B 개요](/area-b/)(공고 구조화)와 [영역 C 개요](/area-c/)(적합도 분석이 A 프로필을 어떻게 소비하는지)에서, 프로젝트 전체 면접 관점은 [전체 프로젝트 면접](/flow/interview-whole-project)에서 이어진다.

## 5. 이 영역 단골 면접 질문 5개

면접에서 A를 물어볼 때 반복되는 다섯 질문과 핵심 답의 방향이다. (상세 모범답안은 각 세부 페이지에 있다.)

**Q1. 이 프로젝트에서 사용자 데이터의 "주인"은 누구인가요?**
A는 인증 데이터와 프로필(스펙 원천)을 소유하는 영역이고, 프로필의 **단일 쓰기 책임자**입니다. B·C·D·E·F는 이 프로필을 **읽기만** 합니다 — 그래서 스펙 정합성 책임이 한 곳으로 모입니다.

**Q2. access 토큰과 refresh 토큰을 왜 다르게 설계했나요?**
access는 짧은 수명의 **무상태 JWT**(서버가 상태를 안 들고 검증), refresh는 **DB에 저장된 opaque UUID**입니다. refresh를 DB(`refresh_token`)에 두는 이유는 회전·폐기·세션 감사(IP/UA 기록) 때문입니다. 무상태 JWT의 약점(즉시 무효화 불가)을 짧은 수명 + refresh 폐기로 보완합니다.

**Q3. AI로 프로필을 분석한다는데, LLM이 점수를 멋대로 만들지 않나요?**
아닙니다. **서버가 점수를 계산합니다.** LLM에는 각 기준의 원점수(0~100)와 근거·개선 문장만 요구하고, 직무군별 가중치로 총점을 합산하는 건 서버입니다. 이 뉴로-심볼릭 분리 덕에 재현성과 직무군 공정성을 서버가 강제합니다.

**Q4. 키가 없는데 AI 기능이 어떻게 동작하나요?**
폴백 구조입니다. 키가 없으면 즉시 결정론적 **규칙엔진**으로 동작하고 status는 SUCCESS로 남습니다. LLM 호출이 떠도 실패하면 규칙엔진 결과에 폴백 status를 붙입니다. 운영자가 응답·로그의 status로 폴백 발생을 추적할 수 있습니다. 공통 폴백 사다리(자체 OSS → 상용 → Mock)는 [AI 공급자·폴백](/flow/ai-providers-fallback)에 있습니다.

**Q5. 동의(consent)는 왜 그냥 컬럼 하나로 안 하고 이력 테이블로 만들었나요?**
개인정보보호 요건상 "언제 동의·철회했는지"를 완전히 재구성할 수 있어야 하기 때문입니다. `user_consent`는 현재값을 덮어쓰지 않는 **append-only 이벤트 이력**이고, "현재 동의"는 같은 type의 최신 1행이 동의 상태인지로 판정합니다. AI 실행은 데이터 활용 동의가 켜져 있어야만 통과하는 게이트로 묶여 있습니다.
