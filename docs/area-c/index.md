# 영역 C 개요 — 적합도·전략·대시보드 (AI #12-18)

> 영역 C는 한 **지원 건(Application Case)** 에 대해 "이 공고에 **지원해도 되나** / **무엇을 보완하나** / **다음 어디로** 가야 하나"를 하나의 흐름으로 답하는 취업 전략 영역이다. 설계 정체성은 **뉴로-심볼릭**: 점수·판단은 결정적 규칙엔진이 소유하고, LLM은 설명 텍스트만 만든다.

상단 네비게이션: [영역별 심화 전체 개요](/areas/) · 인접 영역 [영역 B — 지원 건·공고 분석](/area-b/) ← C → [영역 D — 가상 면접](/area-d/) · [전체 흐름](/flow/) · [AI #1-34 맵](/flow/ai-function-map)

## 1. 이 영역이 책임지는 것

영역 C는 지원 건 하나하나에 대해 다음 세 질문을 끊김 없이 이어서 답한다.

| 질문 | 산출물 | 담당 화면 |
| --- | --- | --- |
| **지원해도 되나?** | 적합도 점수 + 지원 판단(APPLY/COMPLEMENT/HOLD) + 신뢰도 | `FitAnalysisPanel` |
| **무엇을 보완하나?** | 부족역량 3단계 + 학습 로드맵 + 자격증 추천 | `LearningRecommendationPanel` |
| **다음 어디로?** | 지원 전략 3단계 액션 + 장기 경향 + 다음 지원방향 | `StrategyPanel`, `/analysis` |

이 영역의 결정적 특징은 **AI가 점수를 정하지 않는다**는 점이다. 점수·판단·신뢰도는 규칙엔진(`MockFitAnalysisAiService`)이 결정적으로 확정하고, LLM은 그 결과를 사람이 읽을 문장으로 풀어쓸 뿐이다. 이 한 줄이 영역 C의 거의 모든 설계 결정을 지배한다.

소유 데이터는 `fit_analysis`(불변·재분석마다 INSERT), `career_analysis_run`(장기 집계 + 캐시), `dashboard`(요약 재투영) 계열이며, 결과는 자기 영역이 소유하고 다른 영역의 원본은 읽기전용으로만 참조한다.

:::tip 이 페이지가 답하는 면접 질문
"영역 C가 정확히 뭘 하나요?" / "전체 6영역 중 C는 어디에 끼어 있나요?" / "왜 AI 점수를 안 믿는 구조로 만들었나요?"
이 개요만 막힘없이 말할 수 있으면, 하위 세부 페이지는 디테일을 채우는 역할이다.
:::

## 2. 6개 영역 속에서 C의 위치

CareerTuner는 6명이 **수직 분담**한 6개 영역(A~F)이고, 각 영역이 `사용자 화면 + REST API + 관리자 + AI 기능 + DB 테이블`을 통째로 소유한다. 핵심 단위는 공고가 아니라 **지원 건**이며, 6영역이 한 지원 건을 함께 채운다.

| 영역 | 한 줄 정체성 | AI # | 대표 산출물 | 상세 |
| --- | --- | --- | --- | --- |
| A | 회원·프로필·인증 (신뢰 데이터 원천) | #1~5 | 프로필 요약·기술스택·완성도 | [/area-a/](/area-a/) |
| B | 지원 건·공고·기업 분석 | #6~11 | 공고 구조화·필수/우대·면접포인트 | [/area-b/](/area-b/) |
| **C** | **적합도·전략·대시보드 (여정의 비교 허브)** | **#12~18** | **점수+근거+다음 행동** | **이 페이지** |
| D | 가상 면접·리포트 | #19~23 | 예상질문·답변평가·리포트 | [/area-d/](/area-d/) |
| E | 첨삭·결제·크레딧 | #24~28 | 원문 보존 개선안·요금 추천 | [/area-e/](/area-e/) |
| F | 커뮤니티·고객센터·챗봇 | #29~34 | 후기요약·실제질문·문의 초안 | [/area-f/](/area-f/) |

### 데이터가 어디서 와서 어디로 가나

C는 다른 영역의 원본을 **읽기전용으로만** 참조하고, 자신이 만든 적합도 결과만 소유한다.

```text
[A user_profile 스냅샷] ─┐
                         ├──▶ [C 적합도 #12] ──▶ fit_analysis ──▶ [D 면접 질문 입력]
[B job_analysis 필수/우대]─┘     (점수·판단·신뢰도)
                                       │
[D interview_* 결과] ─────────────────▶ [C #16 장기 경향] (순환 환류)
[E ai_usage_log / 크레딧] ◀── C의 모든 AI 호출이 사용량·과금을 남김
```

- **들어오는 것(읽기전용 참조):** A의 `user_profile` 스냅샷, B의 `job_analysis`(필수·우대·담당업무). C는 이 둘을 비교하는 **허브**라, A와 B의 출력이 여기서 만난다.
- **나가는 것(C가 소유·제공):** `fit_analysis`(적합도 점수·근거)가 **영역 D**의 면접 질문 입력으로 흐른다.
- **되돌아오는 것(순환):** D의 면접 결과·E의 사용 이력이 C의 #16 장기 경향 집계로 환류한다.
- **공통 기록:** C의 모든 AI 호출은 `ai_usage_log`(E 영역 소유)에 사용량을 남기고 크레딧 정책의 대상이 된다.

자세히: [데이터 소유권 경계 맵](/flow/data-ownership) · [지원 건 중심 흐름](/flow/application-case)

## 3. 담당 AI 기능 — #12 ~ #18

영역 C가 소유한 AI 기능은 7개다. 모두 "규칙엔진이 골격, LLM은 설명" 원칙을 공유한다.

| # | 기능 | 한 줄 설명 | 주요 산출물 |
| --- | --- | --- | --- |
| 12 | 공고-스펙 적합도 | 공고 요구조건 vs 프로필 매칭 점수화 | `fitScore`, `conditionMatrix`, `applyDecision` |
| 13 | 부족역량 추천 | 미충족 역량을 3단계로 분류 | `REQUIRED_MISSING` / `PREFERRED_GAP` / `LONG_TERM_GROWTH` |
| 14 | 학습 로드맵 | 부족역량별 3단계 학습 경로 | `learningRoadmap`, `fit_analysis_learning_task` |
| 15 | 자격증 추천 | catalog 기반, 과도추천 억제 | `certificateRecommendations` |
| 16 | 장기 취업경향 | 결정적 집계 → 추세 요약 | `trendSummary`, `recommendedDirections` |
| 17 | 다음 지원방향 | 경향 기반 추천 지원방향 | `recommendedDirections` |
| 18 | 대시보드 요약 | 핵심 지표 1줄 요약 | `dashboard_insight.summary` |

전체 34개 AI 기능의 의존 관계는 [AI #1-34 맵](/flow/ai-function-map)에서, AI 공급자·폴백 전략은 [AI 공급자·폴백](/flow/ai-providers-fallback)에서 다룬다.

:::warning 면접 핵심 한 줄
"LLM은 점수를 정하지 않습니다. 점수는 규칙엔진이 확정하고, LLM은 그 점수를 사람이 읽을 문장으로 풀어쓸 뿐입니다."
이 문장이 영역 C 설명의 중심 메시지다.
:::

## 5. 구현 상태 — 정직하게

과장은 가장 위험하다. 영역 C의 상태는 다음과 같이 구분한다.

| 구현 완료 (현재 동작) | 향후 과제 (키/모델 발급 후) |
| --- | --- |
| 규칙엔진 점수·판단·신뢰도·가드레일 | 자체 OSS 파인튜닝 모델 학습·서빙 |
| Read-through 캐시·3단 폴백 배선 | OpenAI 실 키 연동 활성화 |
| `fit_analysis` 저장·히스토리·학습과제·조건매트릭스 | (배선·계약은 완성, 키만 꽂으면 동작) |
| 오케스트레이터 SSE·프론트·관리자 화면 | |

:::tip 정직한 한 줄
"아키텍처는 완성돼 있고, 현재는 규칙엔진 기준으로 결정론적 데모가 돌아갑니다. 화면과 계약은 실제 LLM과 동일하며, 실 LLM 연동만 키 발급 후 활성화하면 됩니다."
:::

## 6. 영역 C 단골 면접 질문 5개

1. **6개 영역 중 C는 어디에 위치하나요?**
   A의 프로필 스냅샷과 B의 공고 분석(필수·우대)을 읽어 비교하는 **허브**입니다. 결과인 적합도 점수·근거는 D(가상 면접)의 질문 입력으로 흐르고, D·E의 결과는 다시 C의 장기 경향으로 환류합니다.

2. **왜 AI에게 점수를 안 맡겼나요?**
   재현성·설명가능성·책임·비용·가용성 때문입니다. 점수는 규칙엔진이 결정적으로 계산하고, LLM은 설명만 합니다(뉴로-심볼릭).

3. **AI가 틀린 판단을 내면요?**
   `guardApplyDecision`이 사후 검증합니다. `fitScore<70`이거나 필수 미충족이 있으면 APPLY를 COMPLEMENT로 강등하고 자동보정 사유를 남깁니다(AI의 원래 reasons는 유지).

4. **LLM이 죽으면 서비스가 멈추나요?**
   아니요. 자체 OSS → OpenAI → Mock 3단 폴백이고, 마지막 Mock은 순수 규칙엔진이라 항상 성공합니다. 화면은 깨지지 않습니다.

5. **매번 AI를 부르면 비싸지 않나요?**
   `input_fingerprint`(SHA-256) read-through 캐시로 같은 입력은 저장 결과를 재사용합니다. 초기 로드는 무료, 명시적 재생성만 크레딧을 씁니다.
