# 맞춤 게시글 추천 [#32]

> 로그인 사용자의 희망 직무·스킬과 최근 반응 카테고리로 개인화 후보를 만들고, 신선·인기 후보와 기본 7:3으로 섞는 설명 가능한 피드다. 협업 필터링이라고 부르지 않는다.

## 입력 신호

`PersonalizedFeedService`는 두 종류의 사용자 신호를 사용한다.

1. `user_profile.desired_job`, `skills`
2. 사용자가 최근 반응한 게시글 카테고리 최대 5개

프로필 토큰은 제목·태그·회사명·직무명과 부분 일치시키고 최근 카테고리는 동일 카테고리 후보에 사용한다. 본인 글은 개인화 후보에서 제외한다.

## 두 후보 풀

| 풀 | 생성 방식 |
| --- | --- |
| 개인화 | 프로필 토큰 또는 최근 반응 카테고리 일치 |
| 신선·인기 | 최신순 우선, 좋아요 수 보조 |

SQL은 충분한 후보를 뽑고 Java 서비스가 중복 제거와 인터리브를 담당한다.

## 7:3 인터리브

기본 `personalizedRatio=0.7`이다. 누산 기반 가중 라운드로빈으로 개인화와 신선·인기 풀을 장기적으로 7:3에 가깝게 섞는다.

- post ID `Set`으로 중복 제거
- 한 풀이 먼저 끝나면 다른 풀을 이어 붙임
- 설정 비율이 0~1 범위를 벗어나면 0.7로 보정
- 섞은 목록에서 요청 페이지 구간을 자름

개인화만 100% 제공하지 않아 관심사 고착과 신규 글 노출 부족을 줄인다.

## cold start

비로그인, 프로필 없음, 프로필 토큰과 최근 반응이 모두 없는 사용자는 개인화를 건너뛰고 신선·인기 피드를 반환한다. 빈 화면이나 임의 mock 추천을 만들지 않는다.

## 프런트 연결

커뮤니티 정렬의 `personalized` 옵션이 서버의 7:3 피드를 호출한다. 로그인 사용자에게만 맞춤 정렬을 노출하고, 검색어와 동시에 사용하면 최신 정렬로 폴백해 검색 의도와 개인화가 충돌하지 않게 한다.

## 협업 필터가 아닌 이유

현재는 “나와 비슷한 사용자들이 좋아한 글”을 사용자-사용자 행렬로 계산하지 않는다. 프로필과 본인의 최근 반응 카테고리라는 **콘텐츠·행동 신호 기반 후보화**다.

정확한 표현:

> “프로필과 최근 반응을 사용한 콘텐츠 기반 개인화 후보 70%, 신선·인기 30% 혼합 피드”

| 희망 직무·스킬 신호 | 구현 |
| 최근 반응 카테고리 | 구현 |
| 7:3 혼합·중복 제거 | 구현 |
| cold-start 폴백 | 구현 |
| 커뮤니티 맞춤 정렬 UI | 구현 |

| 임베딩 의미검색 | 챗봇 검색 경로와 별도 |

## 면접 답변

> "개인화 피드는 사용자 프로필의 희망 직무·스킬과 최근 반응 카테고리로 후보를 만들고, 최신·인기 풀과 7:3으로 인터리브합니다. 한 풀이 소진되면 다른 풀로 채우고 ID 중복을 제거합니다. 신호가 없으면 신선·인기로 자연스럽게 폴백합니다. 사용자-사용자 유사도를 계산하지 않으므로 협업 필터라고 과장하지 않고, 설명 가능한 콘텐츠·행동 신호 혼합이라고 표현합니다."
