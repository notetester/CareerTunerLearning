# 관심 기반 게시글 AI 추천 [#32]

> 관심 직무·회사·키워드를 자연어로 던지면 bge-m3 임베딩 코사인으로 의미가 가까운 커뮤니티 글을 찾아준다. "2단계 검색"으로 글 수가 늘어도 코사인 비용을 후보 수에 묶고, FAQ 챗봇의 임베딩 자산을 그대로 재사용한다.

## 1. 한 줄 정의 · 이 페이지가 답하는 면접 질문

영역 F의 AI 기능 #32는 **"관심사(자연어) → 의미가 유사한 커뮤니티 글"** 을 돌려주는 시맨틱 추천이다. 핵심 클래스는 `community/search/CommunityPostSearchService.search(query, category)`이고, 사전 적재는 `CommunityEmbeddingService.embedAllPosts()`가 담당한다.

이 페이지가 답하는 면접 질문:

- "게시글 추천은 어떻게 구현했나요? 벡터 DB를 썼나요?"
- "글이 수만 개로 늘어도 코사인 유사도 계산이 느려지지 않게 어떻게 막았나요?"
- "협업 필터링(읽은 글·북마크 기반 개인화)까지 했나요? 안 했다면 왜, 지금은 어디까지인가요?"
- "민감 정보를 추천 근거로 쓰지 않는다는 설계 원칙은 코드에서 어떻게 지켜지나요?"

:::warning 정직하게 먼저 못 박고 시작
기획 문서의 #32는 "읽은 글·북마크·관심 직무·지원 기업을 섞은 **개인화 추천**(협업 필터 + 콘텐츠 기반 혼합)"을 목표로 한다. **실제 코드는 그 중 콘텐츠 기반 의미검색 한 갈래만 구현돼 있다.** 사용자별 행동 데이터(읽음/북마크/검색 이력)를 추천 근거로 묶는 개인화 레이어는 **미구현(계획)**이다. 이 페이지는 "구현된 것"을 정확히 설명하고 "계획된 것"을 분명히 구분한다.
:::

## 2. 왜 이렇게 설계했나 (의도 · 트레이드오프)

### 2-1. 벡터 DB를 도입하지 않았다

커뮤니티 글이 수십~수백 개 규모인 단계에서 Pinecone·pgvector 같은 벡터 인프라를 새로 세우는 것은 과투자다. 그래서 임베딩을 MySQL의 JSON 컬럼(`community_post_embedding.embedding`)에 넣고, 코사인 유사도는 **앱 안에서 직접 계산**한다(`CosineSimilarity.compute`). 이 결정은 FAQ-RAG 챗봇(영역 F에서 가장 오래된 AI 자산)이 이미 같은 방식이라, **임베딩 클라이언트·코사인 유틸을 그대로 재사용**한다. 새 라이브러리도, 새 운영 컴포넌트도 추가하지 않았다.

### 2-2. "2단계 검색" — 코사인 비용을 후보 수에 묶는다

순진하게 짜면 "전체 PUBLISHED 글을 다 임베딩 로드 → 전부 코사인 → 정렬"이다. 글이 N개면 매 검색이 O(N) 벡터 연산이고, 각 벡터는 1024차원이라 비싸다. 그래서 **1단계에서 SQL로 후보를 좁히고, 2단계에서 후보 안에서만 코사인**을 돌린다.

| | 순진한 방식 | 2단계 검색 |
| --- | --- | --- |
| 코사인 대상 | 전체 글 N | SQL 후보 ≤ `candidateLimit`(300) |
| 비용 증가 | 글 수에 비례 | **후보 상한에 고정** |
| 재현율 | 높음 | 후보 완화 단계로 보호 |

트레이드오프는 명확하다. SQL 후보 단계가 키워드 LIKE라서, 의미는 비슷하지만 표면 단어가 하나도 안 겹치는 글은 1단계에서 탈락할 수 있다(재현율 손실). 이를 막으려고 **후보가 부족하면 필터를 단계적으로 완화**한다(§4).

### 2-3. 민감 정보는 추천 근거로 노출하지 않는다

영역 F 최우선 설계 원칙 중 하나가 "게시글 추천은 행동 데이터를 쓰되 **민감 정보는 추천 근거로 노출하지 않는다**"이다. 현재 구현은 임베딩 소스를 글의 `title + "\n" + content`(공개 본문)로만 한정하고, 결과로는 `PostHit(id, title, url, snippet)`만 돌려준다. 작성자 신원·열람 이력 같은 민감 신호는 애초에 파이프라인에 들어가지 않는다. 개인화를 나중에 붙이더라도 이 경계는 지켜야 한다는 의미다.

## 3. 어떤 기술로 구현했나 (실제 클래스 · 테이블 근거)

| 역할 | 실제 구현 |
| --- | --- |
| 진입 서비스 | `CommunityPostSearchService.search(query, category)` |
| 사전 임베딩 배치 | `CommunityEmbeddingService.embedAllPosts()` |
| 임베딩 모델 | bge-m3 (1024차원), `OllamaEmbeddingClient.embed()` 재사용 |
| 코사인 | `CosineSimilarity.compute(a, b)` (FAQ 챗봇 자산 재사용) |
| 설정 | `CommunitySearchProperties` (`ai.community-search.*`) |
| 1단계 후보 쿼리 | `CommunityPostMapper.findSearchCandidates(keywords, category, limit)` |
| 임베딩 저장 | **별도 테이블** `community_post_embedding(post_id, embedding JSON)` |
| 관리자 트리거 | `CommunityEmbeddingAdminController` (배치 임베딩 수동 실행) |
| 결과 DTO | `PostHit(id, title, url, snippet)`, 후보 DTO `PostCandidate` |

핵심은 `community_post_embedding`이 **별도 테이블**이라는 점이다. 1024차원 벡터를 JSON으로 직렬화하면 행당 ~15KB인데, 이걸 `community_post`에 붙이면 기존의 `SELECT cp.*` 류 목록 조회가 전부 무거워진다. 임베딩은 검색 경로에서만 `JOIN`으로 끌어오고, 일반 조회에는 절대 따라붙지 않게 분리했다(회귀 방지).

:::tip 패키지 위치 주의
검색/임베딩 코드는 기획서가 말한 `community/ai`가 아니라 **`community/search/`** 에 있다. 모더레이션(검열·태깅·추출)은 `community/moderation/`이다. 면접에서 "추천 코드 어디 있나요?"엔 `community/search/CommunityPostSearchService`라고 답한다.
:::

## 4. 동작 원리 (흐름 · 표 · 작은 코드)

### 4-1. 사전 적재 (배치)

`embedAllPosts()`는 `findPostsWithoutEmbedding()`으로 **아직 임베딩이 없는 PUBLISHED 글만** 골라, `title + "\n" + content`를 bge-m3로 임베딩하고 `upsertEmbedding(postId, json)`으로 적재한다. 개별 글 실패는 로그만 남기고 다음 글로 넘어간다(전량 중단 방지). 트리거는 관리자 컨트롤러(수동)다 — AFTER_COMMIT 자동 임베딩 훅은 아직 없다.

### 4-2. 검색 한 턴 (`search(query, category)`)

```text
질의 임베딩          embed(query) → 1024차원 벡터
   │
1단계 SQL 후보       findSearchCandidates(keywords, category, 300)
   │   ├─ 후보 < minCandidates(20) AND 카테고리 있음 → 카테고리 떼고 재조회
   │   └─ 여전히 부족              → 키워드까지 떼고 전체에서 재조회
   │   (후보 0건이면 즉시 빈 결과)
   │
2단계 코사인         후보 임베딩 파싱 → CosineSimilarity → 내림차순 정렬
   │
임계 통과 topK       similarityThreshold(0.5) 이상 상위 topK(3)
   │
재선별 1회(루프 아님) 0건이면 retryThreshold(0.35)로 한 번만 더 → PostHit
```

**1단계 SQL**(`findSearchCandidates`)은 `community_post`와 `community_post_embedding`을 `JOIN`하고, `status='PUBLISHED'` + (선택)`category` + 키워드 `LIKE`(제목/본문/`tags_json`에 OR)로 후보를 좁힌 뒤 최신순 `LIMIT`. 키워드는 질의를 공백 분해해 **2글자 이상 토큰 최대 6개**만 뽑는다.

**점진적 완화**가 재현율 안전장치다. 카테고리 필터로 후보가 20개 미만이면 카테고리를 떼고, 그래도 부족하면 키워드까지 떼고 최신 글 전체를 후보로 삼는다. 즉 "정밀하게 좁히되, 너무 좁아지면 풀어서라도 코사인에 줄 후보를 확보"한다.

```java
// CommunityPostSearchService.search() 발췌 (요지)
List<PostCandidate> candidates =
    postMapper.findSearchCandidates(keywords, normalizedCategory, props.getCandidateLimit());
if (candidates.size() < props.getMinCandidates() && normalizedCategory != null)
    candidates = postMapper.findSearchCandidates(keywords, null, props.getCandidateLimit());
if (candidates.size() < props.getMinCandidates())
    candidates = postMapper.findSearchCandidates(null, null, props.getCandidateLimit());
```

**2단계 코사인**은 후보의 임베딩 JSON을 파싱하고, 길이가 질의 벡터와 다르면 건너뛴 뒤(모델 교체로 차원이 어긋난 잔재 방어) 유사도를 매겨 정렬한다. 임계(`similarityThreshold` 0.5) 이상만 topK(3)로 자른다. **여기서 0건이면 완화 임계(`retryThreshold` 0.35)로 딱 한 번만 재선별**한다 — 점점 낮추는 루프가 아니라 1회 폴백이라, "정밀하게 답하되 빈손보다는 약한 후보라도"를 결정적으로 보장한다.

### 4-3. 누가 이 검색을 호출하나

| 호출처 | 설명 |
| --- | --- |
| 커뮤니티 챗봇 에이전트 | `ai/chat/CommunityTools.searchCommunityPosts(query, category)` → LangChain4j `@Tool`. 사용자가 "OO 직무 면접 후기 추천해줘"라고 하면 모델이 스스로 이 툴을 호출 |
| 통합 오케스트레이터 | 커뮤니티 파트에서 같은 서비스 재사용 |
| 관리자 임베딩 트리거 | `CommunityEmbeddingAdminController`로 배치 임베딩 수동 실행 |

즉 현재 #32의 사용자 진입점은 **챗봇 대화**다. 챗봇이 받은 결과(`PostHit`)는 `SearchTrace`에 기록되고, **실제 툴이 돌려준 글의 URL만** 링크 화이트리스트(`^/community/posts/\d+$`)를 통과해 답변에 붙는다 — 모델이 URL을 지어내지 못하게 막는 환각 방어다.

## 5. 구현 상태 (됨 vs 계획) 정직 구분

| 항목 | 상태 | 메모 |
| --- | --- | --- |
| 자연어 의미검색(콘텐츠 기반) | ✅ 구현됨 | 2단계 SQL+코사인, 점진 완화, 1회 재선별 |
| bge-m3 임베딩 별도 테이블·배치 적재 | ✅ 구현됨 | `community_post_embedding`, `embedAllPosts()` |
| 챗봇 툴/오케스트레이터에서 호출 | ✅ 구현됨 | `searchCommunityPosts` 툴 |
| 자동 임베딩 훅(글 작성 시) | ⚠️ 미구현 | 현재 관리자 배치 트리거(수동) |
| **읽은 글·북마크·검색 이력 개인화** | ⚠️ **계획** | 사용자 행동 신호를 추천 입력으로 묶는 레이어 없음 |
| **협업 필터링(사용자-사용자 유사도)** | ⚠️ **계획** | 구현 없음 — 현재는 글-질의 콘텐츠 유사도만 |
| 관심 직무/지원 기업 자동 주입 | ◐ 부분 | C가 입력 제공 가능(경계), F는 query 문자열로 받을 뿐 자동 결합 미구현 |
| 전용 추천 화면(피드) | ⚠️ 미구현 | 진입은 챗봇 대화 경로뿐, 별도 추천 UI 없음 |

:::warning "협업 필터 + 콘텐츠 기반 혼합"이라고 말하지 말 것
이 페이지의 집중 포인트는 "협업필터+콘텐츠기반 혼합"이지만, **혼합 추천은 코드에 없다.** 면접에서 정직하게: "지금은 콘텐츠 기반 시맨틱 검색까지 구현했고, 행동 데이터를 섞는 협업 필터·개인화는 다음 단계 설계"라고 답해야 한다. 구현 안 된 것을 한 것처럼 말하면 꼬리질문에서 바로 무너진다.
:::

## 6. 면접 답변 3단계

1. **무엇** — "#32는 관심사를 자연어로 받아 의미가 가까운 커뮤니티 글을 찾는 시맨틱 추천입니다. 글을 bge-m3로 임베딩해 두고 질의도 임베딩해서 코사인 유사도로 매칭합니다."
2. **어떻게** — "벡터 DB 없이 임베딩을 MySQL JSON 컬럼에 별도 테이블로 두고 앱 안에서 코사인을 계산합니다. 글 수가 늘어도 비싸지지 않게 1단계 SQL로 후보를 상한(300)까지 좁히고 2단계에서 그 후보 안에서만 코사인을 돌리는 2단계 검색입니다. 후보가 부족하면 필터를 단계적으로 완화해 재현율을 지킵니다."
3. **트레이드오프/정직** — "FAQ 챗봇의 임베딩 자산을 재사용해 인프라를 늘리지 않았습니다. 다만 현재 구현은 콘텐츠 기반까지이고, 읽은 글·북마크 같은 행동 데이터를 섞는 개인화/협업 필터는 다음 단계입니다. 민감 정보는 임베딩 소스에서부터 배제해 추천 근거로 노출되지 않게 했습니다."

## 7. 꼬리질문 + 모범답안

:::details Q1. 왜 벡터 DB를 안 쓰고 MySQL JSON에 임베딩을 넣었나요?
글이 수십~수백 개 규모라 전용 벡터 인프라는 과투자입니다. FAQ-RAG 챗봇이 이미 같은 방식(JSON 임베딩 + 앱 내 코사인)이라 `OllamaEmbeddingClient`·`CosineSimilarity`를 그대로 재사용했습니다. 운영 컴포넌트를 늘리지 않고 같은 패턴으로 확장한 것입니다. 규모가 커지면 pgvector 등으로 옮기는 게 다음 선택지지만, 2단계 검색 구조 덕에 그 전까지는 충분합니다.
:::

:::details Q2. 글이 10만 개가 되면 코사인이 느려지지 않나요?
2단계 검색이 그걸 막습니다. 코사인은 전체 글이 아니라 1단계 SQL이 추린 후보(상한 `candidateLimit` 300)에만 돌립니다. 1단계는 인덱스/LIKE 기반 후보 좁히기라 글 수가 늘어도 코사인 대상은 300으로 고정됩니다. 다만 1단계가 키워드 LIKE라 표면 단어가 안 겹치는 의미 유사 글을 놓칠 수 있는데, 후보가 `minCandidates`(20) 미만이면 카테고리·키워드 필터를 떼며 완화해 재현율을 보호합니다.
:::

:::details Q3. 협업 필터링까지 했나요? 개인화는요?
아니요. 현재는 글-질의 사이의 콘텐츠(의미) 유사도까지입니다. 사용자-사용자 유사도나 읽은 글·북마크·검색 이력을 추천 입력으로 묶는 개인화 레이어는 설계만 있고 구현은 안 됐습니다. 행동 데이터를 붙일 때도 영역 F 원칙상 민감 정보는 추천 근거로 노출하지 않아야 해서, 입력 신호 선별이 그 단계의 핵심 과제입니다.
:::

:::details Q4. 검색 결과가 0건이면 어떻게 되나요?
두 겹의 폴백이 있습니다. ① 1단계 후보가 부족하면 카테고리→키워드 순으로 필터를 떼며 후보를 다시 모읍니다. ② 2단계에서 임계(0.5) 통과 결과가 0이면 완화 임계(0.35)로 **딱 한 번만** 재선별합니다. 점점 낮추는 무한 루프가 아니라 1회 폴백이라, "정밀하게 답하되 빈손은 피한다"를 결정적으로 보장합니다. 그래도 0이면 빈 리스트를 돌려 챗봇이 "관련 글을 못 찾았다"고 답합니다.
:::

:::details Q5. 챗봇이 추천 글 링크를 지어내면 어떻게 막나요?
링크 환각 방어가 다층입니다. 검색 서비스가 돌려준 글만 `SearchTrace`에 기록하고, 답변에 붙는 URL은 `^/community/posts/\d+$` 화이트리스트 정규식을 통과해야 합니다. 시스템 프롬프트도 "URL을 메시지에 직접 넣지 마라"고 지시하고, 링크는 실제 툴 출처에서만 시스템이 붙입니다. 즉 모델이 없는 글 ID를 만들어내도 링크로 노출되지 않습니다.
:::

:::details Q6. 임베딩 모델을 바꾸면 어떻게 되나요? 차원이 안 맞으면?
2단계 코사인 직전에 후보 임베딩의 길이가 질의 벡터와 다르면 그 후보를 건너뜁니다. 모델 교체로 1024차원이 아닌 잔재가 섞여도 코사인이 깨지지 않게 하는 방어입니다. 다만 전체를 새 모델로 재임베딩하려면 `embedAllPosts()`를 다시 돌려야 합니다(현재는 관리자 배치 트리거).
:::

## 8. 직접 말해보기

아래를 막힘없이 설명할 수 있으면 #32를 이해한 것이다.

1. "2단계 검색"이 무엇이고, 1단계와 2단계가 각각 무슨 일을 하는지 (후보 상한·코사인 대상의 차이까지)
2. 임베딩을 왜 `community_post`가 아니라 `community_post_embedding` 별도 테이블에 뒀는지
3. 후보 부족 시 점진 완화와, 결과 0건 시 1회 재선별의 차이 (둘 다 폴백이지만 동작이 다름)
4. 지금 구현이 "콘텐츠 기반 의미검색"까지이고 "개인화/협업 필터"는 계획인 이유와 경계
5. 챗봇이 추천 링크를 환각하지 못하게 하는 화이트리스트 메커니즘
