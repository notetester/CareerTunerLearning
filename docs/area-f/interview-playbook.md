# 영역 F 면접 플레이북

> F의 강점은 커뮤니티·고객센터·챗봇을 운영 가능한 경계로 묶은 것이다. 모델 이름보다 비동기 검열, 운영자 확정, 근거 접지와 데이터 생명주기를 설명한다.

## 1분 요약

> "F는 커뮤니티·고객센터·알림과 FAQ·커뮤니티·인테이크 챗봇을 담당합니다. 글 저장 후 비동기 검열로 응답을 막지 않고, 신고 분류는 AI 소견과 운영자 처분을 분리했습니다. 챗봇 링크는 실제 tool trace에서만 만들고 대화와 인테이크 슬롯을 MySQL에 저장합니다. 추천은 프로필·최근 반응 개인화 70%와 신선·인기 30% 혼합이고, Web Speech 입력·문의 첨부·대화 삭제까지 연결했습니다. #29 전용 후기 요약 카드는 아직 #31 부산물과 런타임 요약으로 대체합니다."

## 핵심 패턴

### 비동기 검열

게시글 트랜잭션이 커밋된 뒤 `AFTER_COMMIT` 이벤트와 전용 executor로 검열한다. 롤백된 글은 검열하지 않고 느린 모델 호출이 작성 응답을 붙잡지 않는다.

### 운영자 확정

자동 검열과 신고 분류는 판정 자산을 재사용하더라도 부수효과가 다르다. 신고 AI는 소견을 저장하고 운영자 처분을 대신 확정하지 않는다.

### 근거 접지

모델이 자유롭게 URL을 만들지 않는다. 검색 tool이 반환한 post ID만 trace에 기록하고 `/community/posts/{id}` 화이트리스트를 통과한 링크만 보여 준다.

### 영속 메모리와 슬롯

`MyBatisChatMemoryStore`가 대화 메시지를, `chatbot_intake_slot`이 검증된 case ID·mode와 상태를 저장한다. 재시작 후 PENDING만 복원하고 READY/DONE은 제외한다.

## 자체 모델 표현

F는 qwen3:8b, Gemma 계열과 BGE-M3 같은 로컬 OSS를 task별로 서빙한다. 이는 자체 호스팅 OSS 통합이다. `careertuner-mod`는 이름과 연결은 있으나 베이스·dataset·adapter·평가 provenance가 충분하지 않아 직접 파인튜닝 성과로 과장하지 않는다.

## 자주 묻는 질문

### 추천은 협업 필터인가?

아니다. 내 프로필 토큰과 최근 반응 카테고리로 콘텐츠 후보를 만들고 신선·인기 풀과 섞는다. 다른 사용자와의 유사도 행렬을 계산하지 않는다.

### 대화와 슬롯이 재시작 후 살아 있나?

예. 대화 JSON과 확정 슬롯 모두 DB에 있다. 단 PENDING만 복원해 완료·이탈 상태의 stale 재진입을 막는다.

### 대화를 삭제하면?

본인 소유를 확인하고 인테이크 슬롯을 먼저 삭제한 뒤 memory를 삭제한다. 지원 건과 이미 만든 산출물은 건드리지 않는다.

### 자동 TTL은?

없다. 명시 삭제는 구현됐지만 자동 만료 정책이 있다고 말하지 않는다.

### 음성 입력은?

F 챗봇은 Web Speech API wrapper다. Chrome/Edge 의존성이 있고 vendor 서버를 경유할 수 있다. D의 faster-whisper와 구분한다.

### 문의 파일 첨부는?

프런트가 공통 file API로 업로드한 후 `attachmentFileIds`를 문의 생성·추가 메시지에 보낸다. 서버는 사용자 소유 파일인지 확인하고 thread 응답에 첨부를 포함한다.

### AI 문의 답변을 바로 보내나?

아니다. 초안은 상담사 화면의 보조 자료이며 고객 응답 확정은 사람이 한다.

## #29의 정확한 경계

전용 `SUMMARY` 저장·후기 카드 파이프라인은 없다. #31이 만든 `overallNote`와 요청 시 챗봇 요약이 일부 가치를 제공한다. 번호 기능을 모두 같은 완성도로 과장하지 않는다.
