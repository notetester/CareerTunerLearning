# 영역 F 면접 플레이북

> 커뮤니티·고객센터·챗봇·알림을 "왜 이렇게 만들었나"로 한 번에 설명하는 종합 정리. 기능 나열이 아니라 **설계 결정과 그 이유**를 1분/3분/꼬리질문 단위로 답할 수 있게 만든다.

## 1. 이 페이지가 답하는 면접 질문

이 페이지는 영역 F 전체를 면접에서 막힘없이 설명하기 위한 종합 정리다. 개별 기능의 깊은 설명은 같은 섹션의 다른 페이지로 분리돼 있고, 여기서는 **다섯 가지 핵심 설계 결정**을 면접 답변 형태로 묶는다.

핵심 질문 다섯 개:

1. F가 한 일을 1분/3분으로 말해보라.
2. 왜 챗봇에 LangChain4j 에이전트를 썼나? 그냥 프롬프트로 안 되나?
3. 인테이크 챗봇은 왜 따로 만들었고, 슬롯을 왜 DB가 아니라 코드로 접지하나?
4. 대화 메모리를 왜 DB에 영속화했나?
5. AI가 글을 자동으로 지우면 안 되는 이유, 즉 "운영자 확정 원칙"이 무엇이고 코드로 어떻게 강제했나?

각 결정은 같은 패턴을 따른다 — **LLM은 비결정적이고 틀릴 수 있으니, 틀려도 시스템이 안전하도록 코드가 검증·차단·복구한다.** 이 한 문장이 F 전체를 관통하는 설계 철학이다.

## 2. F를 1분·3분으로 (엘리베이터 피치)

### 1분 버전

"영역 F는 CareerTuner의 **사용자 간 정보 공유와 운영 커뮤니케이션**을 담당합니다. 커뮤니티 게시판(글·댓글·신고·면접후기), 고객센터(1:1 문의·공지·FAQ), 알림/푸시, 그리고 세 개의 챗봇이 들어갑니다. AI는 여섯 가지 역할을 하는데 — 글 검열, 태그 추천, 면접질문 추출, 게시글 추천, 신고 분류, 문의 답변 초안 — 전부 **로컬 Ollama 자체 LLM**으로 돌립니다. 가장 중요한 원칙은 **AI는 운영자 보조이지 자동 처분자가 아니다**라는 것이고, 이걸 코드 레벨에서 강제했습니다."

### 3분 버전 (구조: 무엇 → 왜 → 어떻게 → 정직)

- **무엇**: 커뮤니티·고객센터·챗봇·알림. AI 기능 6종(#29~#34)과 챗봇 3종(FAQ-RAG, 커뮤니티 에이전트, 인테이크 오케스트레이터), 통합 라우터까지.
- **왜 자체 LLM**: 외부 API 키 미발급 환경이라 OpenAI/Anthropic를 못 쓰는 제약이 있었습니다. 그래서 검열·분류 같은 반복 작업을 로컬 Ollama(`gemma4`·`qwen3:8b`·`bge-m3`)로 구현했고, 이건 "구현 완료 → 파인튜닝 승격"을 노리는 팀 공통 서빙 인프라의 출발점이기도 합니다.
- **어떻게 (핵심 패턴 3개)**:
  1. **이벤트 기반 비동기 검열** — 글이 저장(커밋)된 뒤에 `@TransactionalEventListener(AFTER_COMMIT)` + 전용 스레드풀로 30초짜리 LLM 호출을 돌립니다. 사용자 응답을 막지 않고, 롤백된 글은 검열하지 않습니다.
  2. **운영자 확정 분리** — 자동 검열(즉시 차단)과 신고 분류(운영자 참고만)를 `task_type`으로 분리했습니다. 신고는 AI가 판정만 하고 처분은 사람이 합니다.
  3. **슬롯 접지·링크 화이트리스트** — 챗봇에서 LLM 출력을 그대로 안 믿고, 코드가 검증한 툴 호출 결과·정규식 통과 링크만 사용자에게 보여 환각을 막습니다.
- **정직하게**: #29(후기 요약)는 전용 파이프라인이 없고 대체물로 처리합니다. #32(추천)는 자연어 의미검색까지만 구현됐고 개인화는 미구현, 인테이크 챗봇의 슬롯 DB 영속화는 아직 인메모리입니다.

:::tip 3분 버전의 핵심
면접관이 듣고 싶은 건 "여섯 기능을 만들었다"가 아니라 **"비결정적인 LLM을 운영 시스템에 어떻게 안전하게 끼워 넣었나"**다. 무엇보다 패턴(비동기·운영자 확정·접지)을 먼저 말하라.
:::

## 3. 왜 LangChain4j 에이전트인가 (그냥 프롬프트가 아니라)

면접에서 자주 나오는 질문: "챗봇이면 프롬프트 한 번 호출하면 되는데 왜 에이전트 프레임워크를 끼웠나?"

### 답: 챗봇이 커뮤니티 데이터를 "조회"해야 하기 때문

커뮤니티 챗봇은 "스프링 면접 후기 있어?" 같은 질문에 **실제 DB에 있는 글을 찾아서** 답해야 한다. LLM 혼자서는 DB를 모른다. 그래서 LangChain4j의 `@AiService` + `@Tool` 패턴으로, 모델이 필요할 때 스스로 함수(툴)를 호출하게 했다.

`CommunityChatAgent`는 인터페이스 하나뿐이다:

```java
public interface CommunityChatAgent {
    @SystemMessage(fromResource = "prompts/community-chat-system.txt")
    String chat(@MemoryId Long conversationId, @UserMessage String message);
}
```

`@MemoryId`로 대화별 메모리를 묶고, 구현체는 `AiServices.builder()`가 런타임에 만든다. 모델이 "검색이 필요하다"고 판단하면 `CommunityTools`의 read-only 툴 3종을 호출한다:

| 툴 | 역할 | 기반 |
| --- | --- | --- |
| `searchCommunityPosts` | 관심사 자연어 → 의미 유사 글 | #32 2단계 시맨틱 검색 재사용 |
| `getPostContent` | 글 본문(최대 2000자) → 런타임 요약 소스 | #29 대체물 |
| `searchFaq` | FAQ 검색 | FAQ 임베딩 재사용 |

### 트레이드오프와 방어

에이전트는 강력하지만 **툴 호출이 폭주**할 위험이 있다(로컬 모델이 같은 툴을 반복 호출). 그래서 `maxSequentialToolsInvocations=3`으로 캡을 걸었다. 또 qwen3:8b는 **구조화 POJO 반환을 강제하면 tool_call을 건너뛰고 JSON만 즉시 뱉는 충돌**이 실측됐다. 그래서 에이전트는 `String`을 반환하고, 링크는 모델 JSON이 아니라 실제 툴 출력(`SearchTrace`)에서 접지한다.

자세한 LangChain4j @AiService·@Tool·@MemoryId 동작은 [LangChain4j + Ollama](/ai/langchain4j-ollama)를, 에이전트 페이지는 [LangChain4j 에이전트](/area-f/langchain4j-agent)를 보라.

:::warning "왜 프롬프트가 아니라 에이전트?"의 핵심 답
프롬프트는 **모델 안에 든 지식만** 답한다. 에이전트는 **우리 DB에 있는 실데이터**를 가져온다. 커뮤니티 챗봇의 가치는 후자에 있으므로 에이전트가 필요하다.
:::

## 4. 왜 인테이크 챗봇을 따로, 슬롯을 코드로 접지하나

### 4-1. 인테이크 챗봇이 별도인 이유

CareerTuner의 메인 기능은 "지원 건에 맞춘 면접 준비 자동 생성"(영역 D의 AutoPrep)이다. 그런데 그걸 실행하려면 **어떤 지원 건**으로 **어떤 모드**(기본/직무/인성/압박/이력서/회사)로 돌릴지 정해야 한다. 사용자가 폼을 채우는 대신 **대화로** 이 값을 모으는 게 인테이크 챗봇이다.

핵심 경계: **F는 슬롯 수집까지만, 실행 스트림은 D가 한다.** `IntakeAskService.ask()`가 슬롯을 접지한 뒤 ready/nextAsk 판정은 D의 `AutoPrepIntakeService.intake()`에 위임하고, ready가 되면 클라이언트가 `/api/auto-prep/run/stream`(D)에 직접 연결한다. F가 D의 의존성 그래프를 재구현하지 않는다.

### 4-2. 왜 슬롯을 LLM이 아니라 코드가 확정하나

이게 면접에서 가장 깊게 파고들 만한 결정이다. LLM에게 "이 사용자가 고른 지원 건 ID가 뭐야?"라고 물어 그 답을 그대로 쓰면 **환각된 ID**로 엉뚱한 케이스를 실행할 수 있다.

대신 **슬롯 접지(slot grounding)** 패턴을 쓴다. `IntakeSlotTrace`가 두 가지를 분리해 보관한다:

- **요청 스레드 컨텍스트**(`userId`/`conversationId`): ThreadLocal, 동기 요청 동안만 유효. 툴이 "지금 누구의 대화인지"를 안전하게 읽는다.
- **대화 단위 누적 슬롯**(caseId·mode·originalQuery): `ConcurrentHashMap`에 conversationId로 키링. 여러 턴에 걸쳐 보존돼야 ready 판정이 성립한다.

결정적 불변식: `chooseCase(caseId)` 툴은 **`listCases`가 돌려준 후보 화이트리스트 안에 있는 ID만** confirm한다. LLM이 "37번 케이스로 해줘"라고 해도, 그 사용자의 실제 케이스 목록에 37번이 없으면 코드가 거부한다.

```text
LLM 출력  →  툴 호출(chooseCase(37))  →  코드 검증(화이트리스트?)  →  통과분만 슬롯 확정
                                              └ 실패 시 슬롯에 기록 안 함
```

`IntakeAgentConfig`는 커뮤니티와 의도적으로 분리됐다 — **전용 메모리 윈도우 maxMessages=40**(되묻기로 길어질 수 있어서, 커뮤니티의 공유 20과 별도 빈), 툴캡 3.

자세한 동작은 [인테이크 챗봇](/area-f/intake-chatbot)을 보라.

## 5. 왜 대화 메모리를 DB에 영속화했나

### 답: 새로고침·서버 재시작 후에도 대화가 이어져야 하기 때문

LangChain4j의 기본 `MessageWindowChatMemory`는 **인메모리**라 프로세스가 죽으면 사라진다. 사용자가 챗봇과 5턴 대화하다 새로고침하면 맥락이 날아간다. 그래서 `ChatMemoryStore` 인터페이스를 직접 구현했다 — `MyBatisChatMemoryStore`.

```java
@Override
public void updateMessages(Object memoryId, List<ChatMessage> messages) {
    // 메시지 윈도우 전체를 JSON으로 직렬화해 행 단위로 upsert
    mapper.upsert(toLong(memoryId), ChatMessageSerializer.messagesToJson(messages));
}
```

- `memoryId` = `conversationId`(Long), `chatbot_conversation_memory` 테이블의 PK.
- `messages_json` 컬럼에 윈도우 전체를 JSON으로 저장 → conversationId만 있으면 복원.
- `user_id`로 소유자를 묶어서, 비로그인 익명 대화는 user_id=null로 두고 **복원 대상에서 제외**한다(`findRecentConversation`이 로그인 유저의 최근 대화만 찾음).

### 트레이드오프

매 턴 윈도우 전체를 직렬화·재저장하는 비용이 있다. 하지만 챗봇 대화 윈도우는 20~40 메시지로 작고, 별도 벡터 DB나 세션 스토어 인프라를 새로 띄우는 것보다 **기존 MySQL + MyBatis 한 테이블**로 끝내는 게 운영 단순성에서 이득이다. 직렬화/역직렬화는 LangChain4j가 제공하는 `ChatMessageSerializer`/`Deserializer`를 그대로 쓴다.

자세한 내용은 [챗봇 메모리 영속화](/area-f/chat-memory)를, 직렬화 일반 개념은 [직렬화](/glossary/serialization)를 보라.

## 6. 왜 "운영자 확정" 원칙이고, 코드로 어떻게 강제했나

이게 F의 **최우선 설계 원칙**이다: *"AI는 운영자 보조이지 자동 처분자가 아니다."*

### 6-1. 왜 자동 처분을 막나

- 신고/부적절 분류가 틀리면(거짓 양성) 멀쩡한 사용자가 자동 제재된다.
- 환불·개인정보·정책 답변을 AI가 확정 발송하면 잘못된 약속이 나간다.
- LLM은 비결정적이라 "대부분 맞음"이지 "항상 맞음"이 아니다.

### 6-2. 코드로 어떻게 분리했나 — 같은 두뇌, 다른 부수효과

검열(MODERATION)과 신고(REPORT)는 **같은 `judge()` 판정 함수**를 쓰되 결과를 `task_type`으로 구분하고 **부수효과만 다르게** 했다:

| 경로 | task_type | 부수효과 | 처분 주체 |
| --- | --- | --- | --- |
| 생성 시점 자동 검열 | MODERATION | `toxic && confidence ≥ 0.80`이면 즉시 soft-hide | AI(즉시) |
| 사용자 신고 → 분류 | REPORT | **없음** — 판정 결과만 저장("관리자 참고용") | 운영자 |
| 문의 답변 초안 | — | DB 미영속, 화면에 반환만 | 상담사 |

신고 분류 `classify()`는 자동 숨김도 알림도 없이 `post_ai_result(task_type=REPORT)`에 소견만 남긴다. 실제 처분은 `AdminReportServiceImpl.takeAction()`에서 운영자가 HIDDEN/DELETED/DISMISSED를 확정한다. 여기에 불변식도 있다 — **DELETED는 종착·불가역**(`guardPostNotDeleted`로 역행 차단).

문의 답변 초안도 같은 철학이다. `TicketDraftAiClient.generateDraft()`는 **DB에 저장하지 않고** 매번 즉석 생성해 화면에 반환만 한다. 실제 답변은 상담사가 검토 후 `reply()`로 별도 등록한다 — **초안 ↔ 답변 분리 = 운영자 확정**. 그래서 이 호출은 비동기가 아니라 **동기**다("상담사가 눌렀을 때 즉시 응답").

자세한 분류·제재 동작은 [신고/부적절 분류](/area-f/ai-report-classify)와 [문의 답변 초안](/area-f/ai-support-draft)을 보라.

## 7. 기술 스택과 인프라 결정 한눈에

| 결정 | 무엇 | 왜 |
| --- | --- | --- |
| 자체 LLM | 로컬 Ollama(`gemma4`/`qwen3:8b`/`bge-m3`) | 외부 API 키 미발급. 코드 기본 `localhost:11434`, 원격 GPU로 설정 교체 가능 |
| 비동기 검열 | `@TransactionalEventListener(AFTER_COMMIT)` + `moderationExecutor`(core/max=2) | 30초 LLM이 사용자 응답·DB 커넥션 점유 안 함, 롤백 글 미검열 |
| 트랜잭션 분리 | `PostModerationService`에 `@Transactional` 금지 | 긴 LLM 호출을 트랜잭션에 묶으면 커넥션 풀 고갈 |
| 큐 폐기 정책 | 큐 포화 시 폐기+로그만, `ModerationRetryScheduler`가 회수 | "글은 저장됐는데 사용자는 500" 사고 방지 |
| 임베딩 별도 테이블 | `community_post_embedding`(bge-m3 1024차원) | `SELECT cp.*`에 15KB JSON이 딸려오는 회귀 방지 |
| 2단계 검색 | SQL 후보 좁히기 → 후보 내 코사인 | 글 수 늘어도 코사인 비용이 전체가 아닌 후보에 비례 |
| 라우팅 하이브리드 | 명확구역 argmax(LLM 0회) + 경계구역만 화행분류 | Phase 0 실측: 경계구역(42%)만 LLM, 비용·결정성 양립 |
| 알림 폴링 | Web Push + 30초 visibility-aware 폴링 | SSE 과투자 회피, 탭 숨김 시 중단 |

구조화 출력(format=schema)은 `OllamaClient`에서만 쓰고, 재시도(5xx/타임아웃 한정 지수백오프)도 이 클라이언트만 갖는다. 환각 방어 일반론은 [환각과 방어](/ai/hallucination), 임베딩 개념은 [임베딩](/ai/embedding), 비동기/동기 차이는 [비동기와 동기](/glossary/async-sync)를 보라.

## 10. 꼬리질문 + 모범답안

:::details Q1. 왜 외부 GPT 대신 로컬 Ollama를 썼나? 품질 손해 아닌가?
"외부 API 키 미발급 환경이라 선택의 여지가 일부 있었습니다. 다만 검열·태깅·분류는 입력이 정형화돼 있고 structured output(format=schema)으로 출력을 스키마에 강제하니 작은 모델로도 충분히 동작했습니다. 큰 추론이 필요한 본답변은 qwen3:8b를 쓰되, 환각 방어를 코드로 겹겹이 깔아 품질 격차를 보정했습니다. 또 이건 '구현 완료 → 파인튜닝 승격'을 노리는 팀 공통 서빙 인프라의 출발점이라, 자체 LLM 자체가 자산입니다."
:::

:::details Q2. AFTER_COMMIT 비동기로 검열하면, 검열 전 잠깐 부적절 글이 노출되지 않나?
"맞습니다. 트레이드오프를 의식적으로 받아들였습니다. 30초짜리 LLM을 동기로 묶으면 글 작성 응답이 30초 멈추고 DB 커넥션을 점유해 풀이 고갈됩니다. 그래서 글은 즉시 게시하고 비동기로 검열한 뒤, `toxic && confidence ≥ 0.80`이면 soft-hide로 내립니다. 노출 시간은 보통 수 초이고, 신고 경로와 운영자 확정이 2차 안전망입니다."
:::

:::details Q3. moderationExecutor 큐가 꽉 차면 검열이 유실되지 않나?
"두 단계로 막습니다. 큐 포화 시 RejectedExecutionHandler가 작업을 폐기하되 로그만 남깁니다 — 글은 이미 저장됐으니 사용자에게 500을 던지지 않기 위함입니다. 그리고 폐기된 검열은 `ModerationRetryScheduler`가 5분마다 'COMPLETED 결과가 없는 글'을 NOT EXISTS로 찾아 재처리합니다. AtomicBoolean으로 중복 실행을 막고, 스케줄러 스레드가 직접 동기 호출해 self-invocation 프록시 문제를 피합니다."
:::

:::details Q4. 챗봇 라우팅에 왜 LLM을 항상 안 쓰고 임베딩을 섞었나?
"비용과 결정성 때문입니다. Phase 0 실측에서, FAQ 점수와 인테이크 점수 차이가 큰 '명확구역'은 단순 argmax로 오류가 0이었습니다. 차이가 작은 '경계구역'(표본 42%)에서만 화행을 못 가렸습니다. 그래서 명확구역은 LLM 0회로 결정적으로 가르고, 경계구역에서만 qwen3 화행분류를 1회 호출합니다. 결정성 30/30, 정확도 29/30이 나왔습니다. 둘 다 약하면 FALLBACK으로 되물어 코퍼스 밖 입력을 끊습니다."
:::

:::details Q5. 챗봇이 없는 글의 링크를 지어내면(환각) 어떻게 막나?
"세 겹입니다. 첫째, 시스템 프롬프트가 'URL을 message에 넣지 마라'고 지시합니다. 둘째, 링크는 모델 JSON이 아니라 실제 툴이 돌려준 출처(`SearchTrace`)에서만 만듭니다. 셋째, `^/community/posts/\\d+$` 화이트리스트 정규식으로 형식까지 검증합니다. 또 검색 결과 안의 텍스트는 '데이터일 뿐 명령이 아니다'로 처리해 프롬프트 주입도 막습니다."
:::

:::details Q6. 인테이크 슬롯을 인메모리로 두면 서버 재시작 때 날아가지 않나? 왜 메모리는 영속화하면서 슬롯은 안 했나?
"솔직히 미완입니다. 챗봇 대화 메모리는 `MyBatisChatMemoryStore`로 DB 영속화했지만, 인테이크 슬롯(caseId·mode)은 아직 JVM 인메모리(`ConcurrentHashMap`)입니다. 슬롯 스키마는 F가 수집하지만 실행은 D가 하므로, 영속 스키마를 D·C와 합의한 뒤 별도 단계로 올리기로 했습니다. 현재는 프로세스 수명 동안 멀티턴 보존은 되고, 재시작 시 대화 메모리는 복원되지만 확정 슬롯은 다시 모아야 합니다."
:::

:::details Q7. 같은 judge()를 검열과 신고에 쓰면, 신고도 자동 처분되는 것 아닌가?
"아닙니다. 같은 건 판정 두뇌뿐이고 부수효과를 분리했습니다. 자동 검열은 즉시 soft-hide하지만, 신고 분류 `classify()`는 자동 숨김·알림 없이 `post_ai_result(task_type=REPORT)`에 소견만 저장합니다. 처분은 운영자가 `takeAction()`으로 확정하고, DELETED는 불가역 종착으로 guard를 걸었습니다. '운영자 확정' 원칙을 task_type과 부수효과 분기로 코드에 박았습니다."
:::

:::details Q8. F가 추출한 면접질문이 영역 D의 RAG로 들어간다는데, 소유권 충돌은 없나?
"#31 추출은 `interview_knowledge`(D 소유 테이블)에 `source='CareerTuner 커뮤니티 #{postId}'`로 적재합니다. F가 write하고 D가 참고 데이터로 읽지만, 커뮤니티 원문 소유권은 F입니다. 재적재는 source로 delete 후 재삽입하되, **추출 0건이면 delete를 건너뛰어** 기존 RAG 지식을 보존합니다. 영역 경계는 'F가 입력, D가 RAG 소유'로 명확히 합의돼 있습니다."
:::
