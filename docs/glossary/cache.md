# 캐시 / 캐싱 (Cache)

> 한 번 비싸게 만든 결과를 저장해 두고 다시 쓰는 것. 진짜 어려운 건 "언제 버릴까"(무효화)다.

## 1. 한 줄 정의

**캐시**는 비용이 큰 계산이나 조회의 **결과를 가까운 곳에 저장**해, 같은 요청이 다시 오면 원본을 건드리지 않고 저장본을 돌려주는 기법이다.

## 2. 단어 뜻 (약자·어원)

- **Cache**: 프랑스어 `cacher`(숨기다)에서 온 영어 단어. 원래는 "은닉처, 비밀 저장소"라는 뜻. 컴퓨터에서는 "빠르게 다시 꺼내 쓰려고 미리 숨겨 둔 사본"을 가리킨다.
- 발음은 "캐시"(현금 `cash`와 같다). `caché`(까셰)가 아니다.
- 관련 용어:
  - **Hit(히트)**: 찾는 데이터가 캐시에 있어서 바로 돌려준 경우.
  - **Miss(미스)**: 캐시에 없어서 원본을 다시 계산/조회한 경우.
  - **TTL (Time To Live)**: 캐시 항목의 "유효 수명". 이 시간이 지나면 만료된 것으로 본다.
  - **Invalidation(무효화)**: 더 이상 유효하지 않은 캐시 항목을 버리거나 갱신하는 것.
  - **Read-through(읽기 통과)**: 미스면 캐시 계층이 직접 원본을 읽어 채운 뒤 돌려주는 캐시 패턴.

:::tip
"컴퓨터 과학에서 어려운 건 단 두 가지 — 캐시 무효화, 그리고 이름 짓기." 농담이지만 핵심은 사실이다. 캐시를 **넣는 것**은 쉽고, **언제 버릴지**가 어렵다.
:::

## 3. 왜 필요한가 (없으면 무슨 문제)

같은 결과를 매번 다시 만들면 두 가지 비용이 든다.

| 비용 | 캐시 없을 때 | 캐시 있을 때 |
| --- | --- | --- |
| **시간(지연)** | 매 요청마다 LLM 호출 수 초 대기 | 저장본 즉시 반환(밀리초) |
| **돈(토큰/연산)** | 같은 입력에 매번 토큰 과금 | 입력이 같으면 0원 |
| **부하** | DB/외부 API에 중복 트래픽 | 원본 호출 횟수 급감 |

CareerTuner처럼 **LLM 호출이 곧 돈**인 서비스에서는 캐시가 선택이 아니라 비용 통제 장치다. 사용자가 대시보드를 새로고침할 때마다 AI 요약을 다시 생성하면, 내용은 똑같은데 토큰만 태우고 크레딧만 깎인다.

반대로 캐시를 잘못 쓰면 **낡은 데이터(stale data)**를 보여주는 버그가 생긴다. 그래서 "무엇을 키로 삼아 같음을 판단하는가"가 설계의 핵심이다.

## 4. CareerTuner에서 어디에 썼나 (실제 근거)

내가 담당한 영역 C에는 **AI 요약 결과를 입력 지문 기반으로 캐시**하는 계층이 실제로 구현돼 있다. 시간 만료(TTL)가 아니라 **입력이 바뀌었는지**로 신선도를 판단하는 점이 핵심이다.

| 위치 | 역할 |
| --- | --- |
| `CareerAnalysisRunService.findFreshRun()` | read-through 캐시 코디네이터. 같은 입력 지문의 최신 성공 실행을 재사용 후보로 반환 |
| `CareerAnalysisRunService.fingerprint()` | 입력 핵심 필드 문자열을 SHA-256 hex로 압축한 **캐시 키** 생성 |
| `DashboardServiceImpl.buildSummary()` | 대시보드 AI 요약(`DASHBOARD_SUMMARY`) 캐시 적용·재생성 결정 |
| `AnalysisServiceImpl` | 취업경향 분석(career trend) 요약에 같은 캐시 패턴 재사용 |
| `career_analysis_run` 테이블 | 캐시 저장소. `input_fingerprint`, `result`, `status` 컬럼으로 히트 여부 판단 |
| `frontend/vite.config.ts` | PWA Workbox 정적 캐시. `navigateFallbackDenylist: [/^\/api/]`로 API 응답은 캐시 제외 |

핵심 설계 결정 두 가지를 정직하게 구분한다.

- **결정적 집계는 캐시하지 않는다.** 적합도 평균, 준비도 게이지, 상태별 건수 같은 값은 DB 집계라 비용이 0이므로 매 조회 새로 계산한다. **비싼 LLM 요약만** 캐시한다. (코드 주석: "결정적 집계는 항상 새로 계산하고, 비용이 드는 대시보드 AI 요약만 캐시한다.")
- **캐시 키에서 의미 없는 변동값은 제외한다.** `canonical()` 메서드는 활성 지원 수·평균 적합도·부족 역량처럼 요약 문구를 실제로 좌우하는 필드만 키에 넣고, 크레딧 잔액·이번 달 사용량처럼 요약과 무관한 값은 뺀다. 그래야 불필요한 캐시 미스(재생성)를 막는다.

:::warning 보안: 프론트는 API를 절대 캐시하지 않는다
PWA의 Workbox는 JS/CSS/이미지 같은 **정적 자산**만 캐시한다. `navigateFallbackDenylist: [/^\/api/]` 덕분에 `/api/*` 응답(사용자별 민감 데이터, 토큰 등)은 서비스워커 캐시에 들어가지 않고 항상 백엔드로 직접 나간다. 사용자 데이터를 디스크에 캐시하면 다른 사람에게 노출되거나 로그아웃 후에도 남는 사고가 난다.
:::

[자체 LLM 폴백 전략](/ai/self-llm-strategy)에서는 이 캐시가 **폴백 체인의 첫 단계**(캐시 → 규칙엔진 → OpenAI → Mock) 역할을 하도록 설계돼 있다. 가장 싼 경로(캐시)부터 시도하고, 미스일 때만 비싼 경로로 내려간다.

## 5. 핵심 동작 원리 (표·작은 코드)

### 히트 / 미스 판정 흐름

```text
요청 들어옴
  └─ 결정적 집계는 무조건 새로 계산 (비용 0)
  └─ AI 요약은:
       fingerprint = SHA-256(canonical(입력 핵심 필드))
       findFreshRun(userId, type, fingerprint)
         ├─ 최신 실행의 fingerprint가 같고 status != FAILED  → HIT  (AI 미실행, 0원)
         └─ 없거나 / 입력이 바뀌었거나 / 강제 재생성        → MISS (AI 실행 후 record)
```

실제 분기 코드(요지):

```java
Optional<CareerAnalysisRun> cached = forceRefresh
        ? Optional.empty()                                   // 강제 재생성 = 캐시 우회
        : careerAnalysisRunService.findFreshRun(userId, ANALYSIS_TYPE, fingerprint);

if (cached.isPresent()) {                                    // HIT
    summary = parseSummary(cached.get().getResult());
} else {                                                     // MISS → 실제 AI 실행
    DashboardInsightAiResult insight = dashboardInsightAiService.summarize(command);
    run = careerAnalysisRunService.record(/* ... creditUsed ... */);
}
```

### 무효화(invalidation) 방식 비교

| 방식 | 언제 버리나 | CareerTuner 적용 |
| --- | --- | --- |
| **TTL 만료** | 정해진 시간 경과 시 | 미적용(시간이 아니라 입력으로 판단) |
| **입력 지문(fingerprint)** | 입력 데이터가 달라지면 자동 미스 | **채택** — 지원 건/적합도가 바뀌면 키가 달라져 재생성 |
| **명시적 무효화** | 사용자가 "다시 생성" 누름 | `forceRefresh=true` → 캐시 우회 + 크레딧 1 차감 |

이 설계의 장점: 데이터가 안 바뀌면 며칠이 지나도 캐시를 그대로 써 비용을 아끼고, 데이터가 바뀌면 TTL을 기다릴 필요 없이 **다음 요청에서 즉시** 새 요약이 나온다. "낡은 캐시" 문제를 시간이 아니라 내용으로 해결한 것.

### 두 종류의 캐시 비교

| | 백엔드 AI 요약 캐시 | 프론트 PWA 정적 캐시 |
| --- | --- | --- |
| 무엇을 | LLM 생성 결과(텍스트) | JS/CSS/이미지/폰트 |
| 어디에 | `career_analysis_run` 테이블 | 브라우저 서비스워커(Workbox) |
| 키 | 입력 SHA-256 지문 | URL/파일 경로 |
| 목적 | 토큰 비용·지연 절감 | 오프라인·재방문 속도 |
| 절대 안 넣는 것 | 실패(FAILED) 결과 | `/api/*` 응답(민감 데이터) |

## 6. 면접 답변 3단계 (초간단/기본/꼬리질문)

- **초간단**: "캐시는 비싼 결과를 저장해 두고 같은 입력이면 다시 안 만드는 겁니다. 저는 AI 요약 비용을 줄이려고 썼습니다."
- **기본**: "대시보드와 취업경향 AI 요약은 토큰 비용이 들어서, 입력 핵심 필드를 SHA-256으로 지문화해 캐시 키로 삼았습니다. 같은 지문의 성공 실행이 있으면 AI를 다시 호출하지 않고 저장본을 돌려주고, 입력이 바뀌면 키가 달라져 자동으로 재생성됩니다. 반면 적합도 평균 같은 DB 집계는 비용이 0이라 캐시하지 않고 매번 계산합니다."
- **꼬리질문 대비**: "캐시의 진짜 난제는 무효화인데, 저는 TTL 대신 입력 지문 방식을 택했습니다. 데이터가 안 바뀌면 시간 만료를 기다릴 필요 없이 캐시를 계속 쓰고, 바뀌면 다음 요청에서 바로 갱신돼서 '낡은 데이터'와 '불필요한 재생성'을 동시에 줄였습니다. 사용자가 명시적으로 재생성하면 `forceRefresh`로 캐시를 우회하고 크레딧을 차감합니다."

## 7. 꼬리질문 + 모범답안

:::details Q1. TTL을 안 쓰고 입력 지문으로 한 이유는?
TTL은 "시간이 지나면 무조건 버린다"라서 두 가지 손해가 있습니다. 데이터가 그대로인데도 만료되면 같은 결과를 다시 비싸게 만들고(불필요한 미스), 반대로 데이터가 바뀌었는데 TTL이 안 끝났으면 낡은 요약을 보여줍니다(stale). 저는 요약을 결정하는 입력 필드만 골라 SHA-256 지문을 키로 썼습니다. 입력이 같으면 영원히 히트, 입력이 바뀌면 즉시 미스라서, 비용과 신선도를 둘 다 잡습니다. 다만 외부 모델 버전 갱신처럼 "입력은 같지만 결과를 새로 뽑고 싶은" 경우는 `forceRefresh`나 promptVersion으로 따로 처리합니다.
:::

:::details Q2. 캐시 키에 크레딧 잔액 같은 값을 넣지 않은 이유는?
캐시 키에는 **결과를 실제로 좌우하는 입력만** 넣어야 합니다. 크레딧 잔액이나 이번 달 사용량은 요약 문구를 바꾸지 않는데도 매 조회마다 미세하게 달라집니다. 이걸 키에 넣으면 내용이 똑같은데도 키가 매번 달라져 캐시가 항상 미스가 됩니다 — 캐시를 끈 것과 같죠. 그래서 `canonical()`에서 활성 지원 수·평균 적합도·부족 역량 같은 안정적 핵심 필드만 직렬화해 키를 만들었습니다.
:::

:::details Q3. 실패한 AI 응답도 캐시하나요?
아니요. `findFreshRun`은 `status`가 `FAILED`인 실행은 재사용 후보에서 제외합니다. 실패를 캐시하면 일시적 장애가 지문 수명 동안 굳어버려 사용자가 계속 실패만 보게 됩니다. 성공(또는 Fallback로 채운 유효 결과)만 캐시하고, 실패는 다음 요청에서 다시 시도하게 둡니다. 사용량 로그와 크레딧 차감도 실제 AI가 돌아간 `record` 시점에만 남깁니다.
:::

:::details Q4. 프론트에서 API 응답은 왜 캐시 제외했나요?
PWA 서비스워커 캐시는 디스크에 남고 오프라인에도 보입니다. `/api/*`는 사용자별 적합도, 면접 기록, 토큰 같은 민감 데이터라 디스크 캐시에 남기면 보안 사고입니다 — 공용 기기에서 다른 사람이 보거나, 로그아웃 후에도 잔존할 수 있습니다. 그래서 `navigateFallbackDenylist: [/^\/api/]`로 API는 캐시·SPA 폴백 대상에서 빼고 항상 백엔드로 직접 나가게 했습니다. 정적 자산(JS/CSS/이미지)만 캐시합니다.
:::

:::details Q5. read-through 캐시가 뭔가요?
미스가 났을 때 **호출자가 직접 원본을 읽는 게 아니라 캐시 계층이 알아서 원본을 읽어 채운 뒤 돌려주는** 패턴입니다. CareerTuner에서는 `CareerAnalysisRunService`가 그 코디네이터 역할을 합니다 — 히트면 저장본을, 미스면 AI를 실행해 `record`로 저장하고 그 결과를 반환합니다. 호출 측(대시보드/취업경향)은 "캐시에서 가져왔는지 새로 만들었는지"를 신경 쓰지 않고 결과만 받습니다.
:::

1. 히트와 미스를 한 문장씩으로 정의하고, CareerTuner에서 미스가 나는 세 가지 경우를 말해보라.
2. "캐시 무효화가 어렵다"는 말의 의미를, TTL과 입력 지문 방식의 트레이드오프로 설명해보라.
3. 어떤 데이터는 캐시하고 어떤 데이터는 캐시하지 않았나? 그 기준(비용)을 근거로 설명해보라.
4. 프론트 PWA 캐시에서 `/api`를 제외한 이유를 보안 관점에서 30초 안에 설명해보라.
