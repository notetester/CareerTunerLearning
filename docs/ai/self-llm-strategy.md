# C 영역 자체 LLM 전략

> CareerTuner는 거대한 언어 모델을 처음부터 사전학습하지 않았다. 공개 베이스 모델인 Qwen2.5-3B에 도메인 LoRA를 학습하고, 서버 규칙엔진과 결합해 적합도 설명을 생성한다. 학습·평가·서비스 연결까지 확인했지만 소스 기본 provider는 여전히 OpenAI다.

## 한 문장으로 설명하기

"한정된 데이터와 GPU로 범용 LLM을 새로 만드는 대신 3B 공개 모델을 LoRA로 취업 전략 설명에 적응시키고, 점수와 지원 판단은 서버 규칙이 소유하도록 분리했습니다. 자체 모델은 실제 서비스 경로에 연결돼 있으며 설정으로 켤 수 있지만, 안정적인 배포 기본값은 OpenAI입니다."

## 왜 처음부터 만들지 않았나

LLM 사전학습은 대규모 말뭉치, 다수 GPU, 긴 학습 시간과 안전성 평가가 필요하다. CareerTuner가 해결하려는 문제는 한국어 전체를 새로 학습하는 일이 아니라 이미 구조화된 프로필·공고 근거를 바탕으로 **설명 품질을 도메인에 맞추는 일**이다.

따라서 다음 역할을 분리했다.

| 역할 | 담당 |
| --- | --- |
| 언어·상식 기반 | 공개 Qwen2.5-3B 베이스 모델 |
| CareerTuner식 설명 | LoRA 어댑터 |
| 적합도 점수·지원 판단·근거 집합 | 서버 규칙엔진 |
| 최신 자격·공고 근거 | 외부 데이터 provider와 생성 시점 snapshot |
| 장애 대응 | provider 폴백과 검증 가드 |

## 왜 전체 파인튜닝이 아니라 LoRA인가

전체 가중치를 갱신하면 메모리와 체크포인트 크기, 학습 시간이 모두 커진다. LoRA는 원본 가중치를 고정하고 작은 저랭크 행렬만 학습한다. 이 프로젝트에서는 다음 이점이 결정적이었다.

- 단일 공유 GPU에서도 반복 실험이 가능하다.
- 베이스 모델과 어댑터를 분리해 실험 버전을 비교하기 쉽다.
- 어댑터 교체와 롤백 비용이 낮다.
- 서비스가 필요로 하는 것은 범용 지식의 재학습보다 설명 형식과 어조의 적응이다.

LoRA가 언제나 더 좋다는 뜻은 아니다. 충분한 전용 데이터와 계산 자원이 생기고, 전체 모델 행동을 바꿔야 한다는 실험 근거가 생기면 전체 파인튜닝을 다시 검토할 수 있다.

## 실제 구현 경로

소스 기준 핵심 경로는 다음과 같다.

```text
FitAnalysisServiceImpl
  -> FallbackFitAnalysisAiService (@Primary)
       -> OssFitAnalysisAiService
            -> CareerAnalysisOssClient
                 -> Ollama 호환 API / careertuner-c-career-strategy-3b
       -> OpenAI provider
       -> 규칙·Mock 안전망
```

자체 모델 활성화에는 `CAREERTUNER_ANALYSIS_AI_PROVIDER=oss`와 유효한 OSS base URL이 필요하다. 저장소 기본값은 `openai`이므로 **연결돼 있다**와 **모든 배포에서 기본으로 사용한다**를 구분해야 한다.

자세한 상태와 근거는 [모델 근거 매트릭스](/ai/model-evidence-matrix)에서 한 번 더 대조한다.

## 상용화 제약: 베이스 모델 라이선스

자체 모델을 학습했다고 해서 곧바로 상용 서비스의 기본 경로로 쓸 수 있는 것은 아니다. 파인튜닝의 출발점인 공개 베이스 모델에도 라이선스가 있고, 그 조건이 사용 범위를 정한다.

- CareerTuner C가 적응에 사용한 베이스는 **Qwen2.5-3B**다.
- 이 베이스의 공식 메타데이터상 라이선스는 **연구용(비상업) 조건**이다. 학습·평가·데모 같은 연구 목적 사용은 가능하지만, 그 산출물을 그대로 상용 서비스의 기본값으로 두는 것은 라이선스 범위를 벗어날 수 있다.
- LoRA 어댑터는 우리가 학습했지만, 병합된 최종 모델은 여전히 베이스 가중치를 포함하므로 **베이스의 라이선스 제약을 그대로 물려받는다.**

그래서 상용화 관점에서는 이 문제를 **선결 과제(blocker)**로 다룬다. 해소 경로는 셋 중 하나다.

1. **상용 허가가 있는 베이스로 교체**해 동일한 LoRA 레시피를 재학습한다.
2. **상업 친화 라이선스(예: Apache-2.0 계열)의 베이스**를 골라 처음부터 다시 적응한다.
3. **외부 provider(Claude·OpenAI 등)를 기본 경로로 사용**하고, 자체 모델은 라이선스가 정리된 뒤 승격한다.

현재 배포가 **규칙엔진 + 외부 provider 폴백**을 기본값으로 두는 데에는 안정성·가용성뿐 아니라 이 라이선스 제약도 함께 작용한다. "자체 모델을 학습했다"와 "그 모델을 상용 기본값으로 켠다"는 서로 다른 결정이며, 후자는 라이선스 정리가 전제된다.

> 면접에서 정직하게: "공개 3B 베이스에 LoRA로 도메인 적응을 완료했지만, 그 베이스가 연구용 라이선스라 상용 기본값으로 바로 켤 수는 없습니다. 상용은 상업 친화 베이스 재학습이나 외부 provider 경로로 풀고, 점수·판단은 어느 경로든 규칙엔진이 소유하도록 설계했습니다."

## 뉴로-심볼릭 경계

자체 모델과 외부 모델 모두 최종 점수를 마음대로 정하지 않는다. 서버가 먼저 결정론적으로 다음 값을 계산한다.

- `fitScore`
- `applyDecision`
- 일치 역량과 부족 역량
- 필수 요건 충족 여부

모델은 이 skeleton을 받아 사용자에게 읽기 좋은 설명을 만든다. 응답이 계약을 위반하거나 서버 판단을 바꾸면 후처리 가드가 거절하거나 교정한다. 이 구조 덕분에 provider를 바꿔도 핵심 판단의 의미가 흔들리지 않는다.

## 학습과 검증

`ml/career-strategy-llm/`에는 다음 재현 경로가 있다.

1. 구조화 입력과 목표 설명을 JSONL로 조립한다.
2. 학습·검증 split과 manifest를 고정한다.
3. Qwen2.5-3B에 LoRA/QLoRA 학습을 수행한다.
4. 출력 스키마, 판단 불변식, hard-case를 검증한다.
5. Ollama 호환 인터페이스로 모델을 띄워 Spring provider 경로를 확인한다.
6. 실제 서비스의 저장·가드까지 포함한 E2E baseline을 측정한다.

평가에서 중요한 것은 문장이 자연스러운지만 보는 것이 아니다. 서버가 만든 판단값을 바꾸지 않는지, 근거 없는 자격증이나 역량을 추가하지 않는지, JSON 계약을 지키는지를 함께 본다.

## 모델 크기·디코딩 실험에서 배운 것

자체 모델 트랙에서는 "더 큰 모델이 항상 낫다", "파인튜닝하면 무조건 좋아진다" 같은 통념을 그대로 받아들이지 않고 직접 측정했다. 공개해도 되는 교육용 결론만 정리한다.

### 7B가 3B를 못 이긴 게 아니라, 구분되지 않았다

같은 태스크에서 7B 베이스와 3B를 비교했을 때, 7B가 **능력에서 열등했던 것이 아니라** 우리 평가 규모(소표본·단일 시드)에서 **의미 있는 품질 차이를 만들지 못했다.** 반면 비용은 분명히 컸다 — 대략 지연 1.8배, VRAM 2.1배 수준이다. 그래서 결론은 "7B가 나쁘다"가 아니라 **"차이가 확인되지 않는데 비용만 크므로 3B를 유지한다"**였다.

교훈: 모델 크기 결정은 "큰 게 좋다"가 아니라 **측정 가능한 품질 차이 대비 비용**으로 내려야 한다. 표본 안에서 차이가 드러나지 않으면, 그건 큰 모델을 정당화하지 못한다.

### 파인튜닝은 공짜가 아니다 — 근거 이탈이 늘 수도 있다

같은 프로필·공고 입력으로 근거 일치(grounding)를 측정했더니, 도메인 LoRA를 얹은 모델이 베이스보다 **오히려 근거를 벗어난 서술을 더 많이** 냈다(60케이스에서 베이스 6건 대 LoRA 11건). LoRA는 설명을 더 구체적·전략적으로 만들지만, 그 구체성이 때때로 입력에 없는 사실을 끌어들인다.

교훈: 파인튜닝으로 좋아지는 축과 나빠지는 축이 다를 수 있다. 그래서 **사실 차단은 결정적 게이트가 맡고, LoRA에는 전략 설명의 구체성만 맡긴다.** 이 역할 분리가 [가드레일](/area-c/guardrails)과 [뉴로-심볼릭 설계](/area-c/neuro-symbolic)의 핵심 근거다.

### 구조 강제(GBNF)와 LoRA는 다른 문제를 푼다

출력이 정해진 형식(JSON 스키마 등)을 지키게 하는 방법은 여러 층에 있다.

| 방법 | 무엇을 강제하나 | 한계 |
| --- | --- | --- |
| **GBNF 같은 문법 강제 디코딩** | 디코딩 단계에서 토큰이 문법(스키마)을 벗어나지 못하게 막는다 | 형식은 보장해도 내용의 도메인 적합성·어조는 바꾸지 못한다 |
| **LoRA 파인튜닝** | 가중치에 도메인 어조·설명 습관을 새긴다 | 형식을 100% 보장하지는 않는다 |

둘은 대체재가 아니라 층이 다르다. **형식은 문법 강제로, 어조·도메인성은 LoRA로, 사실·점수는 규칙엔진으로** 나눠 맡기면 각 층이 자기 문제만 책임진다.

### 자체 모델을 평가하는 방법

작은 표본에 단일 판정으로 "좋아졌다"고 말하지 않기 위해 평가 설계 자체를 강화했다.

- **표본 대량화**: 소수 예시가 아니라 케이스 수를 늘려 통계적 노이즈를 줄인다.
- **반복 측정**: 같은 입력을 여러 번 돌려 출력 변동성을 함께 본다.
- **베이스 대비 비교**: 파인튜닝 모델을 항상 같은 조건의 베이스와 나란히 놓는다. "좋다"가 아니라 "무엇 대비 좋다"를 만든다.
- **이중 판정**: 규칙 기반 검사(계약·근거 위반)와 의미 기반 판정(semantic judge)을 함께 써서 한쪽이 놓치는 오류를 교차로 잡는다.

교훈: 자체 모델의 성과는 단일 정확도 숫자가 아니라 **재현 가능한 비교 설계**로 말해야 한다.

## RAG를 왜 함께 검토했나

LoRA는 말하는 방식과 도메인 패턴을 가중치에 적응시키지만, 최신 자격 제도나 변하는 기업 정보를 보장하지 않는다. 반대로 RAG는 최신 근거를 넣을 수 있지만 모델의 출력 습관까지 안정화하지는 않는다. 둘은 대체재가 아니라 서로 다른 문제를 푼다.

C 영역에서는 RAG hard-case 실험까지 수행했지만, 단순 검색 주입은 품질·복잡도 대비 이득이 충분하지 않아 현재 결론을 `KEEP_RAG_DISABLED`로 기록했다. 그래서 현재는 무조건 벡터 검색을 켜지 않고 다음 원칙을 따른다.

- 정형 판단은 서버 규칙으로 고정한다.
- 생성 시점에 필요한 외부 근거만 수집하고 snapshot으로 저장한다.
- RAG는 독립 평가에서 개선이 입증될 때만 다시 활성화한다.

"LoRA+RAG를 지향한다"와 "현재 모든 요청에서 RAG가 켜져 있다"는 다른 주장이다.

## 실패와 폴백

자체 endpoint가 없거나 응답 계약을 지키지 못하면 서비스 전체를 실패시키지 않는다. 다음 provider 또는 규칙 기반 경로가 이어받고, 저장되는 분석에는 어떤 경로가 사용됐는지 추적할 수 있는 모델·상태 정보가 남는다.

다만 폴백은 품질을 동일하게 보장한다는 뜻이 아니다. 운영에서는 provider별 사용량과 실패율, 가드 거절률을 함께 관찰해야 한다.

## 꼬리질문

### 자체 모델이 지금 실제로 동작합니까?

예. 학습 결과를 Ollama 호환 인터페이스로 띄우고 Spring OSS provider에서 호출하는 경로와 회귀 검증이 있다. 다만 endpoint 설정이 없는 일반 배포는 기본값인 OpenAI를 사용한다.

### 왜 3B입니까?

프로젝트가 요구하는 것은 범용 추론 경쟁보다 구조화된 근거를 일관된 한국어 설명으로 바꾸는 일이다. 단일 공유 GPU에서 반복 가능한 크기와 서비스 지연을 우선했고, 판단값을 규칙엔진으로 분리해 작은 모델의 부담을 줄였다.

### 무엇을 검증했습니까?

출력 계약, 판단 불변식, 근거 일치, hard-case, provider 폴백, 실제 저장 경로를 검증한다. 단일 정확도 숫자보다 재현 가능한 manifest와 실패 조건을 함께 남기는 방식이다.

### 왜 자체 모델을 기본값으로 켜지 않았습니까?

학습 완료와 운영 기본 승격은 별도 결정이다. 상시 GPU 가용성, 지연, 장애율, 비용, 외부 모델 대비 품질이 릴리스 기준을 지속적으로 만족해야 기본값을 바꿀 수 있다. 여기에 더해 베이스 모델의 **연구용 라이선스** 제약도 상용 기본 승격을 막는 별도 조건이다.

### 상용 서비스에 자체 모델을 바로 쓸 수 있습니까?

지금 형태로는 아니다. 파인튜닝의 베이스인 Qwen2.5-3B가 공식 메타데이터상 연구용(비상업) 라이선스라, 병합 모델을 상용 기본 경로로 두면 라이선스 범위를 벗어날 수 있다. 상용화하려면 상업 친화 라이선스 베이스로 같은 레시피를 재학습하거나, 외부 provider를 기본 경로로 쓰고 자체 모델은 라이선스 정리 후 승격해야 한다. 이 제약이 현재 규칙엔진 + 외부 provider 기본값의 이유 중 하나다.

### 7B로 키우면 더 낫지 않습니까?

직접 비교했지만 우리 평가 규모에서는 7B가 3B 대비 의미 있는 품질 차이를 보이지 않았고, 지연과 VRAM 비용만 뚜렷하게 늘었다(대략 1.8배·2.1배). "큰 모델이 좋다"가 아니라 "측정된 차이 대비 비용"으로 판단해 3B를 유지했다.
