# 환각(Hallucination)과 그라운딩(Grounding)

> LLM은 모르는 걸 모른다고 하지 않고, 사실이 아닌데 그럴듯한 답을 자신 있게 만들어낸다. 이걸 막는 설계가 그라운딩이다.

## 1. 한 줄 정의

환각은 LLM이 근거 없는 내용을 사실처럼 생성하는 현상이고, 그라운딩은 모델 출력을 검증 가능한 근거(검색된 문서·서버 계산값)에 묶어 환각을 줄이는 기법이다.

## 2. 단어 뜻(약자·어원)

| 용어 | 뜻 |
| --- | --- |
| Hallucination | "환각". 정신의학에서 실제 자극 없이 지각하는 현상. LLM에 비유로 차용 — 입력에 근거가 없는데 내용을 "지각해" 출력한다 |
| Grounding | "접지/근거 부여". 전기에서 회로를 땅에 묶듯, 모델 출력을 외부 근거(ground truth)에 묶는다는 뜻 |
| RAG | Retrieval-Augmented Generation. 검색으로 근거 문서를 가져와 프롬프트에 넣고 생성 |
| Ground truth | 검증된 정답/사실. 모델이 의지해야 할 기준 |

환각은 모델이 "거짓말"을 하는 게 아니라, 다음 토큰을 확률로 예측하는 구조상 **그럴듯함(plausibility)** 을 최적화할 뿐 **사실성(truthfulness)** 을 보장하지 않기 때문에 생긴다.

## 3. 왜 필요(없으면 무슨 문제)

환각 대책이 없으면 다음이 그대로 사용자에게 나간다.

- **없는 사실 생성**: 지원자 프로필에 없는 자격증을 "보유함"으로 판정, 공고에 없는 요구사항을 지어냄.
- **수치 날조**: "합격률 87%" 같은 단정. 근거가 전혀 없어도 숫자는 그럴듯하게 나온다.
- **일관성 붕괴**: 점수는 90점인데 설명은 "역량이 크게 부족"처럼 출력 내부가 모순.
- **신뢰 붕괴**: 취업 의사결정 도구에서 한 번이라도 거짓 판정이 나오면 제품 신뢰가 무너진다.

::: warning 핵심 원칙
**AI 단독 판정 금지.** 점수·합격 가능성 같은 의사결정 수치는 LLM이 최종 확정하면 안 된다. LLM은 설명·문장 생성에만 쓰고, 숫자와 판정은 서버 규칙·검증으로 확정한다.
:::

## 4. CareerTuner에서 어디에 썼나(실제 근거)

CareerTuner는 환각을 4겹으로 막는다. 영역 C(적합도/취업경향)에서 가장 명확하게 구현돼 있다.

**(1) 의사결정 수치는 서버가 확정 — 뉴로-심볼릭(neuro-symbolic)**

`FitAnalysisPromptCatalog.FIT_EXPLAIN_SYSTEM_PROMPT`는 자체 설명 모델(C_FIT_EXPLAIN)에게 명시적으로 못박는다.

```text
적합도 점수(fitScore)와 지원판단(applyDecision), 매칭/부족 역량은
서버 규칙엔진이 이미 계산해 입력으로 주어진다.
너는 점수나 판단을 새로 만들거나 바꾸지 않는다.
입력에 없는 회사명·기술·자격증·수치를 추가하지 않는다.
합격 보장·합격률 단정 같은 표현을 쓰지 않는다.
```

즉 "기호(규칙엔진)가 계산, 신경망(LLM)이 설명"이라는 분리. 점수가 출력에서 사라지지 않게, 서버는 받은 점수 배점도 다시 검증한다 — `FitAnalysisServiceImpl.scoreBreakdown(...)`이 필수/우대/프로젝트/경험/프로필 항목별로 합이 점수와 맞도록 재계산한다.

**(2) 신뢰도는 AI가 아니라 결정적 계산**

`FitAnalysisConfidence.evaluate(command)`는 입력(공고 분석·프로필) 충실도로 신뢰도를 100점에서 감점식으로 산정한다. 코드 주석 그대로:

```java
// 신뢰도는 AI 판단이 아니라 입력 상태 기반의 결정적 계산이라
// mock/실 AI 모두 동일하게 산정된다.
FitAnalysisConfidence confidence = FitAnalysisConfidence.evaluate(command);
```

입력이 비면 "추정에 가깝다"는 사유를 함께 노출해, 근거가 약할 때 사용자에게 솔직히 알린다.

**(3) 면접 평가는 RAG로 근거 주입**

`interview.rag` 패키지: 루브릭·질문은행·기업 정보를 MySQL 원본 + Qdrant 벡터로 보관(`InterviewKnowledgeService`, `QdrantClient`). 평가 시 관련 지식을 검색해 프롬프트에 넣고, 각 문서는 `source`(출처) 필드를 가진다. 자세히는 [RAG와 Qdrant](/ai/rag-qdrant).

**(4) 형식 강제 + 보수적 평가 지시**

`OpenAiResponsesClient`의 구조화 출력으로 응답을 JSON 스키마에 강제한다([구조화 출력](/ai/openai-structured-output)). 적합도 `SYSTEM_PROMPT`는 마지막에 못박는다: "과장하지 말고, 근거가 약하면 보수적으로 평가한다."

| 완화 기법 | CareerTuner 구현 |
| --- | --- |
| 근거 주입(RAG) | `InterviewKnowledgeService` + Qdrant (면접) |
| 서버 확정(AI 단독 금지) | `scoreBreakdown`, `FitAnalysisConfidence` (적합도) |
| 형식 강제 | `OpenAiResponsesClient` 구조화 출력 |
| 환각 억제 지시 | `FIT_EXPLAIN_SYSTEM_PROMPT`, `SYSTEM_PROMPT` |
| 출처 표기 | `InterviewKnowledge.source` |
| 안전 폴백 | `FallbackFitAnalysisAiService`(OSS→OpenAI→Mock) |

::: tip 설계와 계획 구분
RAG(면접)·서버 확정 점수·신뢰도 계산·구조화 출력은 **구현됨**. 자체 LLM 설명 모델(C_FIT_EXPLAIN)과 캐시→규칙엔진→OpenAI→Mock 다단 폴백은 **진행/계획 단계**다.
:::

## 5. 핵심 동작 원리(표·작은 코드)

환각을 막는 핵심은 "LLM에게 무엇을 맡기고 무엇을 뺏느냐"의 경계 설정이다.

```text
[사용자 입력]
     │
     ▼
[규칙엔진]  ← 점수·판정·신뢰도를 결정적으로 계산 (LLM 관여 X)
     │  (확정값을 프롬프트 입력으로 주입)
     ▼
[RAG 검색]  ← 관련 근거 문서를 Qdrant에서 가져와 컨텍스트로 주입
     │
     ▼
[LLM]      ← 설명/문장만 생성. 수치·판정은 "변경 금지" 지시
     │  (JSON 스키마로 형식 강제)
     ▼
[서버 검증] ← 배점 합·일관성 재계산, 깨지면 폴백
     │
     ▼
[사용자에게 출처와 함께 노출]
```

| 맡기는 것(LLM) | 뺏는 것(서버/규칙) |
| --- | --- |
| 자연어 설명·요약 | 적합도 점수, 지원 판단 |
| 강점·위험요인 문장화 | 신뢰도 점수·레벨 |
| 학습 사유 서술 | 점수 항목별 배점 합 |

핵심: **숫자는 코드가, 문장은 모델이.** 모델이 만든 숫자는 믿지 않고, 모델이 만든 문장도 RAG 근거·구조 검증을 통과해야 노출된다.

## 6. 면접 답변 3단계(초간단/기본/꼬리질문)

**초간단**: "환각은 LLM이 근거 없이 그럴듯한 답을 지어내는 현상이고, 저는 그라운딩으로 막았습니다. 점수 같은 수치는 서버 규칙엔진이 확정하고 LLM은 설명만 하게 분리했습니다."

**기본**: "취업 의사결정 도구라 거짓 판정이 치명적이라, AI 단독 판정을 금지했습니다. 적합도 점수·신뢰도·지원 판단은 서버에서 결정적으로 계산하고, LLM에게는 이 값을 입력으로 주면서 '바꾸지 말고 입력에 없는 수치·기술을 추가하지 말라'고 시스템 프롬프트로 못박았습니다. 면접 평가는 루브릭·기업 정보를 Qdrant에 색인해 RAG로 근거를 주입하고, 출력은 JSON 스키마로 형식을 강제했습니다."

**꼬리질문 대비**: "이게 뉴로-심볼릭 구조입니다. 기호적 규칙엔진이 검증 가능한 부분을, 신경망 LLM이 표현이 필요한 부분을 담당해, 환각의 영향 범위를 '설명 문장'으로 한정했습니다."

## 7. 꼬리질문+모범답안(3~5)

::: details Q1. RAG를 써도 환각이 남나요?
남습니다. RAG는 근거를 주입할 뿐 모델이 그 근거를 무시하거나 잘못 종합할 수 있습니다(grounding gap). 그래서 CareerTuner는 RAG에만 의존하지 않고, 의사결정 수치는 아예 LLM에서 빼서 서버가 확정합니다. RAG 검색이 실패해도 `InterviewKnowledgeService.retrieve`는 빈 컨텍스트로 best-effort 동작해 흐름을 끊지 않습니다.
:::

::: details Q2. 점수까지 LLM이 매기면 더 똑똑하지 않나요?
똑똑해 보여도 검증이 안 됩니다. LLM 점수는 재현성이 없고(같은 입력에 다른 점수), 근거와 어긋날 수 있습니다. CareerTuner는 `scoreBreakdown`에서 필수·우대 조건의 MET/PARTIAL/UNMET을 가중 합산해 점수를 결정적으로 산출하므로, 누구든 같은 입력엔 같은 점수가 나오고 항목별로 설명 가능합니다.
:::

::: details Q3. 구조화 출력이 환각과 무슨 상관인가요?
형식 환각을 막습니다. 자유 텍스트면 필드 누락·타입 오류·잡설이 섞이는데, JSON 스키마로 강제하면 `fitScore`는 정수, `matchedSkills`는 배열로 형태가 보장됩니다. 형태가 깨지면 파싱 단계에서 잡아 폴백할 수 있고요. 내용 환각은 못 막지만 출력을 검증 가능한 형태로 만들어 다음 검증 단계를 가능하게 합니다.
:::

::: details Q4. "합격률 87%" 같은 수치 날조는 어떻게 막았나요?
프롬프트에서 직접 금지했습니다. `FIT_EXPLAIN_SYSTEM_PROMPT`에 "합격 보장·합격률 단정 같은 표현을 쓰지 않는다", "입력에 없는 수치를 추가하지 않는다"를 명시합니다. 더 근본적으로는 점수 자체를 서버가 주입한 값만 쓰게 해, 모델이 새 숫자를 만들 동기를 없앴습니다.
:::

::: details Q5. 환각을 0으로 만들 수 있나요?
못 만듭니다. 확률적 생성 모델의 본질이라 완전 제거는 불가능합니다. 그래서 전략이 "제거"가 아니라 "영향 범위 축소와 검증"입니다. 치명적 영역(수치·판정)은 LLM에서 빼고, 남은 영역(설명)은 RAG·구조화·보수적 평가 지시·출처 표기로 위험을 낮춥니다. 신뢰도가 낮으면 `FitAnalysisConfidence`가 사용자에게 "추정에 가깝다"고 솔직히 알립니다.
:::

## 8. 직접 말해보기

- 환각이 왜 "거짓말"이 아니라 모델 구조의 부산물인지 한 문장으로 설명해 보라.
- CareerTuner에서 LLM에게 **주지 않은** 일 3가지와 그 이유를 말해 보라.
- 뉴로-심볼릭이라는 단어를 쓰지 않고, 같은 설계를 신입에게 비유로 설명해 보라.
- RAG가 환각을 "줄이지만 없애지는 못한다"를 면접관에게 설득해 보라.
