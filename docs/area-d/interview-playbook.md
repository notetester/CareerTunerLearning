# 영역 D 면접 플레이북 — 면접에서 막힘없이 설명하는 법

> 영역 D(AI 가상 면접) 전체를 한 페이지로 압축한 답변 대본. 1분/3분 설명, 핵심 기술 선택의 "왜", 그리고 단골 질문 12개+모범답안을 묶었다.

## 1. 이 페이지가 답하는 면접 질문

면접관이 묻는다: **"이 프로젝트에서 가상 면접 파트를 설명해 주세요. 무엇을 만들었고, 어떤 기술 선택을 했으며, 왜 그렇게 했나요?"**

이 페이지는 그 한 방을 위해 존재한다. 영역 D의 다른 페이지들(세션 모델, 질문 생성, 멀티에이전트 평가, 폴백 게이트웨이, 자체 LLM 전략…)이 각 주제를 깊게 판다면, 이 페이지는 **그 모든 걸 "말로 풀어내는 순서"** 로 재배열한다. 핵심 메시지는 하나다:

> CareerTuner의 가상 면접은 단순 GPT 래퍼가 아니라, **(1) 생성과 채점을 분리한 2-축 LLM 추상화**, **(2) 자율 멀티에이전트 평가 루프**, **(3) 상용 모델을 디딤돌 삼아 자체 LLM으로 점진 교체하는 전략**을 갖춘 시스템이다.

세부 흐름·테이블·클래스 근거는 영역 D 개별 페이지에서 다루고, 여기서는 그것들을 **면접 답변 형태**로 묶는다.

## 2. 30초·1분·3분 설명 대본

면접에서는 "얼마나 들려줄까"를 상대가 정해준다. 길이별로 세 벌을 준비한다.

### 2.1 30초 (엘리베이터)

> "지원 건(채용공고+내 스펙) 단위로 예상 질문을 만들고, 답하면 멀티에이전트가 채점·검증하고, 끝나면 리포트로 성장 추적까지 하는 AI 모의면접입니다. 핵심 설계는 LLM 호출을 **생성용**과 **채점용**으로 분리하고, 각각 **자체 모델 → Claude → OpenAI** 폴백 체인을 둔 것이고, 지금은 상용 모델로 안정성을 확보하면서 자체 LLM으로 점진 교체 중입니다."

### 2.2 1분 (구조 한 바퀴)

다섯 박자로 끊어 말한다.

1. **단위:** 모든 면접은 공고가 아니라 *지원 건*(`application_case`)에 종속됩니다. 한 회사·직무에 맞춘 맥락이 세션마다 따라옵니다.
2. **흐름:** 세션 생성 → 예상 질문 6개 생성 → 답변 제출·채점 → 리포트. 압박 모드는 답변마다 반박형 꼬리질문이 한 번 붙습니다.
3. **채점:** 단일 LLM 한 방이 아니라 자율 루프입니다. RAG 근거 검색 → 채점 → Critic(적대적 검증·점수 조정) → 필요 시 재평가 → 약한 답변이면 추가 탐색 권장. 각 단계는 `interview_agent_step`에 trace로 남겨 "AI가 무슨 판단을 했는지"를 사용자에게 보여줍니다.
4. **모델 전략:** 생성과 채점은 폴백 정책·모델 티어·과금이 달라 **의도적으로 두 경로로 분리**했습니다. 생성은 자체→Claude→OpenAI, 채점은 자체→OpenAI Provider 분기입니다.
5. **차별점:** 음성·아바타 면접의 원본 미디어를 저장하지 않고 점수/트랜스크립트만 보관하는 프라이버시 설계와, 평가 데이터를 모아 자체 채점 모델을 학습하는 파이프라인입니다.

### 2.3 3분 (구조 + 의도 + 정직한 현 상태)

1분 대본에 다음을 덧붙인다.

- **왜 폴백 체인인가:** 면접 도메인은 자체 LLM 연구가 가장 활발한 곳입니다. 학습이 끝난 task부터 자체 모델로 갈아끼우려면, 그 사이 빈 자리를 메울 안정적 "선생 겸 과도기 런타임"이 필요합니다. 그게 Claude Haiku입니다. 학습이 충분한 task부터 화이트리스트(`OSS_GENERATION_TASKS`)에 올려 자체 모델로 점진 교체하고, 전 task가 덮이면 Claude 게이트웨이를 폐기하는 게 설계 의도입니다.
- **정직한 현재 상태:** 지금 그 화이트리스트는 **빈 집합**입니다. 질문 생성용 학습 데이터가 seed당 1개로 부족해 형식이 불안정(질문 대신 환각)해서, 생성은 전부 Claude→OpenAI 폴백으로 돌리고 있습니다. 채점은 데이터가 많아 안정적이라 자체 모델 코드는 완성됐지만 기본값은 OpenAI입니다. **"코드는 완비, 기본 비활성"** 상태를 솔직히 말하는 게 신뢰를 줍니다.
- **신뢰성 장치:** 단일 채점의 편향을 줄이려고 Critic이 점수를 적대적으로 검증하고, Critic과 원점수가 크게 어긋나면(차이 ≥20) 재채점해 **중간값**을 최종으로 택합니다. 그리고 사용자에게 보여주는 모범답안 = 채점 만점 기준 = 블라인드 복습 채점 기준을 **항상 하나로 고정**해 점수가 일관되게 했습니다.

## 3. 기술 선택의 "왜" — 네 가지 핵심 결정

면접관이 가장 좋아하는 질문은 "왜 그렇게 했어요?"다. 영역 D의 네 가지 결정을 트레이드오프와 함께 외운다.

| 결정 | 무엇을 | 왜 (트레이드오프) |
| --- | --- | --- |
| ① 폴백 게이트웨이 | 자체→Claude→OpenAI 체인 (`FallbackInterviewLlmGateway`) | 단일 provider는 장애·한도·정책 변화에 취약. 체인으로 가용성↑. 대가는 복잡도와 "어느 모델이 답했는지" 추적 필요 |
| ② Claude Haiku를 1차 폴백 기본 | 생성 task의 1순위 외부 모델 | 자체 모델 교체기까지의 *디딤돌*. 빠르고 저렴하며 한국어 품질 충분. GPT-class 전부 쓰면 과금↑·속도↓ |
| ③ SSE 스트리밍 (autoprep) | 자동 준비 파이프라인 진행 상황 푸시 | 면접 준비는 여러 단계라 한 방 응답은 체감이 나쁨. 단계별 푸시로 UX↑. 대가는 인증 헤더 제약(아래 5.2) |
| ④ 자체 면접 모델 학습 | 평가 데이터 → JSONL → 파인튜닝 | 채점은 호출량이 많아 비용이 큼 + 도메인 특화 채점은 자체 모델이 유리. 학습 데이터 확보가 선결 과제 |

### 3.1 왜 폴백 게이트웨이인가 (상세)

핵심은 **"생성과 채점을 한 게이트웨이에 섞지 않는다"** 는 것이다. 생성(질문·모범답안·꼬리·리포트)은 `InterviewLlmGateway`의 폴백 체인을 타고, 채점(평가·Critic)은 `InterviewEvaluatorProvider`가 별도로 OpenAI vs 자체 모델을 분기한다.

코드 주석이 그 이유를 명시한다(`FallbackInterviewLlmGateway`):

> "채점(EVAL)·Critic은 이 게이트웨이가 아니라 `InterviewEvaluatorProvider`가 `OssAnswerEvaluator`로 분기하므로 화이트리스트에 넣지 않는다(**이중 경로 방지**)."

만약 둘을 한 체인에 넣으면, 채점 폴백 정책이 생성 폴백과 얽혀 의도치 않게 달라진다. 생성과 채점은 모델 티어(생성 `gpt-5.4-mini` / 채점 `gpt-5.4`), 과금, 자체 모델 교체 진척이 전부 다르므로 **분리가 정답**이다.

```text
[생성 경로]  InterviewOpenAiClient → FallbackInterviewLlmGateway
             └ OSS(학습된 task만) → Claude Haiku → OpenAI

[채점 경로]  InterviewAgentOrchestrator → InterviewEvaluatorProvider
             └ provider=oss & 빈 존재? OssAnswerEvaluator : OpenAI
```

### 3.2 왜 Claude Haiku가 기본 1차 폴백인가

세 가지를 묶어 말한다. **(1) 역할:** 자체 모델로 갈아끼우기 위한 "선생 + 과도기 런타임". 자체 모델이 아직 못 하는 생성 task를 안정적으로 메운다. **(2) 비용·속도:** 생성은 빠르고 저렴해야 하므로 Haiku급이 적합. 채점만 한 단계 위 모델로 공정성을 챙긴다. **(3) 현실:** 문서 근거상 "Claude Haiku 폴백이 실재하는 곳은 면접(D) 도메인뿐"이다. 다른 영역은 OpenAI로 직행한다.

:::tip 면접 팁
"왜 GPT만 안 쓰고 Claude도?"라고 물으면 — *공급자 다변화로 단일 벤더 장애·한도 리스크를 줄이고, 생성용 저비용 모델(Haiku)과 채점용 정밀 모델(GPT-class)을 티어로 나눠 비용·품질을 동시에 잡았다*고 답한다.
:::

### 3.3 왜 SSE인가

자동 준비(autoprep) 파이프라인은 "공고 분석 → 적합도 → 면접 준비…" 같은 다단계 작업이다. 전부 끝날 때까지 빈 화면을 보여주면 UX가 나쁘다. 그래서 백엔드가 `plan / part-start / substep / part-done / done / error` 이벤트를 **SSE로 푸시**하고, 프런트 reducer(`useAutoPrepRun`)가 단계별 진행을 그린다.

주의: 이 SSE는 **autoprep 오케스트레이터의 것**이지 면접 화면 자체의 것이 아니다. 면접 화면(질문 생성·채점)은 일반 REST + **시간 기반 가짜 진행바**를 쓴다. LLM 호출은 실제 진행률을 알 수 없는 블랙박스라, `90*(1-exp(-elapsed/(estMs*0.6)))` 점근 곡선으로 체감을 개선한다.

### 3.4 왜 자체 면접(채점) 모델인가

채점은 면접에서 가장 호출이 잦은 task다. 호출당 외부 API 비용이 누적되고, 면접 채점은 도메인 특화가 강해 자체 모델이 유리하다. 그래서 **평가가 일어날 때마다 `interview_training_sample`에 학습 데이터를 쌓고**(best-effort), 그걸 JSONL로 export해 파인튜닝하고, LLM-as-judge 하니스로 채점 일관성을 정량 측정한다. 채점 데이터는 풍부해 자체 채점 모델 코드는 이미 완성됐고, base-url만 설정하면 `OssAnswerEvaluator`로 분기된다.

## 4. 멀티에이전트 평가 루프 — 한 장으로 설명하기

채점을 "LLM 한 번 호출"로 답하면 인상이 약하다. **자율 루프**임을 보여준다.

```text
submitAnswer
  └ InterviewAgentOrchestrator.evaluateAnswer (maxTurns=6)
      while: policy.decideNext(ctx) ──┐
        RETRIEVE   RAG 근거 (best-effort, 없으면 skip)
        EVALUATE   채점 → finalScore
        CRITIC     적대적 검증 → adjustedScore
        REEVALUATE |원점수-Critic| ≥ 20 → 재채점, 최종 = (재평가+Critic) 중간값
        PROBE      답변 < 40자 또는 점수 < 50 → 꼬리질문 권장 플래그
        FINISH     더 할 일 없음 → 종료
      각 단계 → interview_agent_step INSERT (agent/action/status/detail/elapsed_ms)
```

| 메커니즘 | 코드 근거 | 의도 |
| --- | --- | --- |
| 2정책 (Rule/Llm) | `RulePolicy` 기본 / `LlmPolicy`(`planner=llm`) | 운영은 결정적 규칙, 시연은 LLM이 매 턴 액션 선택. LLM 실패 시 규칙 폴백, 선택지 1개면 LLM 생략 |
| Critic 중간값 | `REEVAL_THRESHOLD=20`, `reconciled=(re+critic)/2` | 단일 채점의 편향을 적대적 검증+재평가로 완화, 한쪽으로 튀지 않게 |
| trace 투명화 | `interview_agent_step` | 사용자/관리자에게 "AI 판단 과정" 노출 → 채점 신뢰 |
| best-effort 격리 | RAG·training·media | 실패해도 메인 채점 흐름을 끊지 않음 |

:::details 모범답안 = 채점 기준 = 블라인드 복습 기준을 왜 하나로 고정했나
사용자가 보는 모범답안, 채점이 쓰는 만점 기준, 블라인드 복습의 채점 기준이 서로 다르면 점수가 들쭉날쭉해 신뢰가 깨진다. 그래서 질문 INSERT 직후 **트랜잭션 커밋 이후**(`afterCommit`) 백그라운드로 6개 모범답안을 일괄 생성하고, `model_answer`는 비어 있을 때만 갱신하는 **first-writer-wins**(XML 조건부 UPDATE)로 단 하나를 고정한다. 채점 프롬프트는 이 모범답안이 있으면 "그것을 만점(100점) 기준 답안지로 삼고 표현 차이만으로 깎지 않는다"고 지시한다.
:::

## 5. 정직하게 구분하기 — 됨 vs 계획

면접에서 가장 신뢰를 주는 건 "이건 됩니다, 저건 계획입니다"를 정확히 가르는 것이다.

**구현되어 동작함:**
- 예상 질문/꼬리질문 생성, 텍스트·음성·아바타 면접, 멀티에이전트 평가 루프 전체, Critic·재평가 중간값, Rule/Llm 2정책, 폴백 게이트웨이 3종, 평가기 Provider 분기, 모범답안 백그라운드 생성·first-writer-wins, 리포트 캐시·이전 점수 비교, RAG(코드 완비, 기본 off), 학습 파이프라인 4기능, 온디바이스 음성/영상 분석, 관리자 모니터링·프롬프트 뷰, autoprep 통합.

**계획/진행 중 (정직히):**

| 항목 | 현재 | 이유 |
| --- | --- | --- |
| 자체 LLM **생성** | 미가동, 전부 Claude→OpenAI 폴백 | `OSS_GENERATION_TASKS=Set.of()`(빈 집합). QGEN 학습 데이터 seed당 1개로 부족 |
| 자체 LLM **채점** | 코드 완비, 기본값 OpenAI | base-url 미설정 시 폴백. 실제 서빙 전까지 폴백 |
| 질문 생성 입력 | A 프로필 버전 + B 공고·기업 분석 + C 최신 적합도 | 질문 시점 원천 ID·C 핵심 결과를 세션 snapshot으로 고정. C가 없으면 A/B로 계속 |
| 꼬리질문 | 압박 모드 전용 | 모든 모드에 붙이면 길어지고 자체 PROBE 학습이 분산 (의도적 제한) |
| AI 첨삭(개선답변) | 사실상 OSS 평가기에서만 채워짐 | OpenAI strict json_schema 경로는 `improvedAnswer` 미required → 빈 문자열 |
| agent trace | 저장된 step을 클라이언트가 순차 재생 | 실시간 SSE 스트리밍은 후속(인증 헤더 제약으로 fetch-stream 필요) |

### 5.1 흔한 함정: "GPT 래퍼 아닌가요?"

함정 질문이다. 답: *"호출 자체는 LLM API지만, 차별점은 그 위의 오케스트레이션입니다 — 생성/채점 2-축 분리, 폴백 체인, Critic·재평가로 채점 편향을 줄이는 자율 루프, trace 투명화, 그리고 평가 데이터로 자체 채점 모델을 학습하는 파이프라인까지. 단순 래퍼였다면 자체 LLM 전략과 멀티에이전트 trace 테이블이 필요 없었을 겁니다."*

### 5.2 흔한 함정: "SSE에 왜 토큰을 헤더로 못 넣나요?"

브라우저 `EventSource`는 커스텀 헤더(Authorization)를 설정할 수 없다. 그래서 agent trace 실시간 스트리밍은 헤더를 실을 수 있는 fetch-stream 방식으로 가야 하고, 현재는 저장된 step을 클라이언트가 순차 재생하는 방식으로 대체했다(로드맵 후속). 이걸 알고 있으면 "한계까지 파악했다"는 인상을 준다.

## 6. 면접 답변 3단계 공식

어떤 D 질문이든 이 틀에 끼워 넣는다.

1. **정의(무엇):** "X는 …하는 기능/설계입니다." — 한 문장으로 끊는다.
2. **의도(왜):** "단일 방식은 …한 문제가 있어 …하게 분리/추상화했습니다." — 트레이드오프를 한 번 언급한다.
3. **근거(어떻게):** "실제로는 `클래스/테이블`에서 …합니다." — 클래스명·테이블명을 하나만 정확히 댄다.

예) "꼬리질문이요?" → ① 압박 면접에서 답변 약점을 한 번 깊게 파고드는 반박형 후속 질문입니다. ② 모든 모드에 붙이면 면접이 길어지고 자체 PROBE 학습이 분산돼 압박에 집중했습니다. ③ `generateFollowUps`가 압박 외 모드면 `INVALID_INPUT`을 던지고, 답변 선행을 강제하며, `parent_question_id`로 본질문 바로 밑에 인접 정렬합니다.

## 7. 단골 질문 12선 + 모범답안

:::details Q1. 면접 파트를 한 문장으로 설명하면?
"지원 건 단위로 예상 질문을 만들고, 답변을 멀티에이전트가 채점·검증하며, 리포트로 성장을 추적하는 AI 모의면접입니다. LLM 호출을 생성용/채점용으로 분리하고 각각 폴백 체인을 둔 게 핵심 설계입니다."
:::

:::details Q2. 왜 공고가 아니라 '지원 건'이 단위인가요?
"같은 공고라도 지원자마다 강조할 스펙·경험이 다르기 때문입니다. 지원 건(`application_case`)은 공고와 내 프로필을 묶은 단위라, 면접 질문·채점 기준을 그 맥락에 맞출 수 있습니다. 모든 `interview_session`이 `application_case_id`에 FK로 종속되고, 지원 건이 삭제되면 세션도 ON DELETE CASCADE로 함께 정리됩니다."
:::

:::details Q3. 채점은 LLM 한 번 호출인가요?
"아니요. 자율 루프입니다. `InterviewAgentOrchestrator`가 RAG 근거 검색 → 채점 → Critic 적대적 검증 → (이견 크면) 재채점 → (답변 약하면) 추가 탐색 권장 순으로 최대 6턴 돕니다. Critic이 점수를 조정하고, 원점수와 20점 이상 어긋나면 재평가해 중간값을 최종으로 택합니다. 단일 채점의 편향을 줄이는 게 목적입니다."
:::

:::details Q4. 폴백 체인을 설명해 주세요.
"생성 task는 `FallbackInterviewLlmGateway`가 자체 모델(학습된 task만) → Claude Haiku → OpenAI 순으로 폴백합니다. 각 단계에서 `BusinessException`을 잡아 다음으로 넘기고, provider가 하나도 없으면 명확한 예외를 던집니다. 채점은 이 게이트웨이를 안 타고 `InterviewEvaluatorProvider`가 OpenAI vs 자체 모델을 분기합니다 — 이중 경로를 막으려고 의도적으로 분리했습니다."
:::

:::details Q5. 왜 생성과 채점을 다른 경로로 분리했나요?
"폴백 정책·모델 티어·과금·자체 모델 교체 진척이 전부 다르기 때문입니다. 생성은 빠르고 저렴한 `gpt-5.4-mini`, 채점은 공정성을 위해 한 단계 위 `gpt-5.4`를 씁니다. 둘을 한 게이트웨이에 섞으면 채점 폴백이 생성 정책에 얽혀 의도치 않게 달라집니다. 코드 주석에도 '이중 경로 방지'라고 명시돼 있습니다."
:::

:::details Q6. 자체 LLM은 지금 실제로 돌아가나요? (정직성 검증 질문)
"생성은 아직 안 돕니다. 화이트리스트(`OSS_GENERATION_TASKS`)가 빈 집합인데, 질문 생성 학습 데이터가 seed당 1개로 부족해 형식이 불안정(질문 대신 환각)했기 때문입니다. 그래서 생성은 전부 Claude→OpenAI 폴백입니다. 채점은 데이터가 많아 자체 모델 코드가 완성됐지만 기본값은 OpenAI고, base-url을 설정하면 자체 평가기로 분기됩니다. 데이터 보강·재학습 후 task별로 점진 교체하고, 전부 덮이면 Claude를 폐기하는 게 계획입니다."
:::

:::details Q7. Claude Haiku를 기본으로 둔 이유는?
"자체 모델로 갈아끼우기 위한 '디딤돌'이자 '선생'입니다. 자체 모델이 아직 못 하는 생성 task를 안정적으로 메우고, 학습 데이터의 정답 라벨 출처 역할도 합니다. 생성은 빠르고 저렴해야 해서 Haiku급이 적합하고, 채점만 정밀 모델로 공정성을 챙깁니다. 이 프로젝트에서 Claude Haiku 폴백이 실재하는 도메인은 면접(D)뿐입니다."
:::

:::details Q8. 구조화 출력(JSON)은 어떻게 보장하나요?
"같은 JSON 스키마를 provider별로 다르게 처리합니다. OpenAI는 네이티브 `json_schema` strict 모드를, Anthropic은 JSON 모드가 없어 스키마를 프롬프트에 임베드하고 코드펜스를 제거해 파싱하며, 소형 OSS 모델은 `response_format:json_object`에 더해 첫 중괄호부터 마지막 중괄호까지만 추출하는 방어 로직(`extractJsonSpan`)을 둡니다. 점수는 항상 0~100으로 clamp합니다."
:::

:::details Q9. 모범답안과 채점 기준이 어긋나면 어떻게 하나요?
"어긋나지 않도록 하나로 고정합니다. 질문 생성 직후 트랜잭션 커밋 이후(`afterCommit`) 백그라운드로 모범답안을 일괄 생성하고, `model_answer`가 비어 있을 때만 갱신하는 first-writer-wins로 단 하나를 확정합니다. 이 모범답안이 화면 표시, 채점 만점 기준, 블라인드 복습 채점 기준에 똑같이 쓰입니다. 채점 프롬프트도 '기준 모범답안이 있으면 표현 차이로 깎지 말라'고 지시합니다."
:::

:::details Q10. 음성·영상 면접의 프라이버시는 어떻게 처리하나요?
"원본 음성·영상을 저장하지 않습니다(ADR-002 등). 온디바이스 또는 서버 분석으로 점수·트랜스크립트·지표 JSON만 `interview_media_analysis`에 보관하고 원본은 점수 산출 후 폐기합니다. 외부 API 키(OpenAI Realtime, HeyGen)도 서버가 보유하고 단기 토큰만 프런트에 내려 키 노출을 막습니다. `capabilities` 엔드포인트로 키 보유 여부만 알려 프런트가 해당 기능을 사전 비활성화합니다."
:::

:::details Q11. SSE는 어디에 쓰고, 한계는 무엇인가요?
"자동 준비(autoprep) 파이프라인의 단계별 진행 푸시에 씁니다. `plan/part-start/substep/part-done/done/error` 이벤트를 보내 프런트 reducer가 진행을 그립니다. 면접 채점 화면 자체는 일반 REST + 시간 기반 진행바입니다. 한계는 브라우저 `EventSource`가 Authorization 헤더를 못 실어, agent trace 실시간 스트리밍은 fetch-stream으로 가야 하고 지금은 저장된 step을 클라이언트가 순차 재생하는 방식입니다."
:::

:::details Q12. 자체 채점 모델은 어떻게 학습·검증하나요?
"평가가 일어날 때마다 `interview_training_sample`에 학습 데이터를 best-effort로 쌓습니다. 관리자 기능으로 이를 OpenAI 챗 파인튜닝 JSONL로 export하고(최소 10샘플), LLM-as-judge 하니스로 저장 점수와 독립 채점 점수의 일치도(차이 ≤10이면 일치)를 측정해 채점 품질을 정량화합니다. 채점 모델이 안정되면 평가기 Provider를 oss로 토글해 무중단 교체합니다."
:::

:::details Q13. (보너스) 꼬리질문이 왜 압박 모드 전용인가요?
"모든 모드에 꼬리질문을 붙이면 면접이 길어지고, 자체 LLM의 PROBE(탐색) 학습이 여러 모드로 분산됩니다. 그래서 압박 모드에 집중해 '약점을 한 번만 깊게 찌르고 멈추는' 반박형으로 한정했습니다. 다른 모드에서 꼬리질문을 호출하면 `INVALID_INPUT` 예외를 던집니다. 인신공격이 아니라 답변 내용에 대한 압박이라는 톤도 프롬프트에 박혀 있습니다."
:::

:::details Q14. (보너스) 세션은 어떻게 종료되나요?
"별도 '면접 종료' API가 없습니다. 리포트 생성(`getReport`)이 세션을 닫습니다 — `updateSessionResult`로 총점·리포트 JSON·`ended_at`을 동시에 세팅합니다. 리포트는 캐시 우선이라 이미 있으면 재생성 없이 파싱해 돌려주고, 같은 지원 건의 직전 채점 세션 점수와 비교해 성장 추이도 보여줍니다. UX를 단순화하려고 '리포트=종료'로 묶었습니다."
:::

## 8. 직접 말해보기

다음을 소리 내어 답해보고, 막히면 위 섹션으로 돌아간다.

1. 30초 엘리베이터 대본을 멈추지 않고 말해본다.
2. "생성과 채점을 왜 분리했나요?"에 트레이드오프 한 문장을 포함해 답한다.
3. 멀티에이전트 루프의 6개 액션을 순서대로 말하고, REEVALUATE 분기 조건(차이 ≥20)과 중간값 규칙을 설명한다.
4. "자체 LLM이 지금 도나요?"에 **됨/안 됨을 정확히 구분**해 답한다(생성 미가동, 채점 코드 완비·기본 OpenAI).
5. 모범답안을 하나로 고정하는 이유(표시=채점=복습 일관성)와 메커니즘(afterCommit + first-writer-wins)을 잇는다.
