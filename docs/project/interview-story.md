# 프로젝트 면접 스토리 (전체)

> "이 프로젝트에 대해 설명해 주세요." — 이 한 질문에 막힘없이 답하려면, **전체를 한 그림으로 압축**하고, 그 안에 **자기 담당 영역을 정직하게 끼워 넣고**, 기술 선택마다 **"왜"** 를 댈 수 있어야 한다. 이 페이지는 그 셋을 STAR(상황·과제·행동·결과) 프레임으로 묶어, A~F 어느 영역을 맡았든 같은 방식으로 자기 스토리를 만들 수 있게 돕는 학습 자료다.

이 페이지는 외우는 대본이 아니다. **같은 사실(실제 클래스·테이블·AI 번호)을 여러 각도에서 꺼내 쓰는 훈련용** 이다. 더 깊은 말하기 인덱스는 [프로젝트 전체 면접 플레이북](/flow/interview-whole-project)에, 영역별 깊이는 [/area-a](/area-a/)~[/area-f](/area-f/)에 있다.

---

## 1. 핵심 용어 (먼저 풀고 간다)

| 용어 | 뜻 |
| --- | --- |
| **STAR** | Situation(상황) · Task(과제) · Action(행동) · Result(결과). 경험을 구조화해 답하는 면접 프레임 |
| **지원 건 (Application Case)** | "이 사람이 이 회사 이 공고에 지원한다"는 한 건. 공고가 아니라 이 케이스가 모든 분석의 핵심 단위 |
| **수직 분담** | 한 기능을 사용자 화면~백엔드~DB~관리자 화면까지 한 사람이 끝까지 책임지는 분업 방식 |
| **뉴로-심볼릭** | 신경망(LLM)과 규칙엔진(symbolic)을 결합. 점수·판정은 규칙이, 자연어 설명은 LLM이 담당 |
| **폴백(Fallback)** | 1차 수단이 실패하면 자동으로 2차·3차로 넘어가 화면이 깨지지 않게 하는 안전망 |
| **AutoPrep** | 여러 영역(A~F) 준비를 의존 그래프대로 한 번에 병렬 실행하는 AI 오케스트레이터 |

## 2. 한 줄 정의 — 모든 답이 갈라져 나오는 문장

> **"CareerTuner는 '공고'가 아니라 '지원 건(Application Case)'을 핵심 단위로, 프로필·공고분석·적합도·면접·첨삭·결제·커뮤니티를 하나의 케이스에 꿰는 AI 취업 전략 플랫폼입니다. AI 34개 기능이 6개 영역(A~F)에 수직 분담되되, 하나의 오케스트레이터가 의존 그래프대로 병렬 실행합니다."**

1분·3분·5분 스크립트는 전부 이 한 문장의 확대/축소다. 면접관이 "프로젝트 소개"를 시키는 진짜 의도는 세 가지다 — (1) 전체를 한 그림으로 압축하는가(추상화), (2) 자기가 한 일과 안 한 일을 정직히 구분하는가(주인의식), (3) 기술 선택마다 "왜"를 대는가(설계 사고).

## 3. 전체 그림 — 6영역이 한 케이스를 함께 채운다

CareerTuner는 6명이 **기능별 수직 분담**으로 만든다. 각 영역은 자기 사용자 화면·API·관리자 화면·AI 기능·DB 테이블을 통째로 소유하고, 타 영역의 원본 데이터는 **읽기전용**으로만 참조한다. 어느 영역을 맡았든, 자기 자리를 이 표에서 먼저 찾아두면 전체 스토리에 자연스럽게 끼워 넣을 수 있다.

| 영역 | 한 줄 정체성 | AI 기능 번호 | 대표 테이블 | 심화 |
| --- | --- | --- | --- | --- |
| **A 회원·프로필·인증** | 누구인지(인증)와 어떻게 설명하는지(프로필)를 소유하는 기반 신뢰 데이터 계층 | #1~5 | `users`, `user_profile`(+버전 스냅샷), `user_consent` | [/area-a](/area-a/) |
| **B 지원건·공고·기업분석** | 공고 원문을 필수/우대/담당업무/기업현황으로 쪼개는 구조화 추출 중추 | #6~11 | `application_case`, `job_posting`, `job_analysis`, `company_analysis` | [/area-b](/area-b/) |
| **C 적합도·전략·대시보드** | "지원해도 되나·무엇을 보완하나·다음 어디로"를 점수와 근거로 답하는 전략 엔진 | #12~18 | `fit_analysis`, `career_analysis_run`, `dashboard` | [/area-c](/area-c/) |
| **D 가상면접** | 세션→질문→답변평가→리포트까지 모의 면접 한 라운드를 끝까지 도는 영역 | #19~23 | `interview_session/question/answer`, `file_asset` | [/area-d](/area-d/) |
| **E 첨삭·결제·크레딧** | 원문 보존형 첨삭 + 모든 AI가 의존하는 결제·크레딧·사용량 인프라 | #24~28 | `correction_request`, `payment`, `credit_transaction`, `plan` | [/area-e](/area-e/) |
| **F 커뮤니티·고객센터·챗봇** | 사용자 간 정보 공유 + 운영 커뮤니케이션 + 3개의 챗봇 | #29~34 | `community_post/comment`, `notice`, `faq`, `support_ticket` | [/area-f](/area-f/) |

:::tip 영역 불문, 자기 자리를 한 문장으로 고정하라
표의 "한 줄 정체성"을 자기 말로 바꿔 한 문장을 만들어 두면, "본인은 어디를 담당했나요?"에 0.5초 만에 입을 뗄 수 있다. 예: "저는 _ 영역을 맡아 \[소유 테이블/대표 기능\]을 사용자 화면부터 DB·관리자까지 책임졌습니다." 영역별 1인칭 정리는 각 영역의 [interview-playbook](/area-a/interview-playbook) 페이지에 있다.
:::

### 3-1. 사용자 여정 — A에서 F를 가로지른다

```text
 [A 프로필]   스펙 스냅샷(전 영역 공통 입력)
     │
     ▼ 스펙 제공
 [B 공고분석] 공고 구조화: 필수/우대/담당업무/기업현황/면접포인트
     │
     ▼ 필수·우대 조건
 [C 적합도]   점수 + 근거 + 다음 행동 (A 스냅샷 + B 조건 비교)
     │
     ▼ 적합도 전략
 [D 가상면접] 예상질문 → 대화 진행 → 답변 평가 → 리포트
     │
     ▼ 답변·원문
 [E 첨삭/결제] 원문 비수정 개선안(크레딧 소모) · 사용량·요금제
     │
     └──⇢ C로 순환: 사용량·재분석·장기 취업경향
 [F 커뮤니티]  여정 바깥에서 교차: 실제 면접질문 → D 참고, 후기 → B 참고
```

핵심 명제: **A 프로필 스냅샷이 전 영역 공통 입력**이고, 여정은 직선이 아니라 **순환**이다. 단계별 화면은 [사용자 end-to-end 여정](/flow/user-journey), 케이스 중심 데이터 흐름은 [지원 건 중심 흐름](/flow/application-case)에 상세하다.

### 3-2. 단계별로 무엇을 받아 무엇을 넘기나

| 단계 | 영역 | 받는 것(의존 출처) | 만드는 것 | 다음에 넘기는 것 |
| --- | --- | --- | --- | --- |
| 프로필 | A | 이력서·경력·자소서 | 프로필 요약·스킬·완성도 | 전 영역 공통 입력 |
| 공고분석 | B | 공고원문·OCR | 필수/우대/담당업무/기업/면접포인트 | C 기준·D 입력 |
| 적합도 | C | A 스냅샷 + B 필수/우대 | 점수·근거·부족역량·로드맵 | D 질문·E 첨삭·장기경향 |
| 가상면접 | D | B 공고/포인트 + C 적합도 | 질문·평가·리포트 | C 장기경향·E 첨삭 |
| 첨삭 | E | D 답변 + A 원문 | 원문 비수정 개선안 | (별도 저장) |
| 결제/사용량 | E | AI 사용량·크레딧 | 요금제/충전 추천 | 전 영역 과금 토대 |
| 커뮤니티 | F | 후기·태그 | 후기 요약·실제 면접질문 | B/D 참고로 순환 |

이 표 한 장이 곧 "케이스 하나에 6영역이 직렬·순환으로 엮인다"는 메시지의 증거다. 데이터가 흐를 때마다 **타 영역 원본은 절대 덮어쓰지 않는다** — 자세한 경계는 [데이터 소유권 경계 맵](/flow/data-ownership)에 있다.

## 4. 기술 스택과 요청 한 줄의 생명주기

| 축 | 내용 |
| --- | --- |
| 백엔드 | Spring Boot 4.1.0 · Java 21 · MyBatis(JPA 금지) · MySQL 8 · REST `/api/**` |
| 프런트 | React 19 · Vite 8 · TypeScript · Tailwind v4 SPA · `/api/*` Vite 프록시 → 8080 |
| 응답 규약 | 항상 `ApiResponse<T>` 엔벨로프(`common/web/ApiResponse`) |
| 인증 | STATELESS + JWT (Spring Security, OAuth Kakao/Naver/Google, BCrypt) |
| 보조 인프라 | Python 공고추출 워커(B) · Qdrant 벡터DB(D RAG) |
| AI 공급자 | OpenAI · Ollama(자체 OSS) · Anthropic Claude Haiku |

```text
브라우저 SPA (/api/*)
  → Vite 프록시(5173→8080)
  → JwtAuthenticationFilter (Bearer 파싱 → AuthUser → ROLE_*)
  → @RestController (인가·검증)
  → Service (트랜잭션·비즈니스, MyBatis Mapper로 DB)
  → ApiResponse<T> 직렬화
  ※ SSE(오케스트레이터)만 엔벨로프를 안 타고 text/event-stream 직송
```

이 흐름은 6영역 컨트롤러가 전부 공유하는 공통 골격이라, 어느 영역을 설명하든 "내 컨트롤러도 이 4계층 위에 올라간다"로 연결할 수 있다. 더 깊은 4계층·보안 경계는 [전체 아키텍처](/flow/architecture)에 있다.

## 5. AI 통합 — 오케스트레이터와 공통 폴백

6영역의 AI가 따로 노는 게 아니라, **AutoPrep 오케스트레이터**(`ai/autoprep`)가 한 번에 묶어 돌린다. 핵심은 오케스트레이터가 **새 AI 기능을 만들지 않고** 6영역의 기존 도메인 서비스를 핸들러로 감싸 **의존 그래프대로 병렬 실행**한다는 점이다.

```java
// AutoPrepOrchestrator.java — 의존 그래프는 단 두 줄
private static final Map<String, List<String>> DEPS = Map.of(
        "FIT", List.of("JOB"),
        "INTERVIEW", List.of("JOB"));
```

- PROFILE(A)·JOB(B)·WRITE(E)·COMMUNITY(F)는 독립이라 **동시 출발**한다.
- FIT(C)·INTERVIEW(D)만 JOB(공고 분석)이 DB에 커밋된 뒤 시작한다.
- 핸들러가 없거나 비활성이면 `SKIPPED`, 예외는 `FAILED`로 변환해 **한 파트 실패가 다른 파트로 번지지 않는다.**
- 진행 상황은 `SseEmitter`로 `plan` → `part-start` → `substep` → `part-done` → `done`을 실시간 전송한다.

공통 폴백 사상은 `자체 OSS → Claude Haiku(claude-haiku-4-5) → OpenAI → Mock(규칙엔진)`이다. 다만 **정직하게 말하면 영역 편차가 크다** — 자세한 구분은 아래 7장과 [AI 공급자·폴백 전략](/flow/ai-providers-fallback), [AI 오케스트레이터 전체](/flow/ai-orchestrator), [AI 기능 맵](/flow/ai-function-map)에 있다.

## 6. STAR 기반 모범 답변 세트 (전체 + 자기 영역 끼워 넣기)

아래 답변은 **전체 흐름을 주어로** 쓰고, 마지막에 자기 영역을 한 문장으로 잇는 형태다. `[내 영역]` 자리에 자기 영역과 대표 기능을 넣어 연습한다.

### (S/T) 프로젝트 설명 + 맡은 역할

> "CareerTuner는 지원 건 단위로 스펙·면접 답변을 회사에 맞게 조정하는 AI 취업 전략 플랫폼입니다. 6명이 기능별 수직 분담을 했고, 프로필(A)·공고분석(B)·적합도(C)·가상면접(D)·첨삭/결제(E)·커뮤니티(F)가 하나의 케이스에 연결됩니다. 저는 그중 **`[내 영역]`** 을 맡아 `[대표 기능/소유 테이블]` 을 사용자 화면부터 백엔드, DB, 관리자 화면까지 책임졌습니다."

### (A) 어려웠던 점 + 행동

> "가장 어려웠던 건 **AI 결과의 불안정성을 어떻게 사용자에게 안정적으로 보이게 하느냐** 였습니다. LLM은 같은 입력에도 출력이 흔들리고, 키가 없거나 모델이 죽을 수도 있습니다. 프로젝트 전체가 이를 두 가지로 풉니다 — (1) **점수·판정 같은 결정적 값은 서버 규칙으로 확정**하고 LLM은 설명만 맡기는 뉴로-심볼릭, (2) 공급자 하나가 죽어도 화면이 안 깨지는 **폴백 체인**. `[내 영역]` 에서는 `[구체 사례]` 로 이 원칙을 적용했습니다."

> 영역별로 끼워 넣을 구체 사례 예시 — C: 신뢰도/점수를 `FitAnalysisConfidence`로 결정적 계산. D: `FallbackInterviewLlmGateway`로 자체→Haiku→OpenAI 단계 폴백. E: 크레딧을 잔액 직접 수정이 아니라 `credit_transaction` 원장으로 기록. B: 공고를 통째 생성하지 않고 문장 단위 구조화 추출. A: 프로필 변경마다 `user_profile_version`으로 스냅샷 동결.

### (A) 기술 선택 이유

> "영속성을 **MyBatis만** 쓴 건 팀 합의입니다(JPA 금지). 여러 테이블을 조인·집계하는 쿼리가 많아 SQL을 직접 제어하는 편이 유리했습니다. 응답은 전부 `ApiResponse<T>` 엔벨로프로 통일해, 프런트의 `api()` 래퍼가 한 곳에서 에러·토큰 갱신을 처리합니다. AI 호출은 인터페이스에 의존하게 설계해, 구현체(OpenAI)를 Mock·OSS로 갈아끼워도 호출부는 손대지 않습니다."

### (R) 결과

> "결과적으로 케이스 하나에 6영역이 직렬·순환으로 엮이면서도, AI 장애 시에는 Mock으로 완주하고 사용자에게 보이는 숫자는 일관됩니다. 6명이 서로의 테이블을 깨지 않고 병렬 개발할 수 있었던 건 읽기전용 경계와 공통 영역 승인 규칙 덕입니다."

### (R) 개선한다면

> "공통 폴백 사상(OSS→Haiku→OpenAI→Mock)을 **전 영역에 균일하게** 적용하는 게 다음 단계입니다. 현재는 D(면접)와 플래너에서 Haiku 폴백이 실제로 작동하고, 자체 모델 생성 화이트리스트(`OSS_GENERATION_TASKS`)는 빈 집합이라 생성은 외부 모델이 담당합니다. 자체 LLM(Qwen3 LoRA) 파인튜닝 트랙을 라이브 투입까지 끌어올리는 게 목표입니다."

## 7. 구현 상태 — 정직한 구분

면접에서 "다 됐습니다"는 의심을 산다. 됐다/계획이다를 정확히 가르는 게 오히려 신뢰를 준다.

| 구분 | 구현됨(코드 확인) | 계획·목표(설계) |
| --- | --- | --- |
| 케이스 단위 도메인 | A~F 도메인 서비스, 4계층, `ApiResponse` 엔벨로프 | — |
| 인증(A) | STATELESS JWT, Refresh 원장 폐기, OAuth 3종 | — |
| 오케스트레이터 | 의존 그래프 병렬, SSE(`plan`→`part`→`done`) | 프런트 인테이크 챗봇(F) 완전 연결 |
| 뉴로-심볼릭(C) | 결정적 신뢰도·점수·severity, fingerprint 캐시 | — |
| 폴백 | D·플래너 OSS→Haiku→OpenAI, C OSS→OpenAI→Mock | 전 영역 Haiku 적용, 자체모델 생성(화이트리스트 빈 집합) |

| 자체 LLM | 면접·첨삭 파인튜닝 트랙 진행(Qwen3 LoRA), 평가 하니스 | 담당별 자체모델 생성 라이브 투입 |

:::warning 수치 표기 주의
테이블 수는 기준 커밋의 canonical `schema.sql`에서 **172개**로 계수했고, 바뀔 수 있으므로 SHA와 함께 말한다. AI 기능 번호(#1~34)와 영역(A~F)은 제품 분류 기준으로 사용한다.
:::

## 8. 자주 나오는 꼬리질문 + 모범답안

:::details Q1. 왜 '공고'가 아니라 '지원 건'을 핵심 단위로 잡았나요?
사용자의 실제 행동(분석·면접·첨삭·결제)은 "이 회사에, 이 시점의 나로 지원한다"에 매달립니다. 공고를 단위로 잡으면 같은 공고 재지원이나, 시점에 따라 달라지는 내 스펙을 표현하지 못합니다. 그래서 허브를 `application_case`로 두고 프로필·공고·적합도·면접·첨삭·사용량을 한 케이스로 묶었습니다.
:::

:::details Q2. 6명이 한 코드베이스에서 어떻게 충돌 없이 개발했나요?
**수직 분담 + 읽기전용 경계** 입니다. 각자 사용자 화면+API+관리자+AI+DB를 통째로 소유하고, 타 영역 테이블은 읽기전용으로만 참조하며 원본을 절대 덮어쓰지 않습니다. 예를 들어 E 첨삭은 원문을 고치지 않고 `correction_request`에 분리 저장하고, C는 `source_snapshot`을 동결합니다. 공통 영역(`common/`, `schema.sql`, 라우팅, 인증, AI 프롬프트 공통 엔진)은 팀장 승인이 필요해, 경계 충돌을 구조적으로 막았습니다.
:::

:::details Q3. AI 호출이 실패하면 화면이 깨지나요?
안 깨집니다. 영역마다 폴백 진입점이 `@Primary`로 걸려 있어, 자체모델(OSS)→OpenAI→Mock 순으로 떨어집니다. 다단계 오케스트레이션(`AutoPrepOrchestrator`)에서도 한 파트가 죽으면 그 파트만 `FAILED`로 기록하고 나머지는 완주합니다. 마지막 요약에 "완료 N·건너뜀 N·실패 N"이 남습니다.
:::

:::details Q4. 점수까지 LLM에 맡기지 그랬어요? 더 똑똑할 텐데요.
채용 의사결정에서는 "왜 73점인지"를 재현·설명·책임질 수 있어야 제품이 신뢰를 얻습니다. LLM은 같은 입력에도 출력이 흔들리고 근거 추적이 어렵습니다. 그래서 적합도(C) 같은 분석에서는 **점수·판단·severity는 결정적 규칙엔진**이 확정하고 LLM은 자연어 설명만 합니다. 대신 "규칙 품질이 곧 점수 정교함"이라는 부담을 의도적으로 떠안았습니다. 상세: [뉴로-심볼릭](/area-c/neuro-symbolic).
:::

:::details Q5. AI가 여러 단계로 나뉘는데 순서·병렬은 어떻게 처리했나요?
`AutoPrepOrchestrator`가 `PrepPlan`의 step을 의존 그래프대로 실행합니다. 독립 파트(프로필·공고·자소서·커뮤니티)는 `CompletableFuture`로 동시에 출발하고, FIT(C)·INTERVIEW(D)는 JOB(공고 분석)이 DB에 커밋된 뒤 `thenRunAsync`로 시작합니다. 병렬 파트가 같은 SSE emitter에 동시에 쓰므로 `send()`를 `synchronized(emitter)`로 보호하고, 전용 데몬 스레드풀과 5분 타임아웃으로 자원 누수를 막습니다.
:::

:::details Q6. AI 비용 관리는 어떻게 했나요?
두 가지입니다. **(1) 성공 기준 멱등 차감** — `status`가 SUCCESS일 때만 크레딧을 깎고 중복 차감을 막아, 실패한 AI에는 과금하지 않습니다(영역 E의 `credit_transaction` 원장). **(2) 캐시** — 입력을 SHA-256 fingerprint로 키 삼아 같은 입력 재호출을 막습니다. 게이팅은 요금제 정책이 사용권→크레딧 폴백→거절 순으로 판단합니다. 상세: [크레딧·사용량 흐름](/flow/credit-usage).
:::
