# 자동 스토리보드 파이프라인 [영역 C]

> 화면을 직접 캡처·주석 다는 수작업을 없애고, "JSON 스펙 한 벌"에서 실제 구동 DOM을 떠서 HTML·MD·PDF·DOCX·PPTX 5종 문서를 자동 생성하는 파이프라인을 직접 설계했습니다.

## 1. 한 줄 정의

진실의 원천(single source of truth)인 **스펙 JSON**(프레임·콜아웃·여정 정의)을 입력으로, mock 앱을 헤드리스 브라우저로 순회해 **래스터 이미지 0장 — 전부 DOM/SVG 스냅샷**으로 캡처하고, 설명 박스(콜아웃)를 **DOM 요소 위치를 런타임에 실측해** 정렬한 뒤, 여러 발표·제출용 문서 포맷을 코드로 한 번에 만들어내는 빌드 도구입니다.

## 2. 단어 뜻 (용어 풀이)

| 용어 | 뜻 |
| --- | --- |
| 스토리보드 | 화면 흐름·각 화면의 의도·기능 매핑을 정리한 기획/발표 산출물 |
| 진실의 원천(SSOT) | 같은 정보를 여러 곳에 복붙하지 않고 한 곳(여기선 `spec/*.json`)만 고치면 모든 산출물이 갱신되는 기준 데이터 |
| 콜아웃(callout) | 화면 위에 얹는 점선 박스 + 번호 배지 + 캡션. "여기를 보세요" 주석 |
| 래스터(raster) | PNG/JPG 같은 픽셀 이미지. "래스터 0"은 픽셀 이미지를 안 쓰고 DOM/SVG만 쓴다는 뜻 |
| DOM 스냅샷 | 실제 렌더된 페이지의 HTML 구조를 그대로 떠서 저장한 것. 다시 열면 같은 화면이 코드로 재현됨 |
| 앵커(anchor) | 콜아웃을 붙일 위치를 좌표가 아니라 "이 텍스트가 있는 요소" 로 지정한 기준 문자열 |
| 헤드리스 브라우저 | 화면 창 없이 동작하는 브라우저. 여기선 Playwright + 시스템 Chrome |

## 3. 왜 필요한가 (없으면 무슨 문제가 생기나)

발표·제출 직전마다 화면을 캡처하고 그림판/PPT로 네모를 그려 번호를 매기는 작업은:

- **수작업이라 느리고 반복된다.** 화면 11개 × (웹/앱) × 5포맷이면 손으로 못 따라간다.
- **정합성이 깨진다.** 캡션은 고쳤는데 PPT 박스 번호는 옛날 그대로, PDF만 누락 — 포맷마다 따로 관리하면 반드시 어긋난다.
- **UI가 바뀌면 전부 다시.** 버튼 위치 한 칸 밀리면 모든 캡처·박스 좌표가 무효가 된다.
- **픽셀 이미지는 무겁고 검색·diff가 안 된다.** PNG는 repo 용량을 먹고 글자도 흐려진다.

이 파이프라인은 **스펙 한 줄을 고치면 5포맷에 동시 반영**되고, 박스는 좌표가 아니라 텍스트 앵커 기준이라 UI가 밀려도 다시 측정해 정확히 붙습니다. 면접에서 "왜 만들었냐"에 대한 답이 곧 이 4가지 문제 해결입니다.

## 4. CareerTuner에서 어디에 썼나 (실제 파일·경로)

영역 C 산출물로, **git 서브모듈** `docs/storyboard/`(별도 repo CareerTunerDocs) 안의 `C/workbench/`에 전부 있습니다. 공용 도구는 `C/workbench/tools/*.mjs`(Node ESM).

| 단계 | 파일 | 역할 |
| --- | --- | --- |
| 진실의 원천 | `spec/c-flow.spec.json` | 프레임·콜아웃 앵커·캡션(narration)·여정(journey) 정의 |
| 캡처 | `tools/capture-dom.mjs` | mock 앱 순회 → 화면별 DOM 스냅샷 + CSS 1벌 추출 |
| 렌더(HTML) | `tools/render-dom.mjs` | DOM 22개 + CSS 1벌 → 자체완결 단일 HTML, 런타임 실측 오버레이 |
| 문서 생성 | `tools/docgen.mjs` | 범용 MD → HTML/PDF/슬라이드 PNG (marked + mermaid) |
| 슬라이드 | `tools/render-slides.mjs`, `tools/export.mjs` | 16:9 슬라이드 레이아웃 → PNG → PPTX |
| 마크다운 | `tools/md.mjs` | GitHub/PR용 MD(주석 이미지 + Mermaid 여정도) |
| 앵커 보정 | `tools/patch-anchors.mjs`, `tools/patch-callouts.mjs` | 스펙의 콜아웃 앵커·좌표 일괄 수정 |
| 검증 | `tools/verify-static.mjs`, `tools/diag-static.mjs` | 생성 결과 정합성·앵커 누락 점검 |

:::tip 캡처 대상이 실제 우리 앱
캡처는 `npm run dev:mock`으로 띄운 **CareerTuner mock 데모 빌드**(`VITE_USE_MOCK=true`, 백엔드 불필요)를 헤드리스 Chrome이 순회합니다. 데모 사용자는 `김데모`, 캡처 화면은 홈·취업분석 대시보드·장기경향·지원건 상세·적합도 분석(`/applications/1/fit`)과 관리자 운영 화면(분석 통계·적합도/장기분석 프롬프트 운영) 등 C 영역 라우트입니다.
:::

## 5. 핵심 동작 원리 (4단계)

```text
[1] 캡처   capture-dom.mjs   mock 앱 22개 화면(11프레임 × web/app) → DOM 스냅샷 + _app.css 1벌
[2] 스펙   c-flow.spec.json  프레임마다 callout(앵커 텍스트·색·번호) + narration(캡션) + journey
[3] 렌더   render-dom.mjs    DOM + CSS → 단일 HTML, iframe 안 요소를 실측해 점선 박스 얹기
[4] 산출   docgen / md / export  같은 스펙으로 HTML·MD·PDF·DOCX·PPTX 동시 생성
```

**차별화 포인트 1 — 래스터 0, 전부 DOM/SVG.**
`capture-dom.mjs`는 화면을 PNG로 찍지 않고, 렌더된 `document.body`를 복제해 `<script>/<link>`를 제거한 **순수 DOM**으로 저장합니다. CSS는 화면마다 중복 저장하지 않고 `styleSheets`를 모아 `_app.css` **한 벌**만 뽑습니다. 다시 열 땐 이 CSS를 각 iframe `srcdoc`에 주입해 React·서버·이미지 없이 똑같은 화면이 코드로 재현됩니다. 정적 캡처에서 깨지는 것들도 코드로 보정합니다:

- `min-h-screen`/`100vh` 강제 높이를 평탄화 → iframe에서 거대한 빈 공간 제거
- IntersectionObserver 스크롤 리빌(`opacity-0`)을 강제로 `opacity-100` 처리 → below-fold도 보이게
- 캡처 전 페이지 끝까지 스크롤해 모든 리빌 애니메이션을 트리거한 뒤 맨 위로 복귀

**차별화 포인트 2 — 콜아웃을 좌표가 아니라 텍스트 앵커로 런타임 실측.**
스펙의 콜아웃은 **앵커 문자열**(예: 적합도 재분석, 직무 적합도)을 들고 있습니다. `render-dom.mjs`의 주입 스크립트가 iframe 안에서 그 텍스트를 가진 **가장 구체적인(작은) 요소**를 찾아 `getBoundingClientRect()`로 실측하고, 여러 앵커의 **합집합 박스**를 계산해 점선 박스를 얹습니다. 폰트가 밀리거나 글자가 늘어나도 박스가 따라옵니다.

| 모드 | 동작 |
| --- | --- |
| 기본(union) | 앵커 텍스트의 최소 요소들을 합집합 |
| `wrap:"card"` | shadcn 카드(`data-slot="card"`) 조상까지 올라가 카드 전체를 잡음 |
| `wrap:"box"` | 테두리/둥근모서리 기준 가장 가까운 박스(셀·리스트아이템) |

번호 배지가 겹치면 `dedupeBadges`가 박스 테두리를 따라 밀어 항상 보이게 하고, 오프스크린 iframe은 브라우저가 레이아웃을 미루므로 프레임을 **하나씩 뷰포트에 넣고 확정될 때까지 기다리는** 순차 드라이버로 정확도를 보장합니다. (좌표 `nx/ny/nw/nh`는 측정 실패 시 폴백으로만 남겨둡니다.)

**차별화 포인트 3 — 한 스펙 → 5포맷.**
`docgen.mjs`는 marked로 MD→HTML, mermaid로 코드펜스 다이어그램을 렌더한 뒤 결과 DOM을 추출해 CDN 없이 열리는 정적 HTML로 굽고, 같은 페이지를 `page.pdf()`로 PDF, H2/H3 경계로 분할해 16:9 슬라이드 PNG로 만듭니다. 슬라이드는 다시 PPTX로, 문서는 DOCX로 묶여 **발표·제출용 포맷 전부**가 한 데이터에서 나옵니다.

## 6. 면접 답변 3단계

- **초간단(1문장):** "화면 스토리보드를 손으로 캡처·주석 다는 대신, JSON 스펙 한 벌에서 실제 DOM을 떠서 HTML·PDF·PPTX 등 5포맷을 자동 생성하는 파이프라인을 만들었습니다."
- **기본:** "캡처를 PNG가 아니라 DOM 스냅샷으로 떠서 이미지 없이 화면을 코드로 재현하고, 설명 박스는 좌표가 아니라 '이 텍스트가 있는 요소'를 런타임에 실측해 붙입니다. 그래서 UI가 바뀌어도 스펙만 고치면 박스가 따라오고 모든 포맷이 동시에 갱신됩니다."
- **꼬리질문 대응:** "단일 진실의 원천으로 정합성을 보장하는 게 핵심 설계 의도였고, 정적 캡처에서 깨지는 스크롤 리빌·100vh·오프스크린 iframe 레이아웃 throttle 같은 엣지 케이스를 코드로 보정했습니다. 헤드리스 Chrome은 Playwright로 띄우고, 결과는 검증 도구로 앵커 누락까지 점검합니다."

## 7. 자주 나오는 꼬리질문 + 모범답안

:::details Q. 왜 스크린샷(PNG)을 안 쓰고 DOM 스냅샷을 썼나요?
우리 화면이 전부 코드/SVG로 그려져서 DOM만 떠도 손실 없이 재현됩니다. PNG는 용량이 크고 글자가 흐려지며 diff·검색이 안 되지만, DOM 스냅샷은 가볍고 텍스트가 선명하며 git diff로 변경 추적도 됩니다. CSS도 화면마다 중복 저장하지 않고 1벌만 뽑아 더 가볍습니다.
:::

:::details Q. 박스를 좌표가 아니라 텍스트 앵커로 잡는 이유는?
좌표로 박으면 폰트가 밀리거나 데이터가 길어져 레이아웃이 조금만 바뀌어도 박스가 어긋납니다. 앵커는 "이 텍스트를 가진 요소"라서 런타임에 실제 위치를 측정해 붙이므로 견고합니다. 측정이 실패할 때를 대비해 정규화 좌표를 폴백으로 함께 저장해 둡니다.
:::

:::details Q. 오프스크린 iframe에서 측정이 안 되던 문제는 어떻게 풀었나요?
브라우저는 화면 밖 iframe의 레이아웃을 미룹니다(throttle). 22개를 동시에 측정하려 하면 서로 화면 밖으로 밀려 경쟁하며 측정값이 비어 폴백으로 굳는 버그가 있었습니다. 프레임을 하나씩 뷰포트 중앙에 넣고 그 프레임이 "anchored 완료 + 좌표 안정"으로 확정될 때까지 기다린 뒤 다음으로 넘어가는 순차 드라이버로 해결했습니다.
:::

:::details Q. 진실의 원천이 무슨 의미고 어떤 이득이 있나요?
캡션·박스 위치·기능 매핑·여정도를 `spec/c-flow.spec.json` 한 곳에만 적습니다. HTML·MD·PDF·DOCX·PPTX 생성기가 모두 이 한 파일을 읽으므로, 한 줄만 고쳐도 5포맷이 동시에 일관되게 바뀝니다. 포맷마다 따로 관리할 때 생기는 "어디는 갱신, 어디는 누락" 정합성 붕괴를 원천 차단합니다.
:::

:::details Q. 이게 왜 엔지니어링 역량 어필 포인트인가요?
단순 문서 작업을 데이터 기반 빌드 파이프라인으로 재정의했습니다. SSOT 설계, 헤드리스 자동화, 런타임 DOM 측정·합집합 계산, 정적 캡처 엣지 케이스(리빌/100vh/throttle) 보정, 다포맷 산출까지 — 반복 수작업을 자동화로 치환하고 정합성을 시스템으로 보장한 사례라 운영 비용 절감과 설계 사고를 동시에 보여줍니다.
:::
