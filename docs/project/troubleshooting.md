# 어려웠던 문제와 해결 (영역별 대표 사례)

> "기능을 만든 것보다, 실패 케이스를 막은 게 더 어려웠다." CareerTuner는 6개 영역(A~F)이 하나의 **지원 건(Application Case)**을 함께 채우는 구조라, 영역마다 부딪힌 어려움의 성격도 다르다. 이 페이지는 A~F 각 영역의 대표 트러블슈팅을 **증상 → 원인 → 해결 → 배운 점** 형식으로 균형 있게 정리한다.

면접에서 "프로젝트하면서 어려웠던 점"은 거의 항상 나온다. 이때 좋은 답은 "기능을 많이 만들었다"가 아니라 "현실 제약(악의적 입력·외부 의존·동시성·환각·정합성)에서 깨지지 않게 만든 과정"을 *원인부터* 설명하는 것이다. 아래 6개 사례는 각각 30초~1분 안에 말할 수 있게 압축돼 있다.

## 1. 한 줄 정의

"코드를 짜는 것"과 "그 코드가 현실의 제약에서 깨지지 않게 하는 것"은 다른 문제다. 6개 영역의 대표 사례는 후자에 대한 기록이며, 한 영역의 무용담이 아니라 **프로젝트 전체에 걸친 공통 엔지니어링 주제**다.

| 영역 | 대표 문제 | 핵심 주제 | 핵심 해결 |
| --- | --- | --- | --- |
| A 회원·프로필·인증 | 동시 401 → 중복 토큰 리프레시 | 클라이언트 동시성 | single-flight 공유 Promise |
| B 지원건·공고분석 | 공고 URL 추출의 SSRF·환각 | 외부 입력 신뢰 경계 | 다단계 IP 검증 + grounding 검증 |
| C 분석·대시보드 | AI 점수를 믿어도 되나 / 비용 | 신뢰성·재현성 | 뉴로-심볼릭 + 가드레일 + 지문 캐시 |
| D 가상 면접 | 자체 LLM 미성숙 + 실시간 진행 | 폴백·스트리밍 | 2-축 폴백 게이트웨이 + SSE |
| E 첨삭·결제·크레딧 | 잔액 정합성·결제 위변조 | 금전 정합성 | 원장(ledger) + 멱등 차감 + 금액 재검증 |
| F 커뮤니티·챗봇 | AI 오판으로 인한 자동 처분·라우팅 비용 | 운영 안전·비용 | 운영자 확정 분리 + 슬롯 접지 + 경계구역만 LLM |

## 2. 단어 뜻 (핵심 용어)

- **single-flight (단일 플라이트)**: 동일한 작업이 동시에 여러 번 요청돼도 실제로는 한 번만 실행하고 나머지는 그 결과를 공유하는 패턴.
- **SSRF (Server-Side Request Forgery)**: 서버가 외부 입력으로 받은 URL을 그대로 요청하게 만들어, 공격자가 서버 내부망(클라우드 메타데이터, 사내 IP)을 대신 찔러보게 하는 공격.
- **grounding (근거 접지)**: LLM이 만든 결과가 실제 원문/허용된 데이터에 실제로 등장하는지 검증해, 없는 사실(환각)을 걸러내는 것.
- **뉴로-심볼릭 (Neuro-Symbolic)**: 숫자·판단은 결정적 규칙엔진(심볼릭)이 소유하고, 설명 텍스트만 LLM(뉴럴)이 만드는 역할 분리.
- **폴백 체인 (fallback chain)**: 1차 경로가 실패하면 2차·3차로 자동 강등해 전체가 죽지 않게 하는 사다리.
- **원장 (ledger)**: 잔액을 직접 덮어쓰지 않고 "변동분 + 변동 후 잔액"을 행으로 쌓아 감사 가능하게 만든 거래 장부.
- **멱등성 (idempotency)**: 같은 요청이 여러 번 들어와도 결과가 한 번 처리한 것과 같도록 보장하는 성질.
- **슬롯 접지 (slot grounding)**: 챗봇이 LLM 출력을 그대로 믿지 않고, 코드 검증을 통과한 값만 대화 상태(슬롯)로 확정하는 것.

## 3. 왜 어려운가 (공통 구조)

여섯 사례는 표면은 달라도 **"외부에서 들어온 무언가를 그대로 믿으면 깨진다"**는 한 가지 구조를 공유한다.

- **외부 입력**(B 공고 URL, F 사용자 질문)을 믿으면 → SSRF·오판
- **외부 모델 출력**(B/C/D 추출·점수·질문, F 분류)을 믿으면 → 환각·비일관·오처분
- **외부 시점**(A 동시 401, E 결제 콜백, B OCR 동시 실행)을 믿으면 → 경쟁 상태·금전 사고

그래서 각 영역의 해결책은 결국 "**검증한 값만, 검증한 경계 안에서 사용한다**"로 수렴한다. 영역마다 그 경계가 무엇이냐만 다를 뿐이다.

## 4. 영역별 대표 사례 — 증상 → 원인 → 해결 → 배운 점

### A · 동시 401로 인한 중복 토큰 리프레시

영역 A는 인증(누구인지)과 프로필(스펙 원천)을 소유하고, B~F가 이를 읽어 쓴다. 그 인증의 클라이언트 측에서 발생한 동시성 문제다.

- **증상**: 페이지 진입 시 access 토큰이 만료돼 여러 API가 동시에 401을 받고, 각자 `/auth/refresh`를 호출 → refresh 토큰 회전(rotation) 환경에서 서로의 새 토큰을 무효화 → 랜덤 로그아웃.
- **원인**: 401 핸들링이 요청별로 독립 실행돼 refresh가 N번 일어나는 경쟁 상태(race condition). access는 무상태 JWT, refresh는 DB에 저장된 opaque UUID라 **refresh가 회전되면 이전 토큰은 즉시 폐기**되는 설계가 오히려 중복 호출에 취약했다.
- **해결**: 모듈 스코프의 공유 Promise(single-flight) 하나로 직렬화.

```ts
let refreshPromise: Promise<boolean> | null = null;
function tryRefresh() {
  if (refreshPromise) return refreshPromise; // 진행 중이면 그 프라미스 재사용
  refreshPromise = (async () => {
    try { /* /auth/refresh 1회 */ }
    finally { refreshPromise = null; } // 끝나면 비워 다음 만료에 대비
  })();
  return refreshPromise;
}
```

첫 401이 refresh를 시작하고, 그 사이 들어온 다른 401들은 **같은 프라미스를 await**한다. 성공 후 각 요청은 새 토큰으로 원요청을 1회만 재시도.

- **배운 점**: JS는 단일 스레드라 OS 락 개념이 없다. "공유 자원을 갱신하는 비동기 작업"은 공유 Promise로 직렬화하는 게 사실상의 비동기 뮤텍스다. 더 깊게: [JWT 인증 흐름](/area-a/auth-jwt).

### B · 공고 URL 추출의 SSRF와 환각

영역 B는 사용자가 던진 공고를 OCR·추출해 필수/우대/담당업무로 구조화한다. "URL을 받아 서버가 직접 fetch한다"는 기능 자체가 두 가지 위험 표면을 만든다.

- **증상 1 (SSRF)**: 사용자가 `http://169.254.169.254/...` 같은 내부 주소를 넣으면 서버가 클라우드 메타데이터·사내망을 대신 긁어올 수 있음.
- **원인 1**: IP 한 줄 검사로는 부족하다 — 30x 리다이렉트로 외부→내부 우회, 검증 시점과 연결 시점 사이 DNS를 바꿔치기하는 DNS rebinding, IPv6/CGNAT 대역 우회가 있다.
- **해결 1**: `JobPostingTextExtractor`에서 다단계 방어를 걸었다.

| 방어 | 막는 것 |
| --- | --- |
| scheme 화이트리스트(http/https) | `file://`, `gopher://` 등 |
| localhost 이름 차단 | `localhost`, `*.localhost` |
| 주소 분류 검사 | loopback, site-local(사내망), link-local, multicast |
| 클라우드 메타데이터 차단 | 169.254.x 메타데이터 IP |
| CGNAT/IPv6 ULA 차단 | 100.64/10, fc00::/7 |
| 리다이렉트 재검증 | 외부→내부로 튕기는 우회 |
| DNS 핀 | 검증한 `InetAddress`로 직접 연결(DNS rebinding 차단) |

핵심은 호스트를 **한 번 resolve해 안전성을 검증한 뒤 그 검증된 주소로 직접 소켓을 열어** 연결하고, 리다이렉트마다 같은 검증을 다시 통과시키는 것이다.

- **증상 2 (환각)**: 작은 자체 추출 모델이 원문에 없는 스킬을 지어내거나, "경력 5년↑"을 신입으로 오분류함.
- **원인 2**: 소형 파인튜닝 모델의 알려진 결함 + LLM 출력을 그대로 신뢰.
- **해결 2**: 결정론 후처리 + grounding. 추출 스킬이 원문에 토큰으로 등장하는지 `validateGrounding`으로 검증하고(임계값 미만이면 규칙엔진으로 폴백), 정규식으로 연차를 보정하고, 길이·패턴으로 업무 문장이 스킬에 섞인 것을 걸러낸다.
- **배운 점**: 외부 입력으로 서버가 네트워크 요청을 하거나 모델 출력을 노출하는 모든 지점은 잠재적 사고 표면이다. "검증 시점 ≠ 사용 시점"(TOCTOU)을 의식해 검증한 값을 그대로 써야 한다. 더 깊게: [텍스트 추출·OCR(SSRF)](/area-b/text-extraction-ocr).

### C · "AI 점수를 믿어도 되나"와 반복 호출 비용

영역 C는 적합도 점수·지원 판단·학습 로드맵을 낸다. 점수가 채용 의사결정을 좌우하므로 신뢰성이 곧 제품 가치다.

- **증상**: LLM에 "이 사람 몇 점?"을 물어 그 숫자를 쓰면 같은 입력에도 점수가 흔들리고(재현 불가), 근거를 못 대고, 매 조회마다 과금되고, 모델이 죽으면 화면도 죽는다.
- **원인**: 숫자·판단을 LLM에 맡긴 것 자체가 문제.
- **해결**: 세 겹으로 분리했다.
  1. **뉴로-심볼릭** — 점수·판단·신뢰도는 규칙엔진(`MockFitAnalysisAiService`)이 결정적으로 확정하고, LLM은 그 결과를 사람이 읽을 설명 문장으로만 푼다.
  2. **가드레일** — AI가 `APPLY`를 내도 `(fitScore >= 70 AND requiredUnmet == 0)`이 아니면 `guardApplyDecision`이 `COMPLEMENT`로 강등하고 자동보정 사유를 남긴다("합격 보장" 같은 단정 금지).
  3. **지문 캐시** — `input_fingerprint = SHA-256(canonical JSON)`이 같으면 저장 결과를 재사용해 매 조회 AI 재실행을 막는다(초기 로드 무료, 명시적 재생성만 크레딧 차감). `source_snapshot`으로 분석 시점 입력을 동결해 나중에도 감사·재현이 가능하다.
- **배운 점**: "AI를 쓴다"와 "AI에게 결정을 맡긴다"는 다르다. 숫자·판단은 코드가 책임지고 LLM은 설명만 하게 분리하면 신뢰성·비용·가용성을 동시에 잡는다. 더 깊게: [뉴로-심볼릭 설계](/area-c/neuro-symbolic) · [가드레일](/area-c/guardrails) · [캐시·지문](/area-c/caching-fingerprint).

### D · 미성숙한 자체 LLM과 실시간 면접 진행

영역 D는 모의 면접 한 라운드를 끝까지 돌린다. 자체 LLM(OSS) 교체가 가장 활발한 실험장이라, "아직 못 미더운 모델"과 "끊기면 안 되는 사용자 경험"을 동시에 다뤄야 했다.

- **증상**: 자체 생성 모델이 학습 데이터 부족(질문 생성은 seed당 1개)으로 형식이 불안정해 질문 대신 환각/프로필을 뱉음. 동시에 면접 진행은 실시간이라 한 번 끊기면 체감 품질이 급락.
- **원인**: 한 게이트웨이에 생성·채점을 섞으면 폴백 정책이 의도치 않게 얽히고, 미성숙 모델에 운명을 걸면 화면이 깨진다.
- **해결**:
  - **2-축 폴백 게이트웨이** — 생성축(`InterviewLlmGateway` → 자체 → Claude Haiku(claude-haiku-4-5) → OpenAI)과 채점축(`InterviewEvaluatorProvider`)을 의도적으로 분리. 생성 task 화이트리스트(`OSS_GENERATION_TASKS`)는 현재 **빈 집합**이라 생성은 사실상 Claude→OpenAI 폴백으로 동작하고, 학습이 충분한 task부터 점진 교체한다. Claude는 자체모델로 가는 "디딤돌"로 명시적으로 다뤄진다.
  - **모범답안 first-writer-wins** — 질문 생성 직후 `afterCommit` 백그라운드로 모범답안을 미리 만들고, 조건부 UPDATE로 단 하나만 살아남게 해 "표시용 = 채점 기준"이 항상 일치하게 함.
  - **SSE 스트리밍** — 진행 상황을 실시간으로 흘려보내 끊김 없는 체감을 유지.
- **배운 점**: 미성숙한 자체 모델은 "전부 켜기/끄기"가 아니라 **task 단위 화이트리스트로 점진 교체**하고, 폴백 정책이 다른 경로(생성 vs 채점)는 섞지 않는다. 더 깊게: [폴백 게이트웨이·2-축 추상화](/area-d/fallback-gateway) · [SSE 스트리밍](/area-d/sse-streaming) · [답변 평가](/area-d/answer-evaluation).

### E · 크레딧 정합성과 결제 위변조

영역 E는 첨삭(콘텐츠 개선)과 과금 인프라(결제·크레딧·사용권)를 함께 책임진다. 금전·잔액은 1원도 틀리면 안 되는 정합성 문제다.

- **증상**: ① 결제 콜백에서 클라이언트가 보낸 금액을 그대로 믿으면 위변조 가능. ② AI 사용 시 잔액을 직접 `UPDATE`하면 동시 차감·중복 차감 사고가 난다.
- **원인**: 외부(PG 콜백)·동시(여러 AI 호출)를 신뢰한 설계.
- **해결**:
  - **원장(ledger) 분리** — 잔액을 직접 갱신하지 않고 "변동분 + 변동 후 잔액"을 `credit_transaction`/`benefit_transaction`에 행으로 기록해 모든 차감/충전이 감사 추적 가능.
  - **"결제는 잔액을 만들지 않는다"** — Toss 2단계(ready/confirm)에서 confirm 시 서버가 금액·주문을 재검증하고, 실제 지급은 `billingService`의 공통 메서드가 원장에 기록.
  - **멱등·원자적 차감** — `AiChargeService.charge`/`consumeByFeature`가 조건부 원자 차감 + 멱등키로 중복을 막도록 완성·테스트됨.
- **정직한 갭**: 차감 엔진은 단위 테스트를 통과하지만 **운영 경로에 아직 미배선**이라, 현재 AI 실행 시 `ai_usage_log`와 결과 행만 쌓이고 실제 잔액 차감은 일어나지 않는다. 면접에서는 "엔진은 완성됐고 연결만 남았다"고 정직하게 말한다.
- **배운 점**: 금전은 "마지막에 한 번 빼기"가 아니라 **기록(원장) → 검증(콜백 재검증) → 멱등(중복 방지)**을 처음부터 설계에 넣어야 한다. 더 깊게: [크레딧 시스템](/area-e/credit-system) · [결제 흐름](/area-e/payment-flow).

### F · AI 오판으로 인한 자동 처분과 라우팅 비용

영역 F는 커뮤니티·고객센터·챗봇을 묶는 공용 공간이라, AI가 잘못 판단하면 무고한 사용자가 차단되거나 잘못된 답변이 확정 발송된다.

- **증상 1 (오처분)**: 신고/부적절 분류를 AI가 자동 처분으로 끝내면 오판 시 피해가 즉시 사용자에게 간다.
- **해결 1**: **AI는 운영자 보조, 자동 처분자가 아니다.** 같은 판정 두뇌(`judge()`)를 재사용하되 `task_type`으로 부수효과를 분리 — 생성 시점 자동 검열만 toxic+confidence≥0.80에서 soft-hide(가역)하고, 사용자 신고 분류는 결과만 저장한 뒤 숨김·삭제는 운영자가 `takeAction()`으로 확정한다.
- **증상 2 (라우팅 비용·환각)**: 통합 챗봇 3종에서 모든 질문에 LLM을 태우면 비싸고 비결정적이며, 슬롯(지원 건·모드)을 LLM 출력대로 믿으면 없는 케이스를 확정할 위험.
- **해결 2**:
  - **경계구역만 LLM** — `UnifiedChatRouter`가 FAQ vs 인테이크 임베딩 점수를 비교해, 차이가 큰 명확구역은 argmax로 결정적 라우팅(LLM 0회), 애매한 경계구역(실측 약 42%)만 화행분류 LLM을 1회 태운다.
  - **슬롯 접지** — `chooseCase`는 `listCases` 화이트리스트 안의 `caseId`만 confirm해, LLM이 지어낸 값은 슬롯이 되지 못한다. 실제 면접준비 실행 스트림은 영역 D에 위임한다.
- **배운 점**: 공용 공간의 AI는 "확정"이 아니라 "제안"으로 다루고, 비용·환각은 "결정적으로 가를 수 있는 건 코드로 가르고, 정말 애매할 때만 LLM"으로 좁힌다. 더 깊게: [신고/부적절 분류](/area-f/ai-report-classify) · [인테이크 챗봇](/area-f/intake-chatbot) · [챗봇 메모리 영속](/area-f/chat-memory).

## 5. 공통 인프라에서 만난 어려움 (영역 횡단)

영역 경계를 넘는 공통 토대에서도 같은 종류의 문제를 만났다. 특정 영역 소유가 아니라 **전 영역이 공유**하는 사례라 따로 묶는다.

| 문제 | 원인 | 해결 |
| --- | --- | --- |
| AI 키 미발급 → 개발 멈춤 | 기능이 외부 키에 강결합 | 인터페이스 뒤 폴백 — `VITE_USE_MOCK` 토글 + 서버 Mock 폴백(데모/테스트의 1급 경로) |
| 인프라(푸시/Ollama) 미설정 시 흐름 끊김 | 기능 코드가 인프라를 "항상 있다"고 가정 | 기본 빈을 폴백 구현으로 두고 실제 구현이 생기면 `@Primary` 교체, 폴백마다 로그 기록 |
| LLM 호출이 트랜잭션을 점유 | LLM 응답이 최대 수분, DB 커넥션 고갈 | LLM 응답을 받은 **뒤에만** 트랜잭션으로 INSERT+상태전이를 묶음("AI는 트랜잭션 밖") |
| OCR 동시 중복 실행 | 같은 공고에 추출이 겹쳐 토큰·비용 낭비 | 생성 컬럼 + `UNIQUE`로 케이스당 동시 진행 1건을 DB 레벨에서 강제 |

:::tip 폴백은 "조용한 실패"가 아니다
모든 폴백 경로는 `log.warn`/`log.info`로 *무엇이 어떤 이유로 폴백됐는지* 흔적을 남기고, 사용량/실패는 공통 `ai_usage_log`에 기록한다. "흐름은 살리되 기록은 남긴다"가 원칙이다.
:::

전체 폴백 사다리와 오케스트레이션은 [전체 흐름 개요](/flow/)에서 한눈에 볼 수 있다.

## 6. 면접 답변 3단계

- **초간단 (1문장)**: "기능 구현보다 실패 케이스 방어가 더 어려웠고, 영역마다 SSRF·점수 신뢰성·폴백·금전 정합성·AI 오처분 같은 문제를 원인부터 추적해 해결했습니다."
- **기본**: 한 영역의 사례 하나를 골라 *증상 → 원인 → 다단계 방어 → 검증/테스트로 명세화* 순서로 1분. (B SSRF, C 뉴로-심볼릭, E 원장이 이야기하기 좋다.)
- **꼬리질문 대응**: "그거 한 줄 검사면 되는 거 아니냐"는 반박에는 → 우회 경로(리다이렉트·DNS rebinding·동시성·환각)를 들고 "그래서 검증한 값만 그대로 쓴다"로 마무리.

## 7. 자주 나오는 꼬리질문 + 모범답안

:::details Q1. SSRF에서 site-local만 막으면 충분한가요?
아니다. loopback, link-local, multicast, 클라우드 메타데이터(169.254.x), CGNAT(100.64/10), IPv6 ULA까지 분류 검사로 막았다. 더 중요한 건 ① 리다이렉트를 따라갈 때마다 같은 검증을 재실행하고 ② 검증한 주소로 직접 연결해 DNS rebinding을 차단한 것. 단일 IP 검사가 아니라 "검증-사용 일관성"이 핵심이다.
:::

:::details Q2. AI 점수를 안 믿는다면서 왜 LLM을 쓰나요?
숫자·판단과 설명을 분리하기 때문이다. 점수·지원판단·신뢰도는 규칙엔진이 결정적으로 확정(재현·감사 가능)하고, LLM은 "왜 그런지"를 사람이 읽을 문장으로만 푼다. 가드레일이 사후 보정까지 하므로, AI가 과장된 판단을 내도 코드가 강등한다.
:::

:::details Q3. 자체 LLM이 아직 못 미더운데 왜 도입했나요?
"전부 켜기/끄기"가 아니라 task 단위 점진 교체이기 때문이다. 생성 task 화이트리스트를 빈 집합으로 두면 그 task는 Claude→OpenAI 폴백으로 안전하게 돌고, 학습 데이터가 충분히 쌓인 task부터 화이트리스트에 추가한다. Claude는 자체모델로 가는 디딤돌이다.
:::

:::details Q4. 크레딧 차감을 왜 잔액 UPDATE 한 줄로 안 하나요?
동시 차감·중복 차감·감사 불가 때문이다. 원장(ledger)에 변동분과 변동 후 잔액을 행으로 쌓으면 모든 거래를 추적·재구성할 수 있고, 조건부 원자 차감 + 멱등키로 동시·중복 요청을 막는다. 결제 콜백 금액도 서버가 재검증해 위변조를 차단한다.
:::

:::details Q5. 챗봇에서 모든 질문에 LLM을 태우면 안 되나요?
비싸고 비결정적이다. 임베딩 점수 차가 큰 명확구역은 argmax로 결정적으로 가르고(LLM 0회), 정말 애매한 경계구역만 화행분류 LLM을 1회 태운다. 슬롯도 LLM 출력을 그대로 믿지 않고 화이트리스트 검증을 통과한 값만 확정(슬롯 접지)한다.
:::

:::details Q6. 폴백이 많으면 "조용한 실패"라 디버깅이 어렵지 않나요?
그래서 폴백마다 어떤 이유로 폴백됐는지 로그를 남기고, 사용량/실패는 공통 `ai_usage_log`에 기록한다. 폴백은 "조용히 무시"가 아니라 "흐름은 살리되 기록은 남긴다"로 설계했다.
:::

## 8. 직접 말해보기

1. A~F 중 한 영역의 대표 문제를 골라 *증상 → 원인 → 해결 → 배운 점* 4단계로 90초 안에 설명해 보라. (추천: B SSRF, C 뉴로-심볼릭, E 원장)
2. 면접관이 "그거 한 줄 검사/한 줄 UPDATE면 되는 거 아니에요?"라고 반박했다고 가정하고, 우회 경로나 동시성 사고를 들어 2분간 방어해 보라.
3. 여섯 사례가 공유하는 공통 구조("외부에서 들어온 것을 그대로 믿으면 깨진다")를 한 문장으로 요약해 보라.
