# 퀴즈 모음

> 퀴즈는 외운 걸 확인하는 시험이 아니라 **입으로 막히는 지점을 찾는 도구**다. 틀린 문제는 점수가 아니라 "복습 좌표"로 읽어라.

이 페이지는 퀴즈 영역의 **인덱스**다. 영역별 퀴즈로 들어가는 입구이자, "퀴즈를 어떻게 써서 면접 준비로 연결할지" 사용법을 정리한 곳이다. 개별 문제는 각 영역 페이지에 있고, 학습 페이지 본문에도 같은 형식의 퀴즈가 끝에 붙어 있다.

## 1. 퀴즈 영역 소개

CareerTuner Learning의 모든 학습 페이지는 끝에 `QuizBox` 퀴즈가 2~3개씩 붙어 있다. 이 영역은 그 퀴즈들을 **주제별로 모아 한 번에 점검**할 수 있게 만든 곳이다. 두 가지 문제 유형을 쓴다.

| 유형 | 형태 | 채점 | 쓰는 목적 |
| --- | --- | --- | --- |
| 객관식 | 4지선다 | 클릭 즉시 정답/오답 표시 | 개념을 **알아보는지**(인식) 빠르게 확인 |
| 주관식 | 정답보기형(모범답안만 제공) | 자가 채점 | 개념을 **말로 꺼내는지**(인출) 훈련 |

핵심은 주관식이다. 객관식은 보기를 보면 "아 이거" 하고 골라지지만, 면접은 보기가 없다. **먼저 입으로 답을 소리 내어 말한 뒤** 모범답안을 펼쳐 비교하는 방식으로 풀어야 효과가 난다.

## 2. 영역별 퀴즈 (바로가기)

| 영역 | 링크 | 다루는 범위 |
| --- | --- | --- |
| 백엔드 | [백엔드 퀴즈](/quizzes/backend) | Spring Boot, MVC/REST, MyBatis, JWT/Security, 예외 처리, 검증 |
| 프론트엔드 | [프론트엔드 퀴즈](/quizzes/frontend) | React, Vite, 라우팅, 상태관리, API 레이어/토큰 리프레시, PWA, Capacitor |
| AI | [AI 퀴즈](/quizzes/ai) | 적합도/취업경향/대시보드 분석(영역 C), 프롬프트 카탈로그, RAG, 오케스트레이터 |
| 프로젝트 | [프로젝트 Q&A 점검](/quizzes/project-qna) | "프로젝트 소개해보세요" 류, 내 역할(영역 C), 어려웠던 문제, 협업 |

:::tip 추천 순서
용어가 입에 안 붙으면 [용어집](/glossary/)을 먼저 한 바퀴 돌고 → 자기 담당인 [AI 퀴즈](/quizzes/ai)와 [프로젝트 Q&A](/quizzes/project-qna)부터 → 그다음 [백엔드](/quizzes/backend)·[프론트엔드](/quizzes/frontend) 순으로 넓히는 걸 권한다. 내 손으로 짠 영역 C(적합도·취업경향·대시보드 분석)를 가장 자신 있게 말할 수 있어야 한다.
:::

## 3. 왜 퀴즈로 점검하나

읽고 "이해했다"는 느낌(친숙함)과 시험에서 "말로 꺼내진다"(인출)는 전혀 다른 능력이다. 면접에서 막히는 대부분은 몰라서가 아니라 **인출 연습을 안 해서**다.

| 퀴즈 없이 읽기만 하면 | 퀴즈로 점검하면 |
| --- | --- |
| "다 아는 내용 같다"는 착각에 빠진다 | 실제로 답이 나오는 칸과 막히는 칸이 갈린다 |
| 어디를 복습할지 모른다 | 틀린 문제가 곧 복습 좌표가 된다 |
| 보기 없이 설명하는 연습이 없다 | 주관식으로 무보기 설명을 강제로 연습한다 |
| 점수가 늘 비슷하게 느껴진다 | 같은 퀴즈 재시도로 향상이 눈에 보인다 |

## 4. 사용법 — 틀린 건 학습 페이지로 복습

퀴즈는 "맞혔다/틀렸다"에서 끝내면 의미가 절반이다. 틀린 문제마다 **연결된 학습 페이지로 되돌아가는 루프**를 돌려야 한다.

```text
[1] 주관식이면 먼저 소리 내어 답한다 (보기·해설 펼치기 전에)
[2] 해설/모범답안과 비교 → 빠진 키워드·틀린 인과를 표시
[3] 틀렸거나 막혔으면 → 해당 학습 페이지 본문으로 복습
[4] 하루 뒤 같은 퀴즈를 다시 푼다 (간격 반복)
```

영역별 퀴즈에서 막히면 아래 매핑으로 바로 복습 페이지를 찾는다.

| 막힌 주제 | 복습할 페이지 |
| --- | --- |
| 응답 봉투 구조가 헷갈린다 | [ApiResponse 엔벨로프](/glossary/api-response-envelope), [백엔드 개요](/backend/) |
| 토큰 만료/리프레시가 안 떠오른다 | [JWT와 Spring Security](/backend/jwt-security), [API 레이어·토큰 리프레시](/frontend/api-layer-jwt-refresh) |
| 적합도 분석을 설명 못 하겠다 | [적합도 분석](/ai/fit-analysis) |
| 점수를 AI가 직접 매기냐는 질문이 막힌다 | [적합도 분석](/ai/fit-analysis), [OpenAI 구조화 출력](/ai/openai-structured-output) |
| 내 역할(영역 C)을 한 줄로 못 말한다 | [내 역할 — 영역 C](/project/my-role), [CareerTuner 한눈에](/project/overview) |

:::warning 자기 영역은 정직하게
영역 C 안에서도 **구현된 것**(적합도 분석 `FitAnalysisAiService`·`fit_analysis`, 장기 취업경향 `CareerAnalysisRunService`·`career_analysis_run`, 대시보드 `DashboardInsightPromptCatalog`)과 **설계 단계**(자체 LLM 커리어전략 모델 `careertuner-c-career-strategy`)를 섞어 말하지 말자. 퀴즈 해설도 이 구분을 따른다. 면접에서 계획을 구현처럼 말하면 꼬리질문에서 무너진다.
:::

## 5. 면접 셀프체크 — 점수 말고 이 기준으로

퀴즈를 다 풀었다면, 채점 점수보다 아래 4가지 자가 체크가 더 중요하다. 면접관이 실제로 보는 건 정답률이 아니라 **설명의 구조**다.

| 체크 항목 | 통과 기준 | 못 넘기면 복습 |
| --- | --- | --- |
| 30초 요약 | 어떤 주제든 한 문장으로 정의가 나온다 | 각 페이지의 "한 줄 정의" |
| 왜 필요한가 | "없으면 무슨 문제"를 짝지어 말한다 | 각 페이지 "왜 필요한가" |
| 내 코드 근거 | 실제 클래스/파일/테이블 이름을 댄다 | "CareerTuner에서 어디에 썼나" |
| 꼬리질문 방어 | 한 단계 더 깊은 질문에 안 막힌다 | "자주 나오는 꼬리질문" |

::: details 셀프체크를 객관식으로만 통과하면 안 되는 이유
객관식은 4개 보기 중 고르는 **재인(recognition)** 능력만 검증한다. 면접은 빈 종이에서 답을 만드는 **회상(recall)**이다. 객관식을 다 맞혀도 주관식에서 막히면, 아는 게 아니라 "보면 알아보는" 상태일 뿐이다. 반드시 주관식을 입으로 먼저 답하고 통과 여부를 판단하자.
:::

## 6. 답변 3단계 (퀴즈 활용법 자체를 면접식으로)

- **초간단(1문장):** 학습 페이지마다 붙은 `QuizBox` 퀴즈로 객관식은 인식을, 주관식은 무보기 설명을 점검합니다.
- **기본:** 객관식은 즉시 채점으로 개념 인식을, 주관식은 모범답안을 펼치기 전에 소리 내어 답해 인출을 훈련합니다. 틀린 문제는 연결된 학습 페이지로 돌아가 복습하는 루프를 돕니다.
- **꼬리질문 대응:** 점수보다 30초 요약·왜 필요한가·내 코드 근거·꼬리질문 방어 4가지 셀프체크 기준으로 판단하고, 특히 내 담당 영역 C는 구현된 것과 설계 단계를 정직하게 구분해 말합니다.

## 7. 자주 나오는 꼬리질문 + 모범답안

:::details 1. 객관식을 다 맞혔는데도 면접에서 막히는 이유가 뭔가요?
객관식은 보기를 알아보는 재인 능력이고, 면접은 보기 없이 답을 만드는 회상 능력입니다. 둘은 다른 뇌의 작업이라, 인식만 연습하면 무보기 상황에서 막힙니다. 그래서 주관식을 모범답안 펼치기 전에 소리 내어 답하는 인출 연습이 핵심입니다.
:::

:::details 2. 틀린 문제를 어떻게 다뤄야 하나요?
틀린 문제는 점수가 아니라 복습 좌표로 봅니다. 문제마다 연결된 학습 페이지가 있으니, 빠진 키워드나 틀린 인과를 표시한 뒤 그 페이지 본문으로 돌아가 다시 읽고, 하루 뒤 같은 퀴즈를 재시도해 간격 반복으로 굳힙니다.
:::

:::details 3. 어느 영역 퀴즈부터 풀어야 하나요?
내 담당인 영역 C 관련 [AI 퀴즈](/quizzes/ai)와 [프로젝트 Q&A](/quizzes/project-qna)부터 풉니다. 내 손으로 짠 적합도·취업경향·대시보드 분석을 가장 자신 있게 말할 수 있어야 하기 때문입니다. 그다음 백엔드·프론트엔드로 넓혀 팀 프로젝트 전체를 같은 깊이로 설명할 수 있게 합니다.
:::

:::details 4. 주관식은 정답이 안 보이는데 어떻게 채점하나요?
주관식은 정답보기형이라 모범답안 한 문단이 해설로 제공됩니다. 먼저 입으로 답한 뒤 모범답안과 비교해, 핵심 키워드가 다 나왔는지·인과(왜 그렇게 했는지)가 맞는지를 스스로 채점합니다. 빠진 부분이 있으면 그게 복습 지점입니다.
:::

:::details 5. 점수가 높으면 면접 준비가 끝난 건가요?
아닙니다. 점수보다 30초 요약·왜 필요한가·내 코드 근거(실제 클래스/파일/테이블)·꼬리질문 방어 4가지를 무보기로 통과하는지가 기준입니다. 특히 구현한 것과 설계 단계인 것을 정직하게 구분해 말할 수 있어야 진짜 준비된 상태입니다.
:::

## 8. 직접 말해보기

1. 영역별 퀴즈 중 하나를 골라, 주관식 문제를 **모범답안을 펼치기 전에** 소리 내어 답해보세요. 그다음 모범답안과 비교해 빠진 키워드 2개를 찾아 적어보세요.
2. 가장 자신 없는 주제 하나를 정해, "한 줄 정의 → 왜 필요한가 → 내 코드 근거(클래스/테이블 이름) → 예상 꼬리질문 1개" 순서로 30초 안에 말해보세요.
