# 프로젝트 면접 Q&A 점검

> "CareerTuner는 채용공고에 맞춰 스펙과 면접 답변을 조정해주는 AI 취업 전략 플랫폼이고, 저는 그 안에서 적합도 분석·취업경향 분석 같은 AI 분석 기능(영역 C)을 맡았습니다."

이 페이지는 개념 암기용이 아니라 **면접 시뮬레이션**이다. 프로젝트를 설명할 때 가장 자주 받는 질문을 30초/1분 단위로 자기 입으로 답하는 훈련을 한다. 각 질문 아래 모범답안 골격을 두되, 실제 면접에서는 외운 티 내지 말고 키워드만 챙겨서 자기 말로 풀어라.

## 1. 한 줄 정의

면접 Q&A 점검은 **프로젝트 소개 → 내 역할 → 어려웠던 점 → 기술 선택 이유 → 개선점** 다섯 축을 30초·1분 버전으로 미리 말해보는 셀프 리허설이다.

## 2. 왜 이 다섯 축인가

면접관이 프로젝트에서 검증하려는 건 "이 사람이 실제로 손을 댔고, 판단을 했는가"이다. 다섯 축은 그걸 정확히 찌른다.

| 축 | 면접관이 확인하려는 것 | 가장 흔한 함정 |
| --- | --- | --- |
| 프로젝트 소개 | 한 문장으로 정리 가능한가 | 기능 나열만 하고 "왜"가 없음 |
| 내 역할 | 진짜 내가 한 게 뭔가 | 팀 전체 성과를 내 것처럼 말함 |
| 어려웠던 점 | 문제 정의·해결 능력 | "야근했다" 류의 비기술적 답 |
| 기술 선택 이유 | 트레이드오프를 아는가 | "팀에서 정해줘서요" |
| 개선점 | 메타인지·성장 가능성 | "없습니다"(자기객관화 실패) |

## 3. 답변 분량 감각 (30초 / 1분 / 꼬리질문)

- **30초**: 결론 한 줄 + 근거 한두 개. "무엇을 했고 결과가 어땠다"까지.
- **1분**: 30초 답 + 구체적 상황(어떤 문제 → 어떤 선택 → 어떤 근거).
- **꼬리질문**: 면접관이 파고들 지점을 미리 예상하고 한 단계 더 깊은 디테일(클래스명·수치·트레이드오프)을 준비.

:::tip
좋은 답변은 "피라미드"다. 결론을 먼저 던지고, 시간이 허락하는 만큼 아래로 내려가라. 면접관이 "그만"이라고 하면 거기서 멈춰도 답이 완결되도록.
:::

## 4. CareerTuner에서 끌어 쓸 실제 근거 (영역 C)

답변에 디테일을 넣을 때 쓸 수 있는 **실제 클래스·파일·테이블**. 외울 필요는 없고, 어디에 뭐가 있는지 위치 감각만 잡아둔다.

| 주제 | 끌어 쓸 근거 | 상태 |
| --- | --- | --- |
| 적합도 분석 | `OpenAiFitAnalysisAiService`, `FitAnalysisAiResult`, `fit_analysis` 테이블 | 구현됨 |
| Fallback 설계 | `FallbackFitAnalysisAiService`, `MockFitAnalysisAiService`, `OssFitAnalysisAiService` | 구현됨 |
| 취업경향 분석 | `OpenAiCareerTrendAiService`, `CareerAnalysisRunService`, `career_analysis_run` | 구현됨 |
| 대시보드 요약 | `DashboardInsightAiCommand`, `DashboardInsightPromptCatalog` | 구현됨 |
| 프롬프트 카탈로그 | `FitAnalysisPromptCatalog`, `CareerTrendPromptCatalog` | 구현됨 |
| 오케스트레이션 | AutoPrep `FitPrepHandler`, SSE 진행보고, JOB(B)→FIT(C) 의존그래프 | 구현됨 |
| 자체 LLM 전략모델 | `careertuner-c-career-strategy`(Qwen/Gemma), 학습데이터 `ml/career-strategy-llm` | **설계 단계** |
| 자동 스토리보드 | `docs/storyboard/C` spec→캡처→5포맷 파이프라인 | 구현됨 |

:::warning 정직하게 구분하라
자체 LLM 커리어전략 모델은 **설계 단계(미구현)**다. 면접에서 "지금 돌아갑니다"라고 말하면 꼬리질문에서 무너진다. "현재는 OpenAI 기반으로 동작하고, 비용·종속성을 줄이기 위해 자체 LLM으로 대체하는 설계를 해뒀다"는 식으로 구현/계획을 분리해서 말하라.
:::

## 5. 핵심 답변 골격 (다섯 축 모범 뼈대)

### 프로젝트 소개 (1분)
"CareerTuner는 채용공고에 맞춰 스펙과 면접 답변을 조정하는 AI 취업 전략 플랫폼입니다. 핵심 단위가 '공고'가 아니라 **지원 건(Application Case)**이라는 게 차별점인데, 같은 공고라도 지원자마다 강조할 스펙과 전략이 다르기 때문입니다. Spring Boot 4 + MyBatis 백엔드, React 18 프런트, 그리고 공고추출 Python 워커와 RAG용 Qdrant로 구성됐고, 6명이 기능별 수직 분담으로 개발했습니다."

### 내 역할 (1분)
"저는 영역 C, **AI 분석 파이프라인**을 맡았습니다. 지원 건과 공고를 입력받아 적합도 점수·매칭 기술·부족 역량·추천 학습/자격증·지원 전략을 산출하는 `FitAnalysisAiService`를 구현했고, 여러 지원 건을 모아 반복되는 부족 역량과 지원 패턴을 분석하는 장기 취업경향 분석(`CareerAnalysisRunService`), 대시보드 요약 인사이트까지 담당했습니다."

### 기술 선택 이유 (예: MyBatis)
"영속성은 JPA가 아니라 MyBatis를 썼습니다. AI 분석 결과처럼 복잡한 집계·조건부 쿼리가 많아서 SQL을 직접 제어하는 게 유리했고, 팀 전체가 SQL에 익숙해 러닝커브가 낮았습니다. 대신 N+1이나 매핑 누락을 막기 위해 `map-underscore-to-camel-case`와 매퍼 XML 테스트를 뒀습니다."

### 어려웠던 점 (LLM 신뢰성)
"가장 어려웠던 건 **LLM 출력의 불확실성**이었습니다. 같은 입력에도 점수가 흔들리고, 가끔 형식이 깨졌습니다. 그래서 두 가지를 했습니다. (1) OpenAI structured output으로 응답 스키마를 강제해 `FitAnalysisAiResult`로 안전하게 파싱하고, (2) 점수와 지원 판정은 LLM 출력을 그대로 믿지 않고 **서버 규칙·검증 로직으로 다시 확정**했습니다. LLM은 근거 생성용, 최종 숫자는 결정적 로직이 책임지는 구조입니다."

### 개선점 (자체 LLM 전환)
"현재 AI 분석은 OpenAI에 의존하는데, 비용과 외부 종속성이 부담입니다. 그래서 커리어 전략 도메인에 특화된 **자체 LLM(Qwen/Gemma 베이스)**으로 대체하는 설계를 해뒀고, `캐시 → 규칙엔진 → OpenAI → Mock`의 Fallback 체인으로 점진 전환할 계획입니다. 아직 학습 데이터 구축 단계입니다."

## 6. 면접 답변 3단계 (예시: "역할이 뭐였나요")

- **초간단(1문장)**: "AI 분석 파이프라인(영역 C)을 맡아 적합도·취업경향·대시보드 분석을 구현했습니다."
- **기본**: 위 1문장 + 적합도 분석이 산출하는 항목(점수·매칭/부족 기술·추천 학습·지원 전략)과 결과 저장 테이블(`fit_analysis`)까지.
- **꼬리질문 대응**: "점수는 LLM이 정하나요?" → "아니요. LLM은 분석 근거를 생성하고, 점수와 지원 판정은 서버의 규칙·검증 로직이 확정합니다. LLM이 죽어도 `Fallback...AiService`로 규칙 기반 결과를 내보냅니다."

## 7. 자주 나오는 꼬리질문 + 모범답안

:::details Q. "본인이 직접 짠 부분과 팀원이 짠 부분을 어떻게 구분하나요?"
"기능별 수직 분담이라 영역 C의 AI 분석 패키지(`fitanalysis/ai`, `analysis/ai`, `dashboard/ai`)는 제가 설계·구현했습니다. 응답 엔벨로프 `ApiResponse`, JWT 인증, 라우팅 같은 공통 영역은 팀장 소유라 변경 시 합의를 거쳤습니다. 공고추출(B), 가상면접(D/E)은 제 분석 결과의 입력·후속이라 인터페이스만 맞췄습니다."
:::

:::details Q. "AI가 틀린 답을 주면 사용자에게 어떻게 보이나요?"
"세 겹 방어가 있습니다. (1) structured output으로 형식 깨짐 차단, (2) 점수·판정은 서버 규칙으로 재확정, (3) AI 호출 자체가 실패하면 `FallbackFitAnalysisAiService`가 규칙 기반 결과나 Mock으로 graceful degradation 합니다. 에러는 `ApiResponse` 엔벨로프의 `AI_UNAVAILABLE` 같은 `ErrorCode`로 일관되게 내려갑니다."
:::

:::details Q. "여러 분석이 순서대로 돌아가야 할 텐데 어떻게 조율했나요?"
"AutoPrep 오케스트레이터를 썼습니다. 공고 분석(JOB, 영역 B)이 끝나야 적합도(FIT, C)와 면접 준비(INTERVIEW, D)가 시작되는 의존 그래프가 있고, `FitPrepHandler` 같은 핸들러가 단계를 실행하면서 SSE로 진행 상황을 실시간 보고합니다. 사용자는 '분석 중'이 멈춘 게 아니라 지금 어느 단계인지 보게 됩니다."
:::

:::details Q. "왜 OpenAI를 쓰면서 자체 LLM도 준비하나요? 모순 아닌가요?"
"단계적 전략입니다. 초기엔 검증된 OpenAI로 빠르게 품질을 확보하고, 트래픽·비용이 늘면 커리어 도메인에 특화 학습한 자체 LLM(Ollama 서빙)으로 옮기는 겁니다. Fallback 체인(`캐시→규칙→OpenAI→Mock`) 덕분에 자체 모델이 약한 케이스는 자동으로 OpenAI로 넘어가, 전환 리스크 없이 점진 이행이 가능합니다. 현재는 설계·데이터 구축 단계입니다."
:::

:::details Q. "기술적으로 가장 자랑스러운 부분 하나만?"
"LLM을 '믿되 검증하는' 구조입니다. AI 결과를 그대로 사용자에게 노출하면 신뢰가 무너지니까, LLM은 근거 생성에만 쓰고 최종 점수·판정은 결정적 서버 로직이 책임지게 분리했습니다. 덕분에 AI가 흔들려도 제품 결과는 일관됩니다. 이 경계 설계가 AI 기능을 '데모'가 아니라 '제품'으로 만든 핵심이라고 생각합니다."
:::

## 8. 직접 말해보기

1. 타이머를 켜고, **"프로젝트 소개 + 내 역할"을 60초 안에** 끊김 없이 말해보라. 기능 나열이 아니라 "왜 이 제품인가 → 내가 무엇을 했나" 순서가 지켜졌는지 녹음해서 확인.
2. "AI가 틀리면 어떻게 되나요?" 꼬리질문에 **클래스명 하나(`FallbackFitAnalysisAiService`)와 ErrorCode 하나(`AI_UNAVAILABLE`)를 반드시 포함**해 30초로 답해보라. 디테일이 답변의 신뢰도를 어떻게 바꾸는지 체감해보는 훈련.

관련 개념이 흔들리면 [ApiResponse 엔벨로프](/glossary/api-response-envelope), [JWT 인증](/backend/jwt-security), [DTO](/glossary/dto)를 먼저 복습하고 오라.
